[S] Here's a sentence that should worry anyone shipping an LLM that reads web pages or emails on your behalf — the model cannot, on a structural level, tell the difference between an instruction from you and an instruction hidden inside a document it's just supposed to summarize.
[O] Which sounds catastrophic until you learn how cheap the fix apparently is — no retraining, no new architecture, just changing how you write the prompt, and attack success drops from over half the time to almost never.
[S] "Apparently" is doing a lot of work in that sentence. A fix that cheap for a problem that structural should make you suspicious, not relieved.
[O] Today's paper claims exactly that, though, so let's see if the data backs up the optimism or the suspicion.
[O] Welcome to Litsearch Audio, where an optimist, a skeptic, and a guest scholar dig into one paper from litsearch dot darvinyi dot com.
[S] Today's paper is "Defending Against Indirect Prompt Injection Attacks With Spotlighting," by Keegan Hines and colleagues at Microsoft, presented at CAMLIS in twenty twenty four.
[O] Joining us is Sophie, a researcher who's spent time with this paper and the broader indirect-injection literature. Welcome, Sophie.
[G] Glad to be here. This is one of those papers that's more useful for what it's honest about than for the headline number, and I think that honesty is worth unpacking.
[S] Sophie, set up the actual vulnerability for us. What does indirect prompt injection mean, concretely?
[G] Start with the structural fact the paper leads with — a large language model consumes one boundary-less stream of tokens. An application typically concatenates its system instructions with whatever external content the model needs to process, a web page, a retrieved document, an email, into that same stream, and the model has no built-in way to mark where trusted instructions end and untrusted content begins.
[O] So it's the classic code-versus-data problem, just imported into language models.
[G] Exactly the framing the authors use. A normal software system would never let user-supplied data be interpreted as executable code, that's a textbook injection vulnerability going back decades. An LLM has that same failure mode by default, because everything is just text to it.
[S] And "indirect" means the attacker isn't even the person talking to the model.
[G] Right, that's the key distinction from a direct jailbreak, which the paper calls a user prompt injection attack — there, the person typing to the model is the adversary, trying to talk past its own safety training. Here, the user is an innocent bystander. The attacker plants instruction-shaped text inside external content the model will later read on the user's behalf, and because models are, in the authors' words, eager to follow anything that looks like an instruction, it can end up executing the attacker's request with the user's own credentials.
[O] Which is a much scarier threat model for anything agentic, plugins, browsing, email access.
[G] That's the authors' motivation exactly. As these systems get more capability and access, an attacker doesn't need to convince anyone of anything, they just need their text to end up somewhere the model will read it.
[S] Had anyone tried to fix this before?
[G] The paper builds directly on earlier work by Yi and colleagues, who introduced a benchmark for measuring this problem and some early defensive framing. This paper pushes much harder on one idea from that lineage, prompt-side defenses any team can ship today, with no fine-tuning and no model access beyond the system prompt you already control.
[O] So walk us through spotlighting itself. What's the actual mechanism?
[G] The core idea is what the authors call a provenance signal. If the model can't structurally separate instructions from data, give it a reliable, unmistakable clue about which block is which, and tell it, in the system prompt, how to treat anything carrying that clue. Three instantiations, each stronger than the last — delimiting, datamarking, and encoding.
[S] Start with the weakest one.
[G] Delimiting is the simplest, and frankly the least novel, it mostly restates prior work. You wrap the untrusted block in special marker tokens and tell the model explicitly that anything between them is untrusted and should never be treated as an instruction. No transformation of the content, just a fence around it.
[O] And the problem is obvious once you say it out loud.
[G] It is, and the authors say it themselves. If an attacker learns what your delimiter looks like, they simply include that same delimiter pattern inside their injected content and escape the fence. A static, guessable marker is not much of a defense against someone who's paying attention.
[S] So datamarking is the fix?
[G] It's the next step. Instead of marking only the boundary, you interleave a special token continuously through the whole untrusted block, the paper's example replaces every whitespace character in the document with a marker symbol, so the model gets a constant, not just an edge-case, signal that it's looking at data rather than instructions.
[O] That sounds like it would mangle the text pretty badly.
[G] It's readable to the model even though it looks strange to a human, and that's the point, it parses the transformed document while treating the marker as a continuous, this-is-not-a-command signal. But a naive version, marking only whitespace, has an obvious hole — an attacker who writes an injected instruction with no spaces slips through completely unmarked.
[S] So how do they close that hole?
[G] Two ways, and this is where the paper is unusually candid. First, mark at tokenizer boundaries instead of strictly whitespace, so even a spaceless attack string still gets interleaved. Second, and more important, don't use a fixed marker at all — generate a fresh, random marking token before every call, drawn from a pool of allowed characters, and update the system prompt to match. Assume the entire system prompt leaks to an attacker, the paranoid but correct assumption for a defender, and a token that's only ever used once is already useless to them by the time they've seen it.
[O] Is there a way to put a number on how hard that is to guess in advance?
[G] The paper gives exactly that. If you're drawing a marking token of length k from a character pool of size N, the odds of an attacker guessing the live token in advance are one over N to the power of k, vanishingly small for any reasonable pool and length, and it resets on every call.
[S] And the third technique, encoding, that's the aggressive one.
[G] Right, this pushes the transformation further still. Instead of interleaving a marker, you transform the entire untrusted block with a well-known reversible encoding, the paper's main example is base64, before it ever reaches the prompt. A sufficiently capable model can implicitly decode that while reasoning about the content, but to a naive pattern-matcher looking for instruction-shaped text, an encoded block simply doesn't parse as commands at all.
[O] Encoding sounds like it should be bulletproof, though, it's not even human-readable text anymore.
[G] It's the strongest of the three in the results, but the authors flag a subtlety. Not every encoding is safe — they explain why a simple, symmetric cipher like ROT13 is a bad choice: an attacker who knows the defense is in place can work backward from it, precompute what their attack looks like after the encoding, and embed that, so the decode step hands back the malicious instruction anyway. The fix is a one-way-feeling transform like base64, where that precomputation doesn't work in the attacker's favor. Without walking through the actual attack text, I'll just say the paper treats this as a demonstrated bypass, not a hypothetical one.
[S] That's a pretty significant crack for a paper to admit to in its own method section.
[O] Or the most reassuring thing about it, depending how you look at it.
[G] It is, and to the authors' credit they don't bury it, it's a full subsection called Adversary Considerations. My read, going a bit beyond what's explicit, is that this candor is the most valuable part of the paper, more than the headline number — it's a checklist of exactly how each version of the defense can fail.
[O] Let's get to the actual numbers, because the headline claim in the abstract is dramatic, over fifty percent attack success down to below two.
[G] The setup: three black-box GPT-family models, text-davinci-003, GPT-3.5 Turbo, and GPT-4, all from mid twenty twenty three, run at a sampling temperature of one. The test corpus is a thousand synthetic documents, each with a variation on a simple keyword-payload attack, hidden text that tries to get the model to abandon its task and output one specific word, which makes success or failure unambiguous to score automatically.
[S] What's the no-defense baseline look like?
[G] Ugly, and inconsistent. On summarization with zero defense, GPT-3.5 Turbo and text-davinci-003 get hijacked well over half the time, roughly sixty percent and up to sixty-three percent. On question answering it's lower, somewhere between fifteen and twenty-four percent, but still far from safe. Just adding a plain instruction telling the model not to take instructions from the document barely moves the needle for GPT-3.5 Turbo.
[O] That's a rough starting point, hijacked more than half the time with nothing in place.
[S] So what's the first thing that actually helps, delimiting?
[G] It cuts that roughly sixty percent baseline by about half for GPT-3.5 Turbo. Meaningful, but the authors themselves say they don't recommend shipping it, precisely because of the bypass we just discussed.
[G] Datamarking is a much bigger jump. On summarization, it takes GPT-3.5 Turbo down to three point one percent and text-davinci-003 all the way to zero percent. On question answering, it's eight percent for GPT-3.5 Turbo, one percent for GPT-4, and zero percent again for text-davinci-003.
[S] Zero percent, twice, on a synthetic thousand-document set, that's clean enough to make me want to see the corpus itself.
[O] Fair, but even discounting for that, encoding pushes further still, doesn't it?
[G] With GPT-3.5 Turbo, encoding gets summarization all the way to zero percent and question answering down to one point eight percent, the best numbers anywhere in the paper. That's the source of the headline claim.
[S] But encoding needs a capable model to actually decode correctly, you said. Does that cost anything?
[G] This is the part that matters most for anyone deciding what to deploy. The authors separately check whether these transformations quietly damage the underlying task, using four benchmarks — SQuAD, IMDB sentiment, and two SuperGLUE tasks, Word-in-Context and BoolQ. Datamarking shows no detrimental effect across any of the four. Encoding is rougher: GPT-4 holds up on most of them, though it still loses roughly ten points of accuracy on Word-in-Context, while GPT-3.5 Turbo's accuracy drops noticeably across the board, since it more often fails to decode cleanly or hallucinates partway through.
[O] So the recommendation basically writes itself.
[G] It does — use datamarking as your safe default across the model family, and only reach for encoding with a high-capacity model like GPT-4, validating task performance for your own use case rather than assuming it transfers.
[S] There's also a few-shot experiment tucked into the appendix, isn't there?
[G] Yes, and it's a good example of the paper's own self-skepticism. A handful of examples showing how to recognize and refuse an injection also drove GPT-3.5 Turbo's attack success below five percent. But the authors flag that result as fragile, any examples you write can only reflect attack patterns you already know about, and it's genuinely hard to keep them from correlating with the very test set you're measuring against, its own quiet form of leakage.
[O] Okay, let me make the strongest case for this paper. This is a defense you can ship this afternoon. No fine-tuning, no new model, no infrastructure change, just a system prompt and a preprocessing step, and a vulnerability that hijacked sessions more than half the time drops to essentially never, for the recommended default, without even needing GPT-4. That's an enormous amount of risk reduction for a few lines of prompt engineering.
[S] I'll grant the cost-benefit ratio is remarkable, but I want to push on what's being measured. Every one of those numbers is against one static, non-adaptive attacker, a keyword payload the authors wrote themselves, designed to be trivially easy to score. Nowhere does an attacker who knows spotlighting is deployed actually try to break it.
[G] That's fair, and the authors more or less concede it. Adversary Considerations reasons through exactly the bypasses you'd expect, a static delimiter escaped by embedding the delimiter, a naive datamarking token dodged by avoiding the marked character, a symmetric cipher defeated by precomputing the encoded attack. But reasoning about a bypass and running it are different things, and the paper only does the former. None of the fixes they propose, dynamic tokens, one-way encodings, are tested against a real adaptive attacker.
[O] To be fair, that's true of a lot of first papers on a defense family. Someone has to establish the technique works against the obvious threat before anyone can meaningfully attack the harder version.
[S] Sure, but then don't put "reduces attack success from fifty percent to below two" in an abstract without a very large asterisk. A reader skimming that number could easily walk away thinking this is close to solved.
[G] I'd also flag a second, quieter gap. The utility check for datamarking, the technique they recommend as the default, is only run on GPT-3.5 Turbo. GPT-4 and text-davinci-003, the other two models used throughout the attack-success experiments, never appear in that task-performance table. So the claim that datamarking costs nothing in task quality is really demonstrated for one model, not three.
[O] Okay, that one lands. I'll concede that's a real hole, not just a hypothetical one, if you're deploying on GPT-4, the strongest evidence you have that datamarking is safe utility-wise is silence, not data.
[S] And the encoding result, the best number in the paper, only holds for high-capacity models by the authors' own recommendation. So the below-two-percent headline isn't something every deployment can reach, teams on smaller or open-weight models are stuck with datamarking's weaker, still large, protection.
[G] All true. The fair summary: spotlighting is well-executed engineering against the threat model it targets, and the paper's candor about its own weak points is more rigorous than most defense papers bother with. But candor about a gap isn't the same as closing it. What's missing is any evaluation against an attacker who optimizes against the defense itself, not a fixed, unsophisticated payload.
[O] Where does that leave someone actually shipping a product today?
[G] Pragmatically, still shipping it. Datamarking with dynamic tokens is cheap, has no measured downside on the benchmarks tested, and closes off the easy version of this attack, the version most real-world adversaries will actually try first. It's a legitimate floor, not a ceiling.
[S] But you shouldn't treat it as a ceiling, that's exactly the trap. If your threat model includes an attacker who knows you're running a defense like this, and for anything worth attacking you should assume that, this paper hasn't told you what happens next.
[G] Right, and that's precisely the gap later work in this space was built to fill. Optimization-based trigger search, treating a marking or encoding transform as just another obstacle to route around rather than a wall, is exactly the adaptive attacker this evaluation never puts in front of the method. Whether datamarking and encoding hold up against that kind of search is, as far as this paper goes, still open.
[O] There's a broader evaluation lesson here too, a defense number is only as strong as the attacker you tested it against.
[S] A lesson eval-minded listeners have heard in every adjacent corner of this field — a benchmark that looks saturated against a fixed test set can still be wide open against anyone who's actually trying.
[G] If there's one sentence to take from this paper, it's that a continuous, hard-to-fake signal of what's trustworthy works remarkably well against the attacker the authors tested, and the paper is refreshingly honest that it hasn't tested the harder one.
[O] My takeaway is more optimistic, this is a deployable, nearly-free risk reduction every team building on external content should already be doing, imperfect as it is.

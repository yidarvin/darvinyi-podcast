[G] Glad to be here. It's a benchmark with an unusually clean idea at its core, so there's a lot to dig into.
[S] Start with why this is even a problem. Code and data get shared all the time now, isn't reproducibility basically solved?
[G] Not even close, and the paper is upfront about the scale of it. It surveys the literature across fifteen fields, and the pattern repeats everywhere: Trisovic and colleagues found one thousand four hundred eighty of two thousand studies they checked had reproducibility errors. Belz and colleagues found four hundred seventy-two of five hundred forty-nine NLP papers. And this hits the authors' own community too.
[O] The ML reproducibility challenge story is the one that got me. That's peer researchers, actively trying, right?
[G] Right, that's the sharpest data point. In the twenty twenty-two edition, only eighteen of twenty-eight papers with both code and data available turned out to be fully reproducible. Six of those twenty-eight stayed unreproducible even after participants corresponded directly with the original authors.
[S] So even direct author contact doesn't close the gap.
[G] Correct, and the paper reads that as evidence this is a skill problem, not just a documentation problem: setting up someone else's environment, resolving dependency conflicts, figuring out which script produces which figure. That's expert labor, and it doesn't scale.
[O] Meanwhile coding agents are visibly getting better at general software work. The paper cites SWE-bench: language models alone solve under five percent of real GitHub issues, agent scaffolds push that past thirty percent.
[S] Which is exactly the kind of number that gets people saying AI will automate science any day now.
[G] The authors push back on that framing directly. Their argument is procedural: before you trust an agent to conduct novel research, you should first check whether it can reliably reproduce research that already exists. It's narrower, and critically, it's checkable. Nobody had built a benchmark that isolates that specific skill.
[O] And building one is apparently harder than it sounds.
[G] Verifying that a single paper's code actually reproduces can take a human researcher hours. Doing that for a hundred papers across disciplines just to build the benchmark would be a project on its own, and that constraint shapes everything about how CORE-Bench gets built.
[O] So how do you build a hundred-paper reproducibility benchmark without spending years verifying each one by hand?
[G] The trick is sourcing every task from CodeOcean, a platform that packages a paper's code and data into a capsule that's already been checked to run. That sidesteps the hours-per-paper verification problem, because the reproducibility groundwork is mostly already done.
[S] That's a big design choice sitting quietly under the whole benchmark. We'll come back to it.
[G] Worth flagging early, yes. The selection funnel is aggressive: they start from five thousand ninety CodeOcean capsules, filter to computer science, medical science, or social science, require Python or R, and apply ten criteria total, things like has a README, runs in under forty-five minutes, needs only a simple command, produces clearly labeled results, low run-to-run variance, under ten gigabytes, and reproducible when they actually test it locally. That funnel lands on ninety capsules, thirty-seven computer science, twenty-eight social science, twenty-five medical science, split evenly into forty-five for training and forty-five held out for testing.
[O] And each of those ninety becomes more than one task.
[G] Three, at three difficulty tiers, all built around the same underlying task questions, one hundred eighty-one of them across the corpus. Things like "report the test accuracy after epoch ten," or, for a vision question, "read the correlation value off this plot." CORE-Bench-Easy hands the agent the code's complete output already generated, pure information extraction, don't run anything. CORE-Bench-Medium hands it a working Dockerfile and a README, execute the container, then extract. CORE-Bench-Hard hands it only the README, no Dockerfile: install the dependencies yourself, figure out the run command yourself, execute, then extract.
[S] So Hard is the one that actually looks like reproducing an arbitrary paper off GitHub.
[G] That's explicitly the intent. Easy and Medium isolate specific sub-skills, retrieval and terminal use, while Hard is meant to approximate the real end-to-end job.
[O] Walk me through what the agent actually is.
[G] Two families, both built on AutoGPT. Plain AutoGPT is close to stock, with one addition, a tool that lets it query a vision-language model to read figures. CORE-Agent is a task-tuned variant: a programmatic check that the submitted report file has the right structure, plus a short list of hand-written hints per tier, drawn from reading the training set's failure logs. Things like "use pdftotext for PDF results" on Easy, "use absolute paths, not environment variables" on Medium, "determine and install dependencies before running anything" on Hard.
[S] How do you grade something where rerunning code might not even give you the same number twice?
[G] They manually reproduce every one of the ninety capsules three times and build a ninety-five percent prediction interval from those runs. An agent's answer counts as correct if it lands inside that range, not just if it matches one fixed value. Worth saying only seventeen of the one hundred eighty-one questions actually turned out to have stochastic answers, but building the grader that way regardless is the more careful choice.
[O] And the actual running of two hundred seventy tasks, that's not trivial either.
[G] Each task runs in its own isolated virtual machine, orchestrated in parallel by a manager process. Running everything sequentially was estimated at over twenty days; the parallel harness gets the full suite done in a little over two hours, and it also stops agents from tampering with the benchmark itself while they work.
[S] Let's get to the scoreboard. What actually won?
[G] CORE-Agent on GPT-4o, at every tier: sixty percent on Easy, fifty-seven point eight percent on Medium, twenty-one point five percent on Hard, on the held-out test set.
[O] And the comparison that matters is against plain AutoGPT on the same backend.
[G] Plain AutoGPT with GPT-4o scores thirty-five point six, thirty-seven point eight, and six point seven percent on the same three tiers. So the task-specific hints roughly doubled Hard-tier accuracy and pushed Easy up by close to twenty-five points.
[S] And the weaker model?
[G] That's where tuning matters even more. CORE-Agent on GPT-4o-mini gets forty-four point four, thirty-two point six, sixteen point three percent. Plain AutoGPT-mini is much worse, eight point nine, two point two, two point two. Going from plain AutoGPT-mini to CORE-Agent-mini took Easy-tier accuracy from eight point nine percent to forty-four point four percent, roughly a five-fold jump, a far bigger relative gain than the same tuning gave GPT-4o.
[O] So a handful of hints and a format checker buy you more when the base model is weaker.
[G] That's the read, yes, the paper frames it as small task-specific adjustments giving weak models the guardrails a stronger model partly has already.
[S] What about cost? I'd assume Hard eats the most budget.
[G] It does, and there's a sharper finding buried in the cost data. They capped every task at four dollars of API spend, then reran CORE-Agent on Hard with a ten-dollar cap instead. GPT-4o's accuracy moved from twenty-six percent to thirty-one percent, a small gain. GPT-4o-mini barely moved at all.
[O] So money wasn't the bottleneck.
[G] Right, and the reason is almost poetic, successful runs are cheap. CORE-Agent with GPT-4o averages fifty-four cents on tasks it solves, versus two dollars and fifty-nine cents on tasks it fails. When an agent is going to get there, it gets there fast. When it's stuck, more budget just buys more repetition of the same failed dependency install.
[S] What actually trips agents up, concretely?
[G] Two clear patterns. First, vision is harder than text across the board: on Easy, CORE-Agent with GPT-4o gets eighty-seven point nine percent of written questions right but only fifty-nine point three percent of vision questions, ones that require reading a value off a figure instead of finding it in terminal output. Second, Python tasks beat R tasks pretty consistently, R capsules more often dump results into full PDF manuscripts the agent has to parse, and R's dependency installs tend to run longer. Since the computer science slice skews heavily Python, that's part of why CS ends up the most reproducible of the three disciplines.

That's not obviously a general capability gain, that's partly reverse-engineering the test.

[G] The framing generalizes well beyond CORE-Bench itself. Treating "reproduce this paper" as a gradeable agent task gives the field something concrete to point automated reviewing and verification tools at, a journal or a conference could, in principle, run something like this before publication rather than after.
[S] It's also a useful case study for agent evaluation generally, the gap between "the agent ran the code" and "the agent understood what the code showed" is exactly the kind of construct-validity question that shows up across agentic benchmarks, not just this one.

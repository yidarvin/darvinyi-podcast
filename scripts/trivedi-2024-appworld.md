[S] Here is a number worth sitting with before we explain anything. Nine everyday apps, four hundred fifty-seven APIs, a simulated database tracking over a hundred fictitious lives across two years — and the best coding agent of twenty twenty-four still finishes under half the easy tasks.
[O] I would flip that reading. Forty-nine percent on a benchmark this unforgiving is not a failure grade, it is the first honest measurement anyone has taken of how far agentic coding still has to go.
[S] Honest is the word I want to poke at. This benchmark does not grade what your code says it did, it grades what actually changed in a database afterward, including whether you caused damage nobody asked for. That is either the most rigorous idea in agent evaluation this year, or an elaborate new way for models to fail.
[O] Welcome to Litsearch Audio. Today's paper is AppWorld, A Controllable World of Apps and People for Benchmarking Interactive Coding Agents, by Harsh Trivedi, Tushar Khot, Mareike Hartmann, Ruskin Manku, Vinty Dong, Edward Li, Shashank Gupta, Ashish Sabharwal, and Niranjan Balasubramanian, out of Stony Brook University, the Allen Institute for AI, and Saarland University. It won a Best Resource Paper award at ACL twenty twenty-four.
[S] And joining us is Bertrand, who has spent real time with this benchmark. Bertrand, welcome. What was actually broken about how agents got tested on tool use before this paper?
[G] Thanks for having me. Tool-use benchmarks before AppWorld mostly checked whether a model picked the right API and filled in the right arguments, graded by comparing its call sequence against a gold one. That works when a task is one or two calls deep. Most tasks people actually want automated are not.
[O] Give me the concrete version.
[G] The paper opens with an instruction like: play my Spotify playlist with enough songs for today's entire workout, and my plan is in a notes app. There is no fixed call sequence for that. The agent has to search the notes app, parse a duration out of free text, figure out the day, then loop over a playlist adding songs until their combined length covers it. Every step depends on what the last one returned.
[S] So it cannot be scripted or matched against one fixed trace.
[G] Right, and the paper names that a strong interaction requirement: a task that is, in principle, unsolvable without observing the environment first, because a critical input only becomes available that way. Most of AppWorld's harder tasks have this property. Prior benchmarks were not wrong, they just quietly assumed away the part that makes automation hard — code with real control flow, adapted as you go, inside an environment with actual state.
[O] And separately, where those tasks lived was its own gap?
[G] Yes. Interactive benchmarks like WebArena and OSWorld existed, but over web pages and desktop GUIs. Nobody had built a controllable ecosystem of personal apps, populated with real people's data and mutable state, where success and side effects can be read straight off a database. That combination is what AppWorld built.
[S] So what's actually running under the hood?
[G] Two pieces. AppWorld Engine, about sixty thousand lines of code, implements nine apps — Amazon, Spotify, Venmo, Gmail, Todoist, a notes app, Splitwise for bill-splitting, a file system, and a phone — operable through four hundred fifty-seven APIs. Behind them sits one SQLite database, a hundred one tables, seven hundred twenty-six columns, roughly three hundred seventy thousand rows, populated with a hundred six fictitious people, ages nineteen to sixty, wired into believable relationships — roommates, coworkers, family — with about two years of plausible activity behind them.
[O] What makes it controllable?
[G] It runs in one Python process, serverless, over an in-memory database, loading a fresh task in under half a second. Time is frozen per task with a library called freezegun, so date-dependent tasks behave identically every run, and agent code is sandboxed with gVisor. The APIs are backed by about one thousand seven hundred eighty unit tests, ninety-eight percent coverage, with documentation auto-generated from the API code itself to keep it reliable. Agents interact through a Jupyter-style stateful shell: write a code block, see the output, reuse variables next block.
[O] And what does that world actually ask the agent to do?
[G] That's AppWorld Benchmark, forty thousand more lines of code: seven hundred fifty tasks built from two hundred fifty scenarios, three variants each. A scenario is a template with placeholders, instantiated by varying both the placeholders and the starting database. Every generated task has to be well-defined, include distractors and realistic hurdles like a declined payment card, and form a contrast set with its siblings to test robustness rather than pattern-matching. On average a task touches one point eight apps, up to six; nine point five unique APIs, up to twenty-six; and about fifty lines of solution code, up to one hundred thirty-four. Comprehensions appear in essentially all tasks, loops in about three quarters.
[S] That's writing a small, correct program under uncertainty, not calling one function.
[G] Which is exactly what led to the paper's best idea: how it grades that program. The evaluator computes a fast, hash-based diff between the database before and after the agent runs. Each task has a small expected-changes set and a separate allowed-but-optional set. The task passes only if every expected change appears in that diff, and nothing appears except expected or allowed changes. Did you do the job, and did you avoid doing anything you shouldn't.
[O] That second half is underrated — most benchmarks would never catch an agent that finishes the task and also cancels the wrong order.
[G] Right, that's the collateral-damage check, enforced by an average of eight programmatic assertions per task, up to twenty-two. Many different valid programs can pass the same task since it checks outcomes, not a specific trace. Two metrics come out of this: Task Goal Completion, the percentage of tasks passing every assertion, and Scenario Goal Completion, which only counts a scenario if all three sibling tasks pass.
[S] And the splits?
[G] A hundred five for training, sixty for development, a hundred sixty-eight normal test tasks, four hundred seventeen challenge tasks. Challenge tasks specifically require APIs from apps held out of demonstrations, Amazon and Gmail, so an agent has to read documentation rather than lean on memorized patterns.
[O] Let's get to the numbers.
[G] Six methods were tested: ReAct, Plan and Execute, Full Code with Reflexion — write the whole program, then reflect on any error and retry — Iterative Parallel Function Calling, and two specialized systems, ToolLLaMA and CodeAct. The first four ran with closed models GPT-4o and GPT-4-Turbo and open models LLaMA-3 and DeepSeek-Coder. GPT-4 and Gemini 1.5 Pro were tried and dropped early: GPT-4 scored five to eighteen percent worse than GPT-4o at six times the cost, Gemini scored four to seventeen percent worse than LLaMA-3 with restrictive rate limits.
[S] So two frontier-adjacent models got cut before the main table even starts.
[G] Correct, worth remembering. The best configuration, ReAct with GPT-4o, gets forty-eight point eight Task Goal Completion on normal tasks, dropping to thirty point two on challenge. The stricter scenario metric falls further, thirty-two point one and thirteen, because the model doesn't reliably solve all three siblings — a thirty to fifty percent relative drop, consistent across methods.
[O] Everyone else?
[G] GPT-4-Turbo's best configuration tops out near thirty-two point seven and seventeen point five. The strongest open model, LLaMA-3 with Full Code and Reflexion, gets twenty-four point four on normal, seven on challenge. DeepSeek-Coder is weaker still. CodeAct and ToolLLaMA score exactly zero on every split — specialized tool-calling training apparently does not transfer to this open-ended setting.
[O] The finding I'd put on a slide is where the difficulty actually comes from, though, not the top line.
[G] Two analyses locate it. Performance degrades smoothly with hardness — ReAct falls from fifty-eight point three at the easiest difficulty level to twenty-one at the hardest, and tasks needing over sixty lines score under twenty regardless of method. Second, they reran GPT-4o with oracle APIs, the exact ones from the gold solution, instead of retrieved ones.
[S] Should be close to a ceiling, if retrieval was the hard part.
[G] It raises scores by at most seven point eight points on normal and nine point eight on challenge, topping out at fifty-four point eight and thirty-five point two — real but modest, especially since retrieval was already strong, an F1 of eighty-seven and seventy-one. So agents mostly know which APIs to call. What defeats them is composing calls into correct, adaptive code once the environment starts talking back.
[S] I'll grant that's a clean diagnosis. Now the case against over-reading it. This is a mid-twenty-twenty-four snapshot — GPT-4o, GPT-4-Turbo, LLaMA-3, DeepSeek-Coder, none of it a reasoning model. No o1, no o3-class system, no Claude three point five or later, no GPT-5. Forty-nine percent could be an entirely different number today.
[G] Fair, and the paper doesn't claim otherwise. Trivedi and colleagues maintain a live leaderboard at appworld dot dev precisely because they expect this table to age.
[S] Second complaint: those zeros. CodeAct and ToolLLaMA at exactly zero is a floor for narrow fine-tuning not generalizing, not a calibrated floor for benchmark difficulty. Chart them next to GPT-4o and you exaggerate the spread.
[O] Legitimate, though the paper does explain why those systems failed rather than just report the number.
[S] Third, the one I care about most: how real is this world? Every one of the four hundred fifty-seven APIs has clean, auto-generated documentation. Real APIs are messier — missing schemas, stale docs, rate limits that misbehave.
[G] To their credit, the authors run one ablation stripping response schemas from the documentation, forcing the agent to discover an API's output shape by calling it rather than knowing it in advance — closer to how real documentation actually behaves. But it's an appendix ablation, not run as a first-class condition alongside the main table.
[O] So the number everyone will quote comes from the friendlier setting. How much does that ablation actually move things, Bertrand?
[G] The paper doesn't rerun the full comparison under it, so there's no table showing the size of the drop. What's clear is the mechanism: missing schemas turn weakly-interactive tasks into strongly-interactive ones. Given how much scores already fall as interaction demands rise elsewhere, my own inference, going past what's written, is that this would push scores down further. That's inference, not a reported result.
[S] Fourth: contamination. The engine and gold solutions sit on a public GitHub repo. How exposed is this?
[G] Better than most static benchmarks, structurally — challenge tasks require unseen-app APIs so memorization doesn't help there, grading is state-based rather than answer-matching, and tasks are procedurally generated rather than static text. But the paper doesn't discuss whether models' training data could include the public repo itself, and the unseen-app guard only covers two of nine apps.
[S] Last one: is there any human number in this paper at all — a ceiling, a timed trial?
[G] No reported human baseline. What exists is that every task has a validated gold solution, so it's solvable by someone with full knowledge of the data, not evidence of how fast or reliably an average person would actually do it. That's a real gap if you want to calibrate forty-nine percent against a human assistant.
[O] Does the paper own up to any of this itself, in its own limitations section?
[G] To its credit, three things, in its own words. Not every real app exposes a usable API — some are browser-only — so a fully capable assistant would still need UI control this benchmark doesn't test. Every task is also single-supervisor: one person assigning one assistant, no coordination between two people's assistants, though the authors note their engine could support that as future work. And they're candid that seven hundred fifty carefully validated tasks makes a solid evaluation set but not nearly enough to train a model on, pointing to bootstrapping or self-instruction as the fix.
[S] Anything else in there worth flagging?
[G] One more: the ethics statement notes the tasks were designed by authors at North American and European institutions and may not reflect digital lives elsewhere. It's a caveat the paper states plainly rather than burying.
[O] Let me land the optimist case. Strip away the dated twenty twenty-four numbers and what's left is an evaluation template nobody had built: grade the resulting world state, not the words describing it, and explicitly penalize side effects nobody asked for. That generalizes past this one benchmark — I think it's the real contribution.
[S] I won't fight that framing. My asterisk is that "the methodology is sound" and "the current scores tell you something durable" are different claims, and only the first is well-supported here. The second needs a current re-run, a response-schema-ablated headline number, and ideally a human baseline.
[G] That's roughly where the evidence sits. The engineering underneath — over a hundred thousand lines of code, validated gold solutions, near-total test coverage — is unusually serious, which is why the grading logic itself holds up. It's the generalization of the specific numbers that's uncertain, not the design.
[O] Does this reach past benchmarking, or is it self-contained?
[G] Further, in my own read rather than the paper's own claim: grading an agent by what changed in a system of record instead of trusting its self-report is the kind of discipline later interactive-agent evaluations leaned into, whether or not they cite this one directly.
[S] If I leave people with one line: the interaction requirement and the state-based grader are the real contribution, and they'll outlive forty-nine percent. Treat that number as already stale and check the live leaderboard before quoting it.
[O] Mine is simpler — this is the first tool-use benchmark I've seen that penalizes an agent for making a mess while finishing your task, and that's exactly the property you want before trusting one near a real Amazon account.
[G] And the paper's own claim, stated precisely: the best twenty twenty-four system managed only about forty-nine percent of normal tasks and thirty percent of challenge tasks, and the bottleneck wasn't knowing which APIs to call, it was composing them into correct, adaptive code.

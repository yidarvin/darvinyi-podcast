DeepSeek's own predecessor, DeepSeek sixty-seven B, is dense, meaning every one of those sixty-seven billion parameters, and its full attention cache, gets used on every single token. During generation, ordinary Multi-Head Attention has to store a key and a value vector for every attention head, in every layer, for every token you've already produced. That's the K-V cache, and it grows with sequence length and head count. For a big dense model, that cache alone can outweigh the model's own weights in GPU memory once you push context length or batch size up, and that directly caps how many requests one machine can serve at once.
[O] And the existing fixes for that weren't free either.
[G] Right. Multi-Query Attention collapses every query head onto one shared key and value head. Grouped-Query Attention is the middle ground, sharing across small groups instead of one. Both shrink the cache a lot, but they do it by literally deleting information the attention mechanism used to have access to, and both measurably lose quality against full Multi-Head Attention.
[S] So it's a dial, cache size on one end, quality on the other, not a free lunch.
[G] That was the assumption going in. Meanwhile, on the training side, there's a second bottleneck. [O] Two separate problems, and this one paper goes after both of them at once, rather than trading one off against the other.
[S] Let's start with attention, since that's the more surprising claim. Walk us through Multi-head Latent Attention.
[G] The core move is low-rank compression. Instead of caching a full key and value per head, M L A down-projects the incoming hidden state into one small compressed vector per token, they call it the latent. At inference time, rather than ever reconstructing per-head keys and values from that latent, the up-projection matrices get algebraically folded into the query and output projection matrices ahead of time. Because matrix multiplication is associative, you can pre-multiply those matrices together offline, so at serving time the model never materializes a full key or value at all. It only ever caches the compressed latent.
[O] So the compression isn't a runtime cost, it's baked into the weights before the model ever serves a token.
[G] Exactly, that's the trick that makes it close to free. There's one wrinkle, though: rotary position embeddings. RoPE is position-sensitive, and applying it to the compressed keys directly breaks that folding trick, because a position-dependent rotation ends up wedged between the two matrices you wanted to pre-multiply, and matrix multiplication doesn't commute. Their fix is a small decoupled path, a separate low-dimensional query and one shared key that carry positional information on their own, alongside the compressed content path. So the actual cache holds two things, the compressed latent and this small decoupled key, not just one.

[S] Splitting a hundred sixty experts across machines has to create a communication problem.
[G] It does, and this is the part I think gets under-appreciated. They train with expert parallelism across eight devices, so a token's chosen experts could in principle be scattered anywhere. Left unconstrained, that floods the network. Their fix is device-limited routing, each token is restricted to experts living on at most three devices, chosen from among the devices holding its highest-affinity experts, and top-K selection happens only within those three.

[O] Picture this. An agent fails a coding problem, and instead of any gradient update or fine-tuning, it just writes itself a note that says "I messed up here, try this instead" — and that alone gets it from GPT-4's eighty percent up to ninety-one percent pass at one on HumanEval.
[S] And my first reaction to that is: show me the baseline you're actually beating, because a system that gets to retry a problem a dozen times against single-shot GPT-4 is not exactly a fair fight.
[O] Sure, but the mechanism is the story here. No reinforcement learning in the classical sense, no reward model touching parameters — just language, fed back into the prompt.
[S] Which is elegant right up until you find the one benchmark where the trick makes it worse. This paper has that benchmark, and we should get into why.
[O] Welcome to Litsearch Audio, where an optimist, a skeptic, and a guest scholar take apart one paper from litsearch dot darvinyi dot com. Today's paper is Reflexion, Language Agents with Verbal Reinforcement Learning, by Noah Shinn and colleagues, published at NeurIPS in twenty twenty-three.
[S] Joining us is Daniel, who's studied this paper closely. Daniel, welcome.
[G] Thanks for having me. This paper is easy to undersell because the core idea sounds almost too simple — the agent talks to itself about what went wrong — but the empirical picture underneath is a lot richer, including the part where it doesn't work.
[O] Let's set up the gap. Why was this even a hard problem?
[G] Classical reinforcement learning teaches an agent to improve by pushing a scalar reward back into the network's weights, and that requires enormous numbers of rollouts and an expensive fine-tuning step. For a large language model acting as an agent — clicking through a compiler, a website, a household simulator — retraining the whole model after every botched episode is just not practical.
[S] And even setting aside the cost, a scalar reward is a terrible teacher. "You failed" doesn't tell you which of your thirty actions was the mistake.
[G] Exactly, that's the credit assignment problem, and the paper leans on it hard. The contemporary alternatives sidestepped weight updates but hit their own ceilings. ReAct interleaves reasoning and acting within a single episode, but it carries no memory across attempts, so it just repeats its mistakes. Self-Refine edits a single output using self-feedback, but nothing persists across trials, and it's not grounded in real environment feedback.
[O] So the gap is: how do you get a frozen model — no weight updates at all — to actually learn something across repeated attempts at the same task.
[G] Right, and their answer is to reframe the policy update as language instead of gradients. They call it verbal reinforcement learning.
[S] Which already sounds like a rebrand of "keep a scratchpad." What's actually novel in the mechanics?
[G] Three distinct language models cooperating in a loop. There's an Actor, which is a frozen LLM — prompted with either chain of thought or ReAct — that generates the trajectory: the sequence of thoughts, actions, and observations, while also reading from a memory buffer. Then there's an Evaluator, which scores that trajectory into a reward. And critically, there's a separate Self-Reflection model, which reads both the trajectory and the score and writes a specific, first-person post-mortem — something like, "I failed because I assumed both people shared the same profession, I should have verified each one separately."
[O] And that written reflection is what gets carried forward.
[G] It gets appended to an episodic memory buffer — a sliding window of the last handful of reflections, capped so it doesn't blow past the context limit. The code setting uses a window of just one prior reflection; the reasoning and decision-making settings use three. That memory feeds back into the Actor's prompt on the next attempt. If the Evaluator says fail again, it reflects again. No weights move at any point — the entire optimization happens inside the prompt.
[S] So what does the Evaluator actually look like, because that seems like the load-bearing piece here. If it's wrong, the whole loop is chasing a bad signal.
[G] It varies by domain, and that variation turns out to matter enormously, we'll get to why. For the household decision-making tasks, it's a hand-written heuristic — things like, did the agent repeat the same action more than three times, or take more than thirty steps without progress. For the reasoning tasks, it's exact-match grading against a ground-truth answer. And for programming, it's the most interesting choice: the agent generates its own unit tests.
[O] Wait, it writes its own tests and then grades itself against them?
[G] Yes. It uses chain-of-thought prompting to produce a set of candidate tests, filters out anything that isn't syntactically valid by actually building the abstract syntax tree, and samples up to six tests to form the suite. That self-generated test suite is what tells the agent whether its code passed, which then triggers a reflection if it didn't.
[S] That's a closed loop grading itself. I'd want to know how often that self-written exam is simply wrong.
[G] Hold that thought, because the paper actually measures it, and it's the single most important number in here.
[O] Before we get to results — Daniel, walk us through what a reflection actually reads like, because "the agent writes itself a note" is doing a lot of work in that sentence.
[G] The paper's own example, from the reasoning setting: the agent gets a HotPotQA-style question wrong, and its self-reflection says something like, it failed because it incorrectly assumed two people shared the same profession, when in fact it needed to identify each profession separately. That specific, causal diagnosis is then sitting in the prompt on the next attempt, so the Actor isn't just retrying blindly, it's retrying with a stated hypothesis about its own error.
[S] Okay, that is a genuinely different object than "try again." It's closer to a coach's note than a retry button.
[O] Let's get to the numbers, because this is where it gets loud. Reflexion improves over strong baselines by twenty-two percent absolute on ALFWorld, twenty percent on HotPotQA, and eleven percent on HumanEval.
[G] Start with ALFWorld, the household decision-making suite — one hundred thirty-four multi-step tasks like finding an object or cleaning something and putting it away. ReAct plus Reflexion solves one hundred thirty of those one hundred thirty-four tasks, and it keeps improving across twelve consecutive trials. The ReAct-only baseline, without reflection, plateaus around trial six or seven and gets stuck at a twenty-two percent hallucination rate — meaning the agent keeps believing it's holding an item it doesn't actually have — with no sign of recovering on its own.
[S] And Reflexion collapses that hallucination rate toward zero?
[G] It does, essentially eliminates it, along with a second failure mode the paper calls inefficient planning — looping through the same surfaces and containers without a plan. Reflexion drives both toward zero over the trial sequence.
[O] What about HotPotQA, the reasoning benchmark?
[G] That one's a striking result for a different reason. The baselines — chain-of-thought alone, ReAct alone, even chain-of-thought given the ground-truth supporting context — fail to probabilistically improve at all. Meaning, of the questions they get wrong on the first try, none of the baselines solve them on a retry. It's a flat line. Reflexion breaks that flatline. And even the version fed ground-truth context still misses thirty-nine percent of questions outright — Reflexion recovers fourteen percentage points of accuracy on top of that, without ever seeing the answer itself.
[S] Fourteen points from just re-reading your own mistake. That's a real result. Is there an ablation isolating what part of the loop is doing that work?
[G] Yes, and it's a clean one. They compare full Reflexion against a version that only carries forward the raw prior trajectory — episodic memory without the verbal reflection step. The verbal self-reflection adds an eight percent absolute boost on top of episodic memory alone. So writing the causal explanation in natural language is doing real work, it's not just "remembering what happened before."
[O] Now the coding results, because ninety-one percent pass at one on HumanEval Python is the headline number everyone quotes.
[G] It is, and it's real — up from GPT-4's own eighty-point-one percent single-sample score. On Leetcode Hard, using a new benchmark the authors built specifically for this paper called LeetcodeHardGym, Reflexion roughly doubles GPT-4's score, from seven-point-five to fifteen percent. And they ran the same loop on Rust code translated through a multi-language compiler, HumanEval Rust climbs from sixty to sixty-eight percent, showing the method isn't Python-specific.
[S] Okay, now the part you told me to hold onto.
[G] Right — MBPP Python. On that one benchmark, Reflexion actually loses to a single GPT-4 sample: seventy-seven-point-one percent versus GPT-4's eighty-point-one percent. And the paper diagnoses exactly why. Remember the self-generated test suite acting as the Evaluator? On MBPP Python, those self-written tests have a false-positive rate of sixteen-point-three percent — meaning sixteen percent of the time, the tests pass on code that is actually incorrect. On HumanEval Python, that same false-positive rate is a mere one-point-four percent.
[O] So the agent is confidently wrong, sixteen percent of the time, and stops trying because its own exam gave it a passing grade.
[G] Exactly. It prematurely reports success on a broken solution because the evaluator it wrote for itself told it, falsely, that it had passed. The method's entire benefit collapses into a liability the moment the self-grading is unreliable.
[S] That is the whole ballgame for me. The method's headline number is downstream of how clean a benchmark's test-writing surface happens to be, not some general property of "reflection helps."
[G] And there's a targeted ablation that confirms both halves are load-bearing, on the fifty hardest HumanEval Rust problems. Baseline accuracy is sixty percent. Remove the self-generated test suite entirely and just let the agent reflect blindly — accuracy drops below baseline, to fifty-two percent, because the agent can't tell when to stop and makes harmful edits to working code. Remove the verbal reflection step and keep only the tests — accuracy sits right at the sixty percent baseline, no gain at all. Only the full combination reaches sixty-eight percent.
[O] So grounding tells the agent whether it failed, and reflection tells it why — and you need both, or you get nothing, or worse than nothing.
[G] That's the paper's framing almost verbatim, and I think it's the most honest sentence in it.
[S] Let's have the actual argument, then. I'll go first, since I've been circling it. My case: two of the three domains here — ALFWorld and HotPotQA — hand the agent a ground-truth success signal between trials. Environment success in ALFWorld, exact-match against the true answer in HotPotQA. The protocol lets the agent retry until three consecutive failures. That's best-of-many sampling under an oracle stopping rule the agent will never have in deployment. Part of what looks like "learning" is just "keep sampling until an all-knowing referee says stop."
[O] I'll take the other side. First, the paper is upfront that the honest, deployable version of this is the coding setting, where the evaluator is self-generated, not an oracle — and that's exactly where Reflexion still sets new state of the art on four of five benchmark-language pairs. Second, there's no free lunch being hidden here: the compute cost of retrying is disclosed, it's not smuggled in.
[S] It's disclosed, but it's not baselined. Reflexion is compared against single-sample GPT-4 at pass at one, but Reflexion spends many model calls per task — generation, testing, reflection, retrying across up to a dozen trials, plus separate calls to the Evaluator and Self-Reflection models. There's no compute-matched or pass-at-k-matched control. We don't actually know how much of the gain is the reflection mechanism specifically, versus simply spending five to ten times the inference budget on the same problem.
[G] Both of you are pointing at real limitations the paper itself doesn't fully resolve, so let me adjudicate with what's actually on the page. Daniel here — the oracle point is correct and it's the sharpest read of the paper: ALFWorld and HotPotQA success signals are privileged information the agent wouldn't have in the wild. Point to the skeptic. But the coding result is a real counterweight, because HumanEval and Leetcode Hard use a self-generated evaluator with no oracle, and the gains still hold, including the doubling on Leetcode Hard, which was built specifically to sit after GPT-4's pretraining cutoff so contamination can't explain it away.
[O] So a point back to me on the coding side.
[G] A partial one — because HumanEval and MBPP themselves predate GPT-4's training cutoff and are plausibly memorized, so the flagship ninety-one percent number should be read with that caveat even though the LeetcodeHardGym result can't be explained by contamination. And on the compute-matched baseline, the skeptic is simply right that the paper doesn't have one. The natural missing experiment is best-of-n sampling at equal token cost, and it's absent. That's a real gap, not a nitpick.
[S] I'll take that point, but I want to be fair back — the MBPP Python regression is exactly the kind of failure a hype-driven paper would have quietly omitted, and they reported it and diagnosed it in detail. That's unusually candid for a paper introducing its own headline method.
[O] Agreed, that's the thing that actually moved me most rereading this. A lot of agent papers from this era only show you the wins.
[G] There's one more data point worth putting on the table, from the appendix rather than the main results: a weak-model run using starchat-beta instead of GPT-4. Accuracy there is twenty-six percent before Reflexion and twenty-six percent after — completely flat. The method only fires when the underlying model is already strong enough to write a useful self-reflection and a useful self-test. It's not a general recipe for weak models to bootstrap themselves up.
[S] So the honest summary is closer to "verbal reflection amplifies a capable model's existing self-evaluation" than "an agent can learn from any failure."
[G] Fair, and consistent with the paper's own limitations section, which notes this can still fall into non-optimal local minima, since it's still policy optimization, just expressed in language rather than gradients.
[O] Zooming out — this is also, pretty clearly, the ancestor of a design pattern that's everywhere now. Memory plus self-critique plus retry is the backbone of a lot of modern coding and web agents.
[S] Which is exactly why the evaluation angle matters so much here, beyond just this one paper. If your agent's improvement is gated by the quality of its own self-grading, then the honest way to report agent progress is never just the headline pass rate — it's the pass rate alongside the evaluator's own false-positive and false-negative rates. Otherwise you can't tell whether you're looking at real capability gains or a generous grader.
[G] That's exactly what MBPP versus HumanEval Python shows in miniature — same method, same base model, and the only thing that changed was how trustworthy the self-generated grading was. Sixteen-point-three percent false positives versus one-point-four was the whole difference between a regression and a state-of-the-art result.
[O] Alright, let's wrap. Daniel, the paper's takeaway in one line?
[G] A frozen language model can improve across attempts by writing itself a natural-language post-mortem and reading it back — no fine-tuning required — but the technique is exactly as reliable as the evaluator you hand it, and the MBPP result proves that isn't a footnote, it's the central constraint.
[O] Mine is that this is a genuinely durable idea — replacing a gradient with a written memo is elegant, and the fact that it's now standard practice in agent design says the community agrees.
[S] And mine is: read the ablations before you read the headline number. Ninety-one percent pass at one is real, but it's sitting on a benchmark with an unusually clean self-test surface, an oracle-stopped retry budget in two of three domains, and no compute-matched baseline — so treat the magnitude with real caution even while the mechanism itself holds up.

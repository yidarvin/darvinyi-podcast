[O] Here's a headline number that sounds like a solved problem: one model hit eighty-two point five percent on real email tasks.
[S] Here's the number sitting right next to it in the same paper: that same class of model can nail eighty-four point five percent of its tool calls, syntactically perfect, and still get the underlying task right only forty-three point eight percent of the time.
[O] Two numbers, one model family, wildly different stories depending on which axis you look at.
[S] Which is exactly why a single pass-fail benchmark score would have hidden this entirely.
[O] Welcome to Litsearch Audio. Today's paper is MCP-RADAR, a multi-dimensional benchmark for evaluating tool use in large language models, from Xuanqi Gao and colleagues at Xi'an Jiaotong University and the University of Massachusetts Amherst, appearing at ICLR twenty twenty-six.
[S] It earned a slot because it's the first benchmark built specifically around the Model Context Protocol, the standard that's quietly become how most models talk to tools now.
[O] And joining us to dig into it is Linnea, a researcher who has spent real time with this paper.
[G] Glad to be here. One thing up front: I'm not one of the authors, so whenever I say the authors show something or they found something, take that literally.
[S] Linnea, set the scene. Why did tool use need its own protocol in the first place?
[G] Because before MCP, every model-to-tool connection was bespoke, and every new tool multiplied the integration work. The authors call it the N-by-M problem, N models, M tools, and a combinatorial mess of custom glue between them. Anthropic proposed the Model Context Protocol as a single standard interface for dynamic tool discovery and invocation, and the paper notes it was adopted across the industry in under a year.
[O] So the protocol solved the plumbing. Did anyone check whether models are actually good at using it?
[G] That's exactly the gap the paper identifies. Existing evals split into two camps, and both miss the target. Knowledge and instruction-following benchmarks test what a model knows or whether it complies, not whether it can drive a tool. And the tool-use benchmarks that already exist have two specific flaws the authors name directly: they can't cleanly separate genuine tool-driven problem-solving from a model reciting an answer it memorized in training, and most lean on synthetic or simulated environments that don't capture how messy real tool interactions actually are.
[S] So the accusation is that prior benchmarks were partly measuring contamination and toy environments, dressed up as tool competence.
[G] That's the authors' framing, yes. It's why MCP-RADAR insists on two things: decontaminated tasks a model genuinely can't solve without a tool, and objective, quantifiable scoring instead of human ratings or a single win-or-lose flag.
[O] Walk us through the construction, Linnea. What is MCP-RADAR actually made of?
[G] Five hundred and seven tasks on the cover page, spanning six domains, split into two fundamentally different task types. Precise Answer tasks live in Math and Web search, a hundred and twenty math tasks across four tools, ninety-four web-search tasks across two tools. Fuzzy Match tasks live in Email, Calendar, File management, and Terminal, a hundred and nineteen email tasks across seventeen tools, twenty-eight calendar tasks, ninety-one file-management tasks, sixty-three terminal tasks.
[S] Those two categories sound like they'd need completely different grading.
[G] They do, and that's the paper's central design choice. Precise Answer tasks have one verifiable ground truth, a number, an expression, a specific noun, so grading checks only the final answer, exact match, no matter which tool path got there. Fuzzy Match tasks don't have a single textual answer; success means invoking the correct tool with the correct arguments, so grading checks the operation itself, tool name and parameters, against one ground-truth call.
[O] Where do the tools actually come from, real integrations, or a simulation?
[G] Both, deliberately. For math and web search, real open-source MCP servers, a calculator server and a DuckDuckGo search server. File management and terminal also run on real, community-built open-source MCP tools. Email and calendar are the exception: high-fidelity mocks, EmailMCP and CalendarMCP, built to replicate Gmail's and macOS Calendar's interfaces, but talking to a local database pre-seeded with a hundred emails and fifty calendar entries instead of a live account. Forty-nine tools in total.
[S] Why mock email and calendar at all, if the whole point is realism?
[G] Safety and reproducibility. You don't want a benchmark run accidentally sending real email or deleting real calendar events, and you want the same starting state on every rerun. The tradeoff is that high-fidelity is the authors' own characterization, nobody's independently verified the mocks behave exactly like the real Gmail or Calendar APIs under edge cases.
[O] And the decontamination step, that one I found genuinely clever.
[G] It is clever. Before any Precise Answer query enters the benchmark, it's handed to Gemini two point five Flash. If Flash can already answer it without touching a tool, it's thrown out. What survives is, by construction, a set of problems that require external tool use rather than parametric recall.
[S] I want to flag that filter for later, using one model to define what counts as requiring a tool has a wrinkle.
[G] Hold that thought, it's worth its own segment. Three metrics matter here. Result Accuracy is the binary exact match I just described. Dialogue Turn Success Rate, DTSR, only applies to Fuzzy Match tasks, the fraction of interaction turns where the model produced a well-formed, applicable tool call, independent of whether the final answer was right. And Computational Resource Efficiency, CRE, captures token cost, normalized across models with max-min scaling so it plots on the same radar chart as the other two.
[O] So DTSR is roughly can it swing the bat, and Result Accuracy is did it hit the ball.
[G] That's a fair way to put it, decoupling syntactic competence from semantic correctness instead of one pass-fail number.
[S] And the models themselves?
[G] Ten of them, run through OpenRouter with a hard cap of ten interaction rounds, go over that and it counts as a failure. Six closed-source: GPT-5, GPT-4o, Gemini two point five Flash, Gemini two point five Pro, Claude Sonnet 4, and Claude 3.7 Sonnet. Four open-source: Qwen3 two-thirty-five B, DeepSeek-V3, DeepSeek-R1, and Llama-4 Maverick.
[S] Let's get to numbers. Where does the closed-versus-open gap actually show up?
[G] Sharpest in math. Gemini two point five Flash tops the table at sixty-one point two percent, GPT-5 close behind at sixty point seven percent. Open-source trails badly there, DeepSeek-V3 at twenty-eight point seven percent, Llama-4 at just twelve point eight percent.
[O] But web search flips that story, doesn't it?
[G] Completely. Web search is a wall for everyone, every one of the ten models scored under thirty percent. Gemini two point five Pro was best at twenty-nine point eight percent, Llama-4 worst at zero point eight percent. And the closed-versus-open gap that's enormous on math nearly disappears here: twenty point seven percent average for closed models against ten point eight for open, under a ten-point margin.
[S] Why would web search level the field when math doesn't?
[G] The paper's read is that web search demands two things at once, picking the right tool and formulating a precise query to extract the answer from noisy results. That compound difficulty seems to swamp whatever edge a bigger, better-trained closed model normally has.
[O] What about the Fuzzy Match side, Email, Calendar, File management, Terminal?
[G] Accuracy tracks task complexity almost exactly. Simple operational domains score high: Gemini two point five Pro hits eighty-two point five percent on both Email and Calendar. Domains that need precise sequential logic crater, the best File-management accuracy is fifty-nine point six percent, the best Terminal accuracy fifty-nine point nine percent, both again Gemini two point five Pro.
[S] Now give me the number you've been building toward.
[G] GPT-4o on File management: an eighty-four point five percent Dialogue Turn Success Rate, but only forty-three point eight percent Result Accuracy. A gap of forty point seven percentage points between executing the calls cleanly and actually solving the task. That's the paper's signature finding, and it recurs across models and domains, not just this one pairing.
[O] So it's not a fluke of one weak model, it's a pattern.
[G] It's a pattern. The authors tested it directly: tell the model up front how many tools a task needs, does that help? The improvement was two and a half to five percent, barely anything. The bottleneck isn't recognizing that more tools are needed, it's deciding which specific tool and how to parameterize it correctly.
[S] What do the failures actually look like, mechanically?
[G] The two task types fail in almost opposite ways. On Precise Answer tasks, the single largest error category is Tool Omission, the model trusts its own parametric knowledge instead of calling a tool at all, at thirty-three point six percent of errors, followed by Faulty Reasoning at twenty-three point five percent, where the tool output was fine but the model's own logic wasn't. On Fuzzy Match tasks it flips to execution failures: Inaccurate Tool Invocation, picking the wrong tool, leads at forty-three point two percent, with Parameter Errors close behind at thirty-four point six percent, things like putting a recipient's email address in the subject line.
[O] Here's my optimist case: frontier models can hold a ten-round tool conversation, emit well-formed calls at rates in the seventies and eighties across several domains, and closed models are already clearing eighty-two point five percent on real operational tasks like email and calendar. That's not AI can't use tools. That's AI mostly can, and the remaining failure is narrow and nameable, wrong tool, wrong argument.
[S] My deflationary case: a forty-point gap between DTSR and accuracy on File management is nearly half the task, not a rounding error. And web search under thirty percent for every model, including GPT-5, isn't AI mostly can, it's a core agentic skill still failing more often than it works.
[G] You're both grabbing a true piece of a bigger picture. On execution mechanics, the evidence is strong, DTSR sits in the seventies and eighties across most closed models in Email and Calendar. But on judgment, picking the correct tool for what a task actually requires, the evidence doesn't support optimism yet.
[S] Give me the sharpest piece of evidence for that.
[G] The hint experiment. Telling models exactly how many tools they need moved accuracy by only two and a half to five percent. If the bottleneck were the model not realizing it needs more tools, that intervention should have worked much better than it did. The deficit is deeper than a missing nudge, it's a shakier internal model of what each tool actually does.
[O] I'll take that concession. Judgment is the real gap, not raw execution.
[S] I'll push back on the other one, though. Linnea, compounding difficulty is still landing at zero point eight percent for Llama-4 and under thirty for literally every model tested. That's not a subtlety, that's a wall.
[G] It is a wall in practice, I'll grant that. I'm only resisting the stronger claim that this proves agentic ability is fundamentally broken in general. The paper's own account of why is the two-hard-problems-stacked explanation, tool selection plus query formulation, not a claim that tool use itself is hopeless.
[S] Fair distinction. A wall, just a more specific one than everything is broken. Now the part I actually want to litigate: the decontamination filter. Remind us how it works.
[G] Every math or web-search candidate query gets run past Gemini two point five Flash first. If Flash can already answer it without touching a tool, it's discarded. What survives is defined, by construction, as genuinely requiring a tool.
[S] And Flash then tops the math leaderboard, sixty-one point two percent, on a set whose entire membership was defined by Flash's own inability to solve it tool-free.
[G] That's a real circularity the authors don't flag themselves. It's a single-model gate, it can't guarantee a stronger model like GPT-5 wouldn't still solve a tool-required problem from memory, and the retained set is biased toward Flash's own blind spots.
[O] I'll push back a little, Linnea. Half a point of separation between the top two models doesn't feel like the headline here. Isn't the bigger, more robust story still that both of them are forty-plus points ahead of Llama-4 at twelve point eight percent?
[G] Fair pushback, the closed-versus-open gap in math is the sturdier finding, it isn't sitting on half a point of noise. I was only making the point that Flash-versus-GPT-5 specifically shouldn't be read as Flash being the better math model.
[S] Second target: exact-match grading on Fuzzy Match tasks. A tool call either matches the one ground-truth name and arguments, or it counts as wrong.
[G] And real operational tasks rarely have exactly one correct form, different but valid file paths, equivalent email phrasings, a different but still-correct step ordering. Single-reference exact match scores all of those as failures. The paper explicitly allows multiple valid tool paths for Precise Answer tasks, but enforces one rigid answer for the harder Fuzzy Match category, an inconsistent standard.
[O] So some of that famous forty-point gap between Dialogue Turn Success Rate and accuracy on File management might just be an overly strict grader, not a genuinely confused model.
[G] That's the honest reading. The paper never quantifies how much of that gap is tool-selection failure versus grading strictness, though some real portion surely is the latter.
[S] Third: the efficiency metric, Computational Resource Efficiency. What does it actually measure?
[G] The text calls it a measure of computational cost, implying lower should be better. But the radar charts draw it as an edge where longer means superior, it's max-min normalized within this paper's own model set, and no absolute token count is reported anywhere. You can't compare it to any other benchmark's efficiency numbers.
[O] That's a real inconsistency, not a nitpick.
[G] Agreed. A smaller one sits right at the top: the headline count of five hundred and seven tasks doesn't reconcile with the paper's own domain table, which actually sums to five hundred and fifteen. Add how lopsided the domains are, a hundred and nineteen email tasks against twenty-eight calendar tasks, and how thin the multi-tool subsets are, single digits in some domains, and the fine-grained rankings deserve real skepticism even where the qualitative story holds up.
[S] And nobody reports a seed or a confidence interval anywhere.
[G] Correct, everything here is a single run. Two models a point or two apart, like Flash and GPT-5 on math, sit comfortably inside what you'd expect from run-to-run noise alone.
[O] Is there anything anchoring these numbers at all, a human baseline, an oracle?
[G] No, there's no human or oracle-tool upper bound anywhere, so you can't tell how far even the best score, eighty-two point five percent on email, sits from a ceiling. And structurally, a three-axis radar profile doesn't remove the need for judgment, it relocates it, you still have to decide how much weight accuracy, efficiency, and turn success deserve for your own use case.
[S] Which is a very different claim from this benchmark tells you which model to deploy.
[G] Exactly right. It tells you where each model is strong and weak. What you do with that profile is still a judgment call, just a better-informed one.
[S] Zoom out. Why should this matter beyond one leaderboard?
[G] Because the core design choice is a template other agentic evaluations should copy. Most tool-use benchmarks today still report one pass-fail number, exactly the practice this paper argues against. Separating did it execute from did it choose correctly is a stronger construct, and it transfers well beyond MCP specifically.
[O] It also lands at the right moment. If every major lab is wiring models into the same protocol, a shared way to measure whether that wiring works has value independent of who's on top today.
[S] Agreed, read the leaderboard positions with the skepticism we just earned.
[G] If there's one line to take from the paper itself, it's this: frontier models have learned the syntax of tool use well before they've learned its semantics, and that gap is where the real work still is.
[O] My takeaway: this is a genuinely useful map of a real capability, and the fact that no model dominates every single axis is the most honest thing about it.

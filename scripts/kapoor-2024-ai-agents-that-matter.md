[S] Somewhere in a lab last year, a system built on tree search, self reflection, and a debugging loop lost a coding benchmark to a script whose entire strategy was asking the same model to try again.
[O] And a much cheaper script did just as well, so before anyone panics, that is not a failure of agents, that is agents finally getting graded honestly.
[S] Honestly, or embarrassingly. One of those complex agents cost over fifty times more than the cheap script for a worse score, and none of the papers proposing it had checked.
[O] That is exactly the paper we are spending the next half hour taking apart.
[O] Welcome to Litsearch Audio, where an optimist, a skeptic, and a visiting scholar dig into one paper from the literature until nothing shaky is left standing. Today's paper is "AI Agents That Matter," by Sayash Kapoor, Benedikt Stroebl, Zachary Siegel, Nitya Nadgir, and Arvind Narayanan at Princeton University, published in the Transactions on Machine Learning Research in twenty twenty four.
[S] It's on the docket because it has become probably the most cited methodology critique in agent research, and citing a paper is not the same as having actually internalized it.
[O] Joining us to keep us honest is Ferdinand, a visiting scholar who has spent real time with this work. Welcome, Ferdinand.
[G] Glad to be here. This paper reads less like a new agent proposal and more like an audit of everyone else's agent proposals, and I think that framing matters for how we should talk about it today.
[S] So set the scene. Why did this need to exist in the middle of twenty twenty four?
[G] By that point over a dozen agent benchmarks had shipped in about a year, covering web browsing, programming, and tool use, and every new "state of the art" agent leaned on more architecture: debugging loops, tree search over generation paths, verbal self reflection. The paper's argument is that agent evaluation copied language model evaluation wholesale, without noticing that agents are a genuinely different thing to measure.
[O] Different how? A benchmark is a benchmark.
[G] Three ways, and the authors are explicit about this. A language model gets one forward pass and a score. An agent wraps a stochastic model you can query repeatedly, so accuracy can be nudged up just by trying more times. Agents also cost dramatically more than one model call, the SWE-Agent authors capped every task at four dollars, which they note translates into hundreds of thousands of tokens. And agents act on external, changing environments, a live web page or a command line, not a fixed input and output pair.
[S] That third one worries me most. A static benchmark assumes the world holds still while you test against it.
[G] Right, and the paper's central worry is that the first difference lets you manufacture accuracy for free. They cite the AlphaCode result: on a competitive programming benchmark, accuracy goes from close to zero percent zero shot to over fifteen percent with a thousand retries, and over thirty percent with a million retries, same underlying model the whole time.
[O] So the leaderboard rewards patience and a bigger API budget, not insight.
[G] That's the diagnosis. And because coding benchmarks usually ship example test cases an agent can check its own work against, that shortcut is nearly free to take: keep sampling until something passes, no architectural cleverness required.
[O] So how do you actually catch that happening in practice? You can't just eyeball a leaderboard.
[G] Their device is a two dimensional Pareto frontier, accuracy on one axis, dollar cost on the other. Any agent with a competitor that is simultaneously cheaper and at least as accurate gets knocked off the frontier, no matter how it's marketed.
[S] Walk me through the actual experiment, because "we made a chart" isn't a method.
[G] They re-evaluate three complex, open source coding agents that had public code on the HumanEval leaderboard: L-D-B, which debugs its own generated code against runtime traces, LATS, which tree searches over alternative generation paths, and Reflexion, which has the model verbally reflect on failed attempts before retrying. They excluded AgentCoder, the leaderboard's actual top entry at the time, because neither working code nor a paper link was available when they ran the analysis.
[O] Worth saying out loud, they're not claiming to have dethroned the best agent that exists, just the best ones anyone could actually check.
[G] Correct, and they say so themselves. Against those three, they built three deliberately unintelligent baselines with zero agent architecture. Retry just reinvokes the same model at temperature zero, up to five times, until the problem's own example tests pass. Warming does the same thing but raises the temperature across attempts, to increase the odds some retry succeeds. Escalation starts with a cheap model, Llama three eight B, and climbs a cost ladder, a larger Llama, then G-P-T three point five, then G-P-T four, only when a cheaper model's output fails the example tests.
[S] And everyone ran under the same conditions?
[G] Every agent and baseline, five runs each, across all one hundred sixty four HumanEval problems, using the version of the benchmark from the L-D-B paper that supplies example tests for every problem, since the original only has them for one hundred sixty one. Mean accuracy and mean total dollar cost are reported with ninety five percent confidence intervals, so it isn't one lucky run.
[O] Now the part I find genuinely generative rather than just deflationary, the joint optimization idea.
[G] That's the constructive half of the paper. Once cost and accuracy are two axes, you can split an agent's lifetime cost into a one time fixed cost, tuning the prompt and few shot examples, and a recurring variable cost, the tokens billed on every subsequent call. Their claim is that spending more upfront on optimization can lower the bill that scales with usage. They demonstrate it by modifying D-S-Py, a framework for compiling language model pipelines, on HotPotQA, a multi hop question answering benchmark, using the Optuna hyperparameter search to jointly tune temperature, the number of few shot examples, which examples, and formatting instructions, optimizing for accuracy and prompt token count at the same time.
[S] So instead of packing in eight few shot examples and hoping, you search for the cheapest configuration that still hits the target accuracy.
[G] Exactly, evaluated on two backbones, G-P-T three point five and Llama three seventy B, with a hundred HotPotQA training samples for optimization and two hundred held out evaluation samples for the actual score.
[O] Give me the headline number, I know you have one.
[G] On HumanEval, Warming with G-P-T four reaches ninety-three point two percent accuracy for two dollars and forty-five cents total. Retry with G-P-T four reaches ninety-two percent for two dollars and fifty-one cents. LATS with G-P-T four, the most expensive complex agent tested, reaches eighty-eight percent for a hundred and thirty-four dollars and fifty cents, roughly fifty-five times the cost of Warming, which scored higher.
[S] Fifty-five times the cost for a lower score. That's not a rounding error, that's an entire published architecture buying nothing.
[G] Their own words are that there is no significant accuracy difference between the warming strategy and the best performing agent architecture. L-D-B with G-P-T four, the strongest complex agent in the comparison, reaches ninety-three point three percent for six dollars and thirty-six cents, statistically indistinguishable from Warming at about two and a half times the price. Even zero shot G-P-T four with no agent scaffolding at all reaches eighty-nine point six percent for a dollar ninety-three, almost on the frontier by itself.
[O] The HotPotQA result is the one I actually want to build on, because it's a positive result, not just a debunking.
[G] For G-P-T three point five, the joint optimization agent hits fifty point nine percent retrieval accuracy against D-S-Py's own random search optimizer at forty-nine point five percent, so it's both slightly more accurate and fifty-three percent cheaper per hundred inferences. For Llama three seventy B it's sixty point one percent against random search's sixty-one point seven percent, slightly less accurate this time, but forty-one percent cheaper.
[S] There has to be a catch, that optimization search itself isn't free.
[G] It isn't. The search costs a fixed two dollars and seventy-one cents for G-P-T three point five, or three dollars and eighty-four cents for Llama, and the crossover point where joint optimization becomes cheaper in total lands around thirteen hundred inferences either way. Past that, they argue the variable cost dominates completely, since a deployed agent might run millions of times.
[O] Then there's the case study that made me genuinely angry on behalf of downstream developers, the NovelQA one.
[G] This is where a benchmark's own accounting actively misleads you. Retrieval augmented G-P-T four on NovelQA scores sixty-seven point nine percent for fifty-two dollars and eighty cents total, long context G-P-T four, fed the entire novel every time, scores sixty-seven point eight percent for ninety-nine dollars and eighty cents, a coin flip apart on accuracy. But NovelQA's leaderboard batches every question about a novel into one pass, which hides the real per-question cost. Priced per single question, long context costs about a dollar eleven against retrieval's five cents, more than twenty times more expensive, while the leaderboard's own batched accounting shows retrieval costing only about half as much. That's roughly a tenfold underestimate of the real gap, baked into the benchmark's own design.
[S] And the overfitting section, the seventeen benchmarks table, that's the one I actually cite when people ask why I don't trust agent leaderboards.
[G] Seven of seventeen surveyed agent benchmarks have no held out test set at all and no stated plan to add one. Of the ones that do have holdouts, only five of ten hold data out at the level of generality the benchmark itself claims to target. The sharpest illustration is WebArena: its top public agent, STeP, scores thirty-five point eight percent, more than double the strongest baseline in the original WebArena paper, achieved partly through policies hardcoded to WebArena's own task set, like reading the current Reddit clone's U-R-L and appending a literal slash-user-slash-username to reach a profile page.
[O] Which breaks the moment a real website changes its URL structure.
[G] By construction. And on reproducibility, rerunning released code directly, LATS was reported at ninety-four point four percent but reproduced at eighty-eight percent, because the authors had silently dropped a HumanEval problem after an execution error and evaluated a subset of test cases for others. The L-D-B paper's own reported G-P-T four zero shot baseline was seventy-five percent, the reproduced number is eighty-nine point six percent, a fourteen and a half point gap that makes L-D-B's improvement over its own baseline look far larger than it is. And the paper ties this to cost directly: SWE-bench alone is over two thousand tasks, SWE-Agent's authors capped spending at four dollars a task, so a single full evaluation run could cost over eight thousand dollars, which is exactly why almost nobody computes real error bars.
[O] My strongest case for what this paper means: I think it's the best thing that could have happened to agent research, because it stops the field from rewarding elaborate architecture for its own sake. If a fifty-five times more expensive tree search can't beat retrying the same model, that tells you where the real bottleneck actually is, the base model's reasoning, not the scaffolding wrapped around it.
[S] I'll grant the reproducibility section completely, that part is airtight, they reran code, found real bugs, and confirmed every one by corresponding with the original authors. But the headline result everyone quotes, that agent complexity doesn't buy accuracy, rests on three retry baselines beating three convenience-sampled agents on one benchmark, at one snapshot of April twenty twenty four pricing. They excluded the actual top leaderboard entry because they couldn't get its code. That's not "state of the art agents lose to a coin flip," that's "the agents we could access lose."
[G] Fair distinction, and I'd add a sharper problem, contamination. HumanEval was, by twenty twenty four, heavily trained on and largely saturated for G-P-T four class models. Zero shot G-P-T four already scores eighty-nine point six percent with no agent at all. The paper argues elsewhere that benchmark overfitting is a more serious problem for agents than training data contamination, but it never checks whether that near-ceiling zero shot score is itself partly a memorization artifact, rather than evidence that agent complexity specifically doesn't help.
[O] That's real, but they do stress test it once. An appendix reruns the same comparison with the June twenty twenty three versions of the models instead of the April twenty twenty four turbo versions, and reports substantially similar results, complex agents still don't clear the baselines.
[S] It's something, but it doesn't touch contamination, since both model vintages plausibly saw HumanEval during training. I'd also push on dollar cost as the second axis generally. It's a snapshot of provider pricing on one day. The paper's own section on this says inference cost comes down over time, so whichever agent looks Pareto optimal today might not be the cheapest option in six months.
[G] The authors would say that's a feature, not a bug, it's why they built a companion calculator so readers can plug in new prices and recompute the frontier themselves. But you're right that a state of the art claim measured in dollars ages the moment prices move, in a way an accuracy number alone doesn't.
[O] Has the field actually taken any of this on board, or is this a widely cited paper nobody's practices actually changed?
[G] Honestly, neither the paper nor anything I have in front of me answers that. What I can say is it's become the default citation for "don't bother building a complex agent," which is a stronger claim than what the paper itself, with its own hedges, actually licenses.
[S] So practically, what should change for anyone building or evaluating an agent tomorrow?
[G] Four things the paper argues directly: report cost alongside accuracy, so a leaderboard win can't be bought with brute-force resampling. Separate benchmarks meant for scientific model comparison from benchmarks meant for a procurement decision, since a downstream developer cares about dollars in a way a model researcher shouldn't. Build held out test sets pitched at the actual generality a benchmark claims, and keep them secret. And standardize evaluation harnesses so two labs running the same benchmark are measuring the same thing.
[O] And the part I'd underline for evaluation research broadly, not just agents: the reproducibility section here is basically a template. Rerun the code, find where the bugs actually are, go confirm with the original authors before you publish the discrepancy.
[S] Agreed, that part I'd hold up as the standard, even where the headline finding is thinner than its citation count suggests.
[G] My one line: this is a paper whose most rigorous evidence, the reproducibility bugs, is close to established fact, and whose most quoted claim, that agent complexity rarely buys accuracy, is a genuinely useful but comparatively under-tested hypothesis the field has cited more confidently than the authors themselves ever claimed.
[O] Mine: even the deflationary parts of this paper are optimistic in the long run, because a field that starts measuring cost honestly is a field about to get a lot more useful, a lot faster.

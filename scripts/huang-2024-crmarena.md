[S] Here's a strange result to start with. Giving a language model better tools can make it perform worse.
[O] That sounds backwards. Tools are supposed to help.
[S] Not always. In this paper, handing a weaker model a set of carefully hand-built functions drops its score by double digits in places, while the exact same functions push a stronger model up by just as much.
[O] Now I want to know why. Let's get into it.
[O] Welcome to Litsearch Audio, where an optimist, a skeptic, and a visiting scholar take apart one paper from the litsearch site. Today it's "CRMArena: Understanding the Capacity of LLM Agents to Perform Professional CRM Tasks in Realistic Environments," from Kung-Hsiang Huang and colleagues at Salesforce AI Research, presented at NAACL twenty twenty-five.
[S] It's on the docket because it's an agent benchmark built around one very specific, underexplored question. Can a language model actually do somebody's job inside a real enterprise software system, not just click around a toy website.
[O] To help us dig in, we've got Lorne, a visiting scholar who's spent real time with this paper. Lorne, welcome to the show.
[G] Glad to be here. This is a benchmark paper with a genuinely clever workaround at its center, and a headline number that's more sobering than the framing might suggest on first pass.
[S] Start with the actual gap, Lorne. Why didn't a good benchmark for this already exist?
[G] The core obstacle is privacy. Customer Relationship Management systems, CRM for short, run on real customer data, names, purchase histories, support tickets, and no enterprise will hand that over for a benchmark. So prior agent benchmarks for work tasks invented their own toy environments instead. The paper names three: WorkBench, WorkArena, and tau-bench.
[O] And those weren't good enough because?
[G] Two reasons. First, the underlying data was too simple. WorkBench models five business objects with essentially zero dependencies between them. WorkArena gets to seven objects at point eight six dependencies per object. Real enterprise data doesn't look like that, a single case is linked to an account, a contact, an order, a product, and a knowledge article, all at once. Second, the tasks themselves were shallow, navigating a web page, filtering a list, not what a service agent or analyst actually spends a day doing.
[S] So the fix is just building a more complicated toy?
[G] Not quite, the fix is to stop building a toy at all. CRMArena generates synthetic data, but uploads it into an actual, live Salesforce organization, the same enterprise software real CRM teams run on. The agent operates inside real Salesforce, against sixteen interconnected business objects modeled on its own Service Cloud schema, at one point three one dependencies per object.
[O] And that's the trick that buys realism without touching a single real customer record.
[G] Right. Nobody's privacy is at risk, since the shoe company, the customers, the support cases, none of it is real. But the environment it lives in, and how the data connects, is exactly what a CRM professional would recognize. The paper's own comparison table is blunt about this: CRMArena is the only one of the four benchmarks compared that combines a real-world environment, realistic tasks, and expert validation, all three at once.
[S] Walk me through how you actually generate a fake company's worth of CRM data and have it hold together.
[G] Four stages. First, CRM data generation, gpt-4o synthesizes entries across all sixteen objects: accounts, contacts, cases, orders, products, knowledge articles. Second, that data gets uploaded into a real Salesforce sandbox, a Simple Demo Org. Third, query instances get generated against that data, natural-language questions paired with computable ground-truth answers. Fourth, an expert study checks the whole thing actually feels real to CRM professionals.
[O] Sixteen objects is a lot to keep internally consistent. How do they stop the model from generating a catalog that doesn't logically connect to anything?
[G] Two mechanisms. The first, object connectivity, generates objects so a case is genuinely tied to the account, the contact, and the agent that would touch it in a real system, mirroring Salesforce's schema directly. The second, more interesting, is latent variables, hidden factors that shape the visible data but are never exposed to the agent being tested. A SHOPPINGHABIT variable governs when and what a given customer buys, so purchase histories look coherent over time. A SKILL variable attached to each support agent determines whether that agent can resolve a case or has to transfer it.
[S] Why does that second one matter so much?
[G] Because without it, there'd be no principled reason for some cases to get transferred and others not, it would just be noise. The SKILL variable is what makes one of the nine tasks, Transfer Count Understanding, solvable at all. It gives the data an underlying causal structure, the way a real support organization has one, even though none of that structure is visible in the sandbox the agent actually queries.
[O] What stops the generation process itself from producing junk, duplicate products, nonsense fields?
[G] A dual verifier. A format verifier discards any mini-batch missing required fields. A content verifier checks task-specific feasibility, for the disambiguation task, it confirms a paraphrased product name is close enough to the original to be recognizable, but not confusable with something else the customer actually bought. Generation happens in batches of ten with a deduplication pass between them, since one call can't produce the twenty-thousand-plus price-book entries a full catalog needs.
[S] And the nine tasks themselves, what do they actually ask the agent to do?
[G] Three personas. A Service Manager cares about resource allocation, tasks like Handle Time Understanding, finding which agent resolves cases fastest, or New Case Routing, picking the best agent for an incoming case. A Service Agent handles individual interactions, Named Entity Disambiguation, figuring out which specific order a customer means by "the running shoes I bought," Policy Violation Identification, checking a transcript against company policy, Knowledge Question Answering, pulling the right answer from a knowledge base. A Service Analyst looks at trends, Top Issue Identification, Monthly Trend Analysis, Best Region Identification.
[O] That's a genuinely broad slice of what a CRM team does day to day.
[G] It is, and all nine came out of interviews with actual CRM professionals, not something the authors invented at a whiteboard.
[S] How many queries total, and how do they keep the agent from just pattern-matching?
[G] Fourteen hand-written seed templates, with placeholders for things like time period or product name, get filled in against the generated database, then paraphrased by an LLM for surface variety. That's one hundred thirty query instances per task, times nine tasks, eleven hundred seventy total. Thirty percent of the queries in five of the nine tasks are deliberately unanswerable, false-presupposition queries, like asking who transferred the most cases in a period when nobody transferred any. A well-calibrated agent has to answer "none," not confabulate.
[O] And the agent interacts with all of this how, exactly?
[G] Two tool regimes. General-purpose access, raw queries in SOQL and SOSL, Salesforce's own structured and fuzzy-search query languages, which can answer anything but require the agent to write a correct query from scratch. And task-specific access, twenty seven hand-crafted Python wrapper functions like "get the agent with the maximum case count," pre-built so the agent just calls the right one. Three agentic frameworks sit on top of both: Act, which predicts an action directly, ReAct, which adds a reasoning step first, and Function Calling, the structured protocol most production agents actually use.
[S] And the realism check, that's the expert study you mentioned earlier?
[G] Ten CRM professionals, recruited through a platform called User Interviews, all working inside Salesforce-style CRM daily. Each got sandbox access, five sampled query instances to actually attempt, then rated the environment against the real systems they use. Ninety percent rated the overall organization Realistic or Very Realistic, and more than seventy seven percent gave the individual objects, cases, accounts, knowledge articles, chat transcripts, similarly high marks.
[S] Okay. Give me the headline number, and don't round it up for me.
[G] Under ReAct, with only general-purpose query access, the best model anywhere in the paper is o1, at fifty seven point seven percent overall. Hand it the twenty seven crafted functions under Function Calling, and o1 climbs to sixty four point three percent. That's the ceiling, nothing else in the paper beats it.
[O] So even the best system available at the time fails more than a third of realistic CRM tasks.
[G] That's the paper's own framing. Below o1, the drop-off is steep. GPT-4o reaches fifty four point four percent under Function Calling, Claude three point five Sonnet gets forty one point eight percent, and the smallest model tested, Llama three point one at eight billion parameters, is essentially non-functional, two point five percent under Act, three point one under ReAct, zero under Function Calling.
[S] You said tools sometimes hurt. Walk me through that.
[G] This is the paper's sharpest finding. Function-calling tools help the strong models and actively hurt several weaker ones. The authors sort functions into four categories, query-only versus calculation, independent versus dependent on another function's output, and remove each category in turn. Pulling the calculation-plus-dependent functions costs GPT-4o twenty six point seven points. The exact same removal helps GPT-4o-mini by four points, and helps Claude-3-Sonnet by three point three points.
[O] So the same tool is scaffolding for one model and a landmine for another.
[G] Exactly that. A weaker model can't reliably chain a dependent function call correctly, so a tool built to make its life easier ends up producing more wrong answers than if it had just been left with raw queries.
[S] What about DeepSeek-R1? I'd expect a strong reasoning model to handle structured tools fine.
[G] That's the paper's odd case out. DeepSeek-R1 scores thirty five point one percent under ReAct, a respectable mid-pack number. Force it into Function Calling, and it collapses to nine percent, which the authors attribute to poor instruction adherence and a poor ability to incorporate execution feedback, it doesn't course-correct when a function call comes back with an error.
[O] Any good news on reliability, at least?
[G] Not really. They run ten repeated trials per task and measure pass at k, the probability that all k independent attempts succeed. All three frameworks degrade at almost exactly the same rate as k climbs from two to ten. None is meaningfully more consistent than the others, and none stays reliable across repeated attempts even where its single-shot accuracy looks decent.
[S] And cost?
[G] GPT-4o under ReAct is the cheapest configuration that's actually competent, about eighteen cents per query and five point four turns to finish a task, versus about twenty three cents and six point nine turns for Claude three point five Sonnet on the same setup.
[O] Let me make the strongest case for this paper, because I think it's underrated as methodology. Privacy was the wall stopping anyone from benchmarking real CRM work honestly. This paper doesn't get around that wall by giving up on realism, it builds a synthetic, schema-faithful, expert-validated stand-in, and the stand-in holds up. Ninety percent of actual CRM professionals looked at the sandbox and said it felt real. That's a template other regulated domains, healthcare records, financial services, could reuse directly.
[S] I'll grant the methodology is clever. But I want to push on how much weight that ninety percent number can bear, Lorne. Ten raters, one recruiting platform, five queries each, and the paper never publishes an inter-rater agreement statistic. Was that a tight consensus, or a couple of enthusiastic raters dragging up the average?
[G] That's a fair gap, unaddressed in the paper. The rating protocol itself is solid, real sandbox access, task completion before the realism rating, not just a screenshot judgment. But without an agreement statistic, "ninety percent said realistic" could hide more disagreement than it shows.
[O] Fine, point taken. But even granting that, the core empirical finding, function calling isn't a uniform upgrade, doesn't depend on the expert study at all. That's a result about model behavior, and it's actionable today for anyone actually building a CRM agent.
[S] Agreed, that part I trust. What I trust less is the self-preference angle nobody tests. The synthetic case notes, the knowledge articles, the whole database, all of it is generated by gpt-4o, the paper says so directly, for cost reasons. So when GPT-4o turns in the strongest non-reasoning score under Function Calling, how much of that is task-solving skill, versus the text simply sounding like GPT-4o's own writing?
[G] The paper doesn't test that, and it's worth being direct about it, that's a real, unaddressed confound. It's subtler than contamination, classic answer leakage is structurally ruled out since the organization is generated fresh, nothing scraped from a pretraining corpus. But stylistic self-preference is a different mechanism entirely, and nothing in the design rules it out.
[O] What about scope? Is "professional CRM tasks" as broad as the title makes it sound?
[G] No, the paper's own limitations section says so directly. It covers three personas, service manager, agent, and analyst, and explicitly excludes sales representatives, a huge chunk of real CRM usage. This is customer-service CRM specifically, not the full discipline.
[S] I'd add the model roster is already dated. This is a November twenty twenty-four benchmark, nothing in the GPT-5, Claude four, or Gemini two point five class is anywhere near this table, because those didn't exist yet. The leaderboard numbers are a snapshot, not a durable ceiling.
[O] Sure, but the qualitative finding, tools help strong models and hurt weak ones, is exactly the kind of result that should generalize across model generations, even if the specific percentages don't.
[G] That's the right way to read it. I'd treat sixty four point three percent as a floor for what agents could do in November twenty twenty-four under this exact tool set, not the ceiling of the task itself. Nobody's measured what a human expert scores on these same eleven hundred seventy queries, so there's no upper anchor to compare it against.
[S] One more thing worth naming. Knowledge Question Answering is graded with F1 against a reference answer, which is a lexical-overlap metric. It'll penalize a correct answer that's phrased differently from the reference, and the paper doesn't report a human spot-check confirming F1 actually tracks correctness there the way exact match cleanly does for the other eight tasks.
[G] Agreed, that's a real measurement-validity gap, and a fixable one, a small human-judged sample on Knowledge QA would settle it either way.
[O] So stepping back, what should actually change because of this paper?
[G] Two things. First, anyone deploying a function-calling agent should test it with and without the hand-built tools before assuming tools are strictly better, this paper shows that assumption breaks for weaker models in a specific, quantifiable way. Second, for the next generation of these benchmarks, the self-preference question deserves a direct test, generate the same environment with two different model families and see whether results move.
[S] It matters for benchmark design more broadly, too. Grounding in an actual production system is a stronger evidence base than a hand-rolled simulation, but only if the thing generating the data isn't also one of the things being scored on it.
[G] And notably, the authors' own follow-up work, released some months after this paper, does extend the persona coverage they flagged as a limitation here, adding more roles and multi-turn interactions, which suggests they took that critique seriously themselves.
[S] Good note to close the discussion on.
[G] If there's one sentence to take from the paper itself, it's this. Real Salesforce sandbox, sixteen interconnected objects, nine expert-derived tasks, and even the best agent available at the time cleared only sixty four point three percent, with the tools meant to help sometimes making a weaker model worse.

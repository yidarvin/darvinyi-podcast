[O] And here is the twist. Hand a human annotator two to six lines to fix, and one of those same zero percent models jumps to sixty percent correct. That is not a broken model, that is a broken measurement.
[S] Or it is proof the vanilla number was measuring the wrong thing all along, which means we should stop trusting single shot leaderboards for coding tasks quite so readily.
[G] Both of those are live readings of the same result, and untangling them is basically the whole paper.
[O] Welcome to Litsearch Audio, where an AI optimist, an AI skeptic, and a visiting scholar take apart one paper from litsearch dot darvinyi dot com. Today's paper is MultiFileTest, subtitled A Multi File Level LLM Unit Test Generation Benchmark and Impact of Error Fixing Mechanisms, from Yibo Wang and colleagues at the University of Illinois Chicago, Salesforce Research, and Scale AI.
[S] It is on the docket because it goes after a real blind spot. Almost every unit test benchmark tests a model on one function or one file, and almost no real codebase looks like that.
[G] Glad to be here. I have spent a lot of time with this paper's tables, and I think the three scenario design is the most interesting methodological move in recent test generation evaluation.
[O] Great to have you, Julian. Let's start with the gap this thing is actually filling.
[S] So Julian, what is wrong with the existing benchmarks, HumanEval, ClassEval, that whole family?
[G] They are all scoped too small. HumanEval hands a model one function. ClassEval scopes to one class. TestEval is function level too. TestGenEval widens to a single file, which is progress, but the code under test still fits on one screen with no dependencies to chase.
[O] And real code does not work that way.
[G] Right, the authors give a clean example. A function in stock dot py imports a class from structure dot py, which itself depends on validate dot py. To even produce a test that runs, the model has to resolve the right import paths across three files, not just reason about the logic in one.
[S] Was nobody testing multi file code before this?
[G] One benchmark touched it, DevBench. But DevBench was built for breadth across general software development tasks, not depth in test generation specifically. The authors point out only about sixteen of its projects are genuinely multi file for unit testing, quality is uneven, and a lot of those projects are hard to even trace back to their source repository.
[O] So MultiFileTest's pitch is the same idea, but built from scratch to be a real unit test benchmark, at triple the scale.
[G] Sixty projects against DevBench's sixteen, and every single one is required to have real cross file dependencies. That is the design principle. Multi file reasoning is not an occasional feature of the dataset, it is guaranteed in every project.
[S] Walk us through how the dataset actually gets built, Julian.
[G] Sixty GitHub projects, twenty each in Python, Java, and JavaScript, chosen under three rules. Appropriate size, two to fifteen files, under sixteen hundred lines of code, so the whole thing fits in a model's context window without truncation. Real inter file dependencies. And reliable provenance, meaning high stars and forks, permissive licenses.
[O] Why cap it that small? That sounds like it is dodging the hardest version of the problem.
[G] The authors are upfront about that trade off. Let project size grow toward production scale, and you stop measuring test generation ability and start measuring retrieval and long context navigation instead, which of fifty files actually matter. Capping it at a handful of files keeps those confounds out and isolates the thing they actually want to measure, can the model reason correctly across a few files.
[S] That is a deliberate, defensible choice, but it does mean multi file here means something narrower than most listeners would picture walking in.
[G] Fair, and I will come back to that. Average project size is under five files, six point one for Python, four point eight for Java, four for JavaScript.
[O] So what actually happens to a model in this benchmark?
[G] Each model gets the whole project plus a language specific, zero shot prompt, temperature zero, and has to emit a full test suite for every file. Those suites are then genuinely executed, pytest for Python, JUnit with JaCoCo for Java, Jest for JavaScript, not just graded by another model reading the code.
[S] What gets measured once it runs?
[G] Four metrics, and this is the part that matters most. Executability rate, does the suite even compile and run. Correctness rate, of the tests that exist, how many pass against the reference implementation. Coverage, line and branch. And a fourth one they introduce themselves, unique contribution, which measures how much coverage each individual test adds that no other test in the suite already covers. That last one guards against a model inflating its coverage number by writing forty near duplicate tests.
[O] And the key structural fact is these four are not independent.
[G] Exactly, they are chained. If a suite has an executability error, say a hallucinated import that throws a module not found error before a single test collects, correctness and coverage both collapse to zero for that entire project, even if the test logic underneath was actually fine.
[S] Which is the whole reason they do not stop at one number.
[G] Right, that is the paper's real contribution, three scenarios run on the same vanilla output. Scenario one is vanilla, exactly what the model wrote, no changes. Scenario two is manual fixing, where CS PhD annotators apply the minimal mechanical edits needed to clear executability and what they call cascade errors, following an objective checklist, while preserving every original assertion. Scenario three is LLM self fixing, where the model is handed its own error messages and the conversation history and asked to repair its own output.
[O] What is a cascade error, as opposed to an executability error?
[G] Executability stops the suite cold, nothing runs. Cascade lets the suite run, but one root cause takes out a bunch of tests at once. Their example is a missing numpy import. The fix is one line, but until it is added, every test that touches that dependency fails, even ones that were logically correct.
[S] I want to push on that minimal, mechanical framing though. Who decides what counts as mechanical?
[G] That is the constraint the protocol tries to enforce. Annotators are instructed to target only errors with an objectively determinable fix, the right import path can be read straight off the file structure, for instance. They are not allowed to rewrite test logic or improve a weak assertion. That is what is supposed to let the manual fix number stand in for potential, rather than a better test writer did this instead.
[O] Let's get to the actual leaderboard. Who wins on vanilla output?
[G] Eleven models total, seven closed, from GPT three point five Turbo up through GPT o1, GPT five mini, both Gemini generations, and Claude three point five Sonnet, plus four small open models, CodeQwen1.5, DeepSeek Coder, CodeLlama, and CodeGemma, all around seven billion parameters. On vanilla output, Gemini three Pro is the strongest overall, seventy seven percent correctness on Python, eighty five percent executability. GPT o1 actually leads on JavaScript, eighty seven percent correctness.
[S] And Java?
[G] Java is where everything falls apart. Gemini three Pro is still best there, but only at sixty two percent correctness. Three of the four small open models, CodeQwen1.5, CodeLlama, and CodeGemma, score exactly zero. Not low, zero executability, zero correctness. They cannot produce one runnable Java test.
[O] Why is Java uniquely brutal?
[G] Stricter syntax. Package declarations, mocking frameworks, type matching, there are just more ways to fail before you even reach the logic of the test.
[S] Okay, so that is vanilla. Now the manual fix numbers, because this is where the paper gets interesting.
[G] This is the headline result. Manual fixing forces executability to one hundred percent everywhere, by construction, and then correctness moves with it. The gains are largest exactly where vanilla was worst. CodeQwen1.5 on Java goes from zero to sixty, a sixty point jump, from two to six lines of hand applied fixes. CodeGemma goes from zero to fifty seven. CodeLlama, zero to thirty six. It is not just the open models either. GPT o1 on Python climbs from sixty to eighty nine. Claude three point five Sonnet climbs from sixty four to ninety two. Gemini three Pro climbs from seventy seven to ninety.
[O] And the rankings actually reorder.
[G] They do, and that is the point the authors lean on hardest. CodeQwen1.5 goes from a model that cannot produce a single working Java test to roughly matching GPT four Turbo once both are fixed. The vanilla leaderboard was partly measuring who avoids a trivial import bug, not who understands the codebase well enough to test it.
[S] What about the self fixing scenario? Can models close that gap without a human?
[G] Mixed, and it splits cleanly by model class. Closed models mostly can. Gemini three Pro self fixes to ninety one percent correctness on Python, essentially matching its own manually fixed score of ninety. Claude three point five Sonnet reaches eighty six, a bit below its manual fix number of ninety two. But self fixing almost never hits one hundred percent executability the way manual fixing does by definition. Claude lands at ninety percent executability after self fixing, not one hundred.
[O] And the small open models?
[G] They collapse. DeepSeek Coder's Java executability falls to twenty percent after attempting self repair, down eighty points from its manual fix baseline. CodeLlama's Python executability drops to five percent. A lot of the time these models do not even produce corrected code, they write prose describing how they would fix it, which obviously does not compile.
[S] Any exception worth flagging?
[G] One good one. On JavaScript, CodeQwen1.5 actually self fixes better than it gets manually fixed, a twenty three point gain over manual. Its vanilla failure was not a code bug at all, it misread the prompt and emitted no tests. A human doing mechanical fixes cannot invent tests from nothing, that is outside the protocol. But an error message saying no tests found is exactly the kind of signal that can prompt the model to try again correctly.
[O] What about that fourth metric, unique contribution? Does anyone actually write efficient tests?
[G] Not really, and it is a nice sanity check on the whole leaderboard. GPT o1 has higher raw coverage than GPT five mini, but a much lower unique contribution score, six point seven five versus fifteen point one, because it is writing thirty six tests per project against GPT five mini's twelve or so. It is buying coverage with volume, not precision. Gemini three Pro posts the best combination of the group, highest coverage and a strong unique contribution score, meaning fewer of its tests are redundant.
[O] Alright, time for the actual argument, and Julian is refereeing. My case first. This paper does something genuinely useful that most benchmarks do not, it separates the model cannot do this from the model tripped on something trivial. A sixty point swing from a two to six line fix is not noise, that is a real signal that raw capability is being hidden by shallow mechanical failures. And the fact that closed models mostly self fix almost as well as a human tells you that gap is closing on its own, without anyone hand holding.
[S] I will grant the diagnostic value. What I do not grant is that the exact rankings deserve much confidence. Twenty projects per language is small, every cell in those tables is an average over twenty heterogeneous repositories. A gap between two models, or the difference between a plus thirteen and a plus twenty nine improvement, sits well inside the kind of noise you would expect from a sample that size. And their own robustness check only tests that claim with one model, GPT three point five Turbo, on one language, Python.
[O] They do run a statistical test across models and scenarios though, significant with a p value under one percent and a large effect size.
[G] True, and worth being precise about what that test does and does not show. It is a paired comparison of correctness rate across scenarios, vanilla against manual, manual against self fix, pooled across nine of the models and all three languages. That is solid evidence that fixing systematically helps, in aggregate. It is not the same as certifying any individual model versus model gap in the main results tables, which is exactly where that concern bites.
[S] Right, that is exactly my point. And I have got two more. First, contamination. These are high star, high fork public GitHub repositories, precisely the kind of code most likely sitting in a model's pretraining data already. There is no decontamination check anywhere in this paper, no held out fresh split, no memorization probe. The authors do reorganize files and rename things to make projects self contained, which might perturb memorized content a little, but that is a side effect, not a measured defense.
[O] That is a fair miss. What is your second one?
[S] The manual fix scenario is the entire spine of the paper's potential story, and its reproducibility is never quantified. Minimal, mechanical, objectively verifiable is a good protocol on paper, but whether a given edit counts as mechanical versus quietly improving the test is a human judgment call made by the annotators, and there is no inter annotator agreement number reported anywhere.
[G] Both of those land, and I will add a third gap while I am at it. There is no test strength metric here at all. Correctness means a test's assertion matches the reference implementation's output, it says nothing about whether that test would actually catch a real bug if one were introduced. No mutation score, no fault detection number, the way some prior test generation work reports. A model can score high correctness with an assertion that could never fail. There is also a smaller, telling data point buried in the appendix. They compare a naive self fix against HITS, an existing iterative debugging method with method slicing, on the same GPT three point five Turbo model. HITS reaches seventy five percent correctness on Java. Their own self fixing only reaches seventeen. A more sophisticated repair loop clearly leaves a lot on the table that single turn self fixing does not capture.
[O] So where does the needle land, Julian?
[G] The qualitative story is trustworthy. Java is the hardest language, a small number of hand applied lines unlocks large gains, and small open models cannot reliably self repair the way closed ones can. Those patterns show up consistently across every table, and they are the kind of finding that would likely survive a bigger sample. What I would not do is treat a small gap between two mid table models as meaningful, and I would not treat the vanilla numbers as contamination free. Read it as a diagnostic, with those caveats attached, not as a definitive leaderboard.
[O] So what does this actually change, if you are building or grading a coding agent?
[G] The practical lesson is, do not let one brittle failure mode, an import path, a missing mock, mask everything downstream in your evaluation. If you are scoring an agent end to end, separate did it understand the task from did it handle environment friction, the way this paper's three scenarios do, or you will systematically underrate models that are one small fix away from working.
[S] And for anyone building the next version of this benchmark, run the robustness check across the full leaderboard, add an agentic or iterative baseline so the zero shot ceiling has some context, and put a contamination check in before publishing a ranking anyone is going to cite.
[O] Agreed on all three, and I would add, this is exactly the kind of unglamorous benchmark work that ages well, because the finding that most test generation failure is trivial and fixable is more durable than any single model's score.
[G] If I had to leave you with one sentence, the authors show that a lot of what looks like an LLM failing to understand code is actually an LLM failing to import it correctly, and once you separate those two things, the real capability gap is smaller than the vanilla leaderboard suggests.
[O] My one sentence, the fact that closed models can mostly self fix their own tests, nearly matching a human, is the most optimistic finding buried in this paper.

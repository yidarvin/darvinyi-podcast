[O] So two concrete design choices follow from that: organize the training data at the repository level instead of the file level, and bake a fill in the middle objective into pretraining instead of treating it as an afterthought.

[S] Walk me through the repository-level piece first, because "we sorted the files" undersells what's actually happening.
[G] They start by crawling public GitHub repositories created before February of twenty twenty-three, keeping only those eighty-seven languages, then run StarCoder-style rule filters, line-length limits, alphabetic-character ratios, which alone cuts the raw data down to about thirty-two point eight percent of its original size. Then, for each repository, they parse the invocation relationships between files with regular expressions, import in Python, using in C sharp, include in C, build a dependency graph, and run a topological sort, so a file always appears after the files it depends on. Cyclic imports break a standard topological sort, so they modify it to pick the file with minimal, not necessarily zero, in-degree, letting it push through cycles instead of getting stuck.
[O] And then the whole ordered repository gets concatenated into one training example.
[G] Right, with a comment carrying the file's path prepended to each file, so that information survives into the model. [G] An n-gram filter. [O] Now the other half of the recipe, fill in the middle. What's actually being trained here?
[G] Take a file, split it into a prefix, a middle chunk, and a suffix, then reorder them so the model sees the prefix, then the suffix, and has to generate the middle in between, wrapped in sentinel tokens marking where each piece starts and ends. [S] Why fifty percent, and not, say, always?

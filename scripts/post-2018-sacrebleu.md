[S] Cited more than three thousand five hundred times, and sacreBLEU, the tool it introduces, is close to the default way machine translation results get reported today.
[O] So let's figure out why a few pages of bookkeeping earned that kind of staying power.
[S] Maya, set the scene. What was actually broken before this paper?
[G] Machine translation had settled on BLEU as its default scoring mechanism, going back to Papineni and colleagues in two thousand two. It's cheap to compute, roughly language-independent, and correlates reasonably with human judgment, so for a decade and a half it became the number every paper reported to claim progress.
[O] The problem is "BLEU" was never really one number.
[G] Right, and Post's point is that almost nobody had said so out loud. BLEU is a parameterized metric. There's the number of reference translations, how the length penalty gets computed with more than one reference, the maximum n-gram order, and how you smooth zero counts. Most of those rarely matter in practice, the max order is basically always four, and there's usually only one reference.
[S] So that's not actually where the damage is.
[G] No. The real damage sits one layer beneath that: how the reference translation itself gets preprocessed before scoring. Tokenization, normalization, lowercasing, splitting compound words. Because BLEU is a precision metric over n-grams, changing how you tokenize the reference changes the entire set of n-grams the system is being graded against.
[O] Which means two papers can both say "BLEU: twenty-eight," and be scoring against two different definitions of what counts as a matching word.
[G] Exactly, and Post shows papers almost never say which choice they made. He tries to reverse-engineer the configuration used by seven well-cited papers, and in most cases he simply can't. Even Vaswani and colleagues, the original Transformer paper, gets logged in his table as "user, or user-lowercase, unclear."
[S] That's the Transformer. The paper practically everything in modern NLP traces back to, and its own BLEU configuration can't be pinned down from the text.
[G] That's the example Post leans on hardest, precisely because it's so foundational. It's not just tokenization, either. Even the dataset can be ambiguous. The WMT fourteen English-to-German test set exists in two versions: one released for the actual evaluation with two thousand seven hundred thirty-seven sentences, and a restored version released afterward with three thousand four. Researchers routinely don't say which one they scored on.
[O] So you've got an underspecified metric, applied to an underspecified dataset, reported with no way to check either.
[S] Okay, walk us through the actual fix. What is Post proposing?
[G] The conceptual move is disarmingly simple. He draws a line between user-supplied and metric-internal processing. People talk about scores being "tokenized" or "detokenized," but that's a confused way to describe it, for BLEU, both the system output and the reference are always tokenized somehow. The real question is who does the tokenizing: the researcher's own pipeline, or the scoring code itself.
[O] And his answer is: never the researcher, at least not for the reference.
[G] Right. The user is free to do whatever they want to their own system's output, decode it, postprocess it, however they like. But the reference has to flow straight into the metric untouched, and the metric applies its own fixed preprocessing to both sides before scoring. If two people run the same metric code on the same reference, they get the same number, full stop.
[S] That sounds obvious in hindsight.
[G] It is, and Post is explicit that this isn't a new algorithm. He borrows the idea from an older, more disciplined community: parsing. Parser quality has been comparable across papers for almost thirty years because of PARSEVAL and a shared codebase called evalb. There's a parameter file, COLLINS dot P-R-M, that settles every annoying edge case once, for everyone: don't count the root node, don't count empty nodes, don't count punctuation, treat ADVP and PRT as equivalent labels. Nobody re-litigates those choices paper to paper, because the tool already answered them.
[O] So the pitch is: do for BLEU what evalb already did for parsing thirty years ago.
[G] Exactly. And the frustrating part is machine translation already had the raw ingredient, WMT's own official scoring script does use metric-internal preprocessing. But every version people actually reached for had a flaw. Moses' scoring script demands the user tokenize first. The WMT script needs the input wrapped in clunky XML. Nematus has a detokenized variant, but it still hands the reference to the user to manage.
[S] So the tool this paper ships, sacreBLEU, is basically closing that last gap.
[G] That's the whole thing. sacreBLEU is a small Python package. It ships and automatically downloads the standard test sets itself, all of WMT from two thousand eight through two thousand eighteen plus recent IWSLT sets, so the user never has to touch a reference file at all. You feed it your detokenized system output, tell it the test set and the language pair, and it applies WMT's own internal tokenization and hands back the official number. And it prints a short version string that records every choice it made, case handling, language pair, reference count, smoothing, test set, tokenizer, and the tool's own version, meant to be pasted directly into your paper's footnotes.
[O] One line, and suddenly your BLEU score is reproducible by construction.
[G] That's the pitch. It's released under an Apache license, and Post also folds in chrF, a character-level metric, as an option, so the same protective-layer idea covers more than just BLEU.
[S] Let's get to the actual numbers, because I want to know if the "up to one point eight" headline holds up under its own table.
[G] It's a controlled little experiment. Post takes exactly one system, the "online-B" submission from WMT seventeen, and scores its output four different ways across twelve language pairs, holding the system itself completely fixed. Basic user-style tokenization with the Moses tokenizer. A version that also splits compound words. A version that maps rare words to an UNK token, meant to simulate a plausible mistake. And the metric-internal WMT tokenization.
[O] And the spread across those four options is the whole argument.
[G] It is. The biggest single gap is one point eight BLEU, on German-to-English, and the average gap across all twelve pairs is close to one point oh, and that's excluding the UNK column entirely, which is treated separately as the most extreme case. On German-to-English specifically: basic tokenization gives thirty-one point two, adding compound-splitting gives thirty-one point seven, and the metric-internal tokenization gives thirty-three point oh. Same system, same output, three different numbers.
[S] And Post is explicit that casing is actually the single biggest lever in there.
[G] Right, cased versus uncased scoring, which a lot of papers just never specify. And these are gaps that regularly exceed what a new method claims as its improvement.
[O] Then there's the reference-count trap, which I think is almost sneakier than tokenization.
[G] It is, because it doesn't even touch preprocessing. Simply changing how many reference translations a test set carries reshapes the whole scale. That same online-B system, scored on English-to-Finnish: with one reference it gets twenty-two point oh four; with two references, twenty-five point two five. That's a jump of about three point two points from adding a second human translation, nothing else changed. And the NIST Arabic-to-English and Chinese-to-English sets, which carry four references each, land systems in the high forties, and now the low fifties. All three of those numbers get reported under the same word, "BLEU," with nothing distinguishing why they sit so far apart.
[S] So a newcomer skimming papers could easily read a Chinese-to-English score in the fifties as evidence that direction is just "easier," when it might mostly be an artifact of how many references the test set happens to carry.
[G] That's precisely the trap Post is naming. And then there's the audit I mentioned earlier, his attempt to recover the actual configuration behind seven well-cited papers. Chiang two thousand five reports a determinable metric-internal, lowercased setup. Bahdanau and colleagues is simply unclear. Luong and colleagues twenty fifteen is user-supplied or metric, unclear which. Wu and colleagues, Google's neural translation system, and Vaswani and colleagues, the Transformer, both land as "user, or user-lowercase, unclear."
[O] Two of the most cited papers in the field's history, and you can't tell from the text which BLEU they're reporting.
[G] That's the paper's closing argument, really. It's not presented as a knock on those authors individually, Post even has a footnote admitting his own research pipeline made this exact mistake once, affecting a score by about half a point. The point is that the incentive structure of the field made this kind of silent inconsistency the norm, not the exception.

Without it, none of the benchmark scores downstream are actually comparable across papers in the first place.
[O] Maya, be honest with me, though. Is a version string and a pip-installable wrapper really one of the field's great papers, or is this just very well-executed formatting advice?
[G] I'd resist "just." Formatting advice doesn't get cited three thousand five hundred times, and it doesn't get every major MT paper since to change its footnotes. But I'll grant the deeper limitation, because Post states it himself, almost as an aside: this paper is explicitly "not concerned with the shortcomings of BLEU as a proxy for human evaluation." He cites the Callison-Burch critique of BLEU directly and then declines to engage with it. sacreBLEU makes BLEU reproducible. It does nothing to make BLEU a better proxy for what a human translator would actually judge as quality.
[S] And that's the part that worries me more than anything in the tables. By making BLEU the clean, citeable, one-line-install default right as more human-correlated learned metrics, BLEURT, COMET, were starting to appear, this paper may have quietly extended BLEU's dominance rather than ending it.
[G] That's a real cost, and a reasonable reading with the benefit of hindsight, though I'd call it a consequence rather than something the paper set out to cause. There's a second cost worth naming too: the fix works by centralizing trust. sacreBLEU becomes the canonical distributor of what the reference translations even are, which is convenient, but it's also a single point of failure. And the tokenization it standardized by default in twenty eighteen, called "13a," is essentially English-centric. Making comparisons fair for Chinese, Japanese, and other non-whitespace-delimited languages required adding language-specific tokenizers in later versions of the tool, so "metric-internal" itself turned out to be its own parameter choice, not neutral ground truth.
[O] So: real, lasting fix for one specific failure mode, comparability. Not a fix, and arguably a mild setback, for the deeper question of whether BLEU should be the number at all.
[G] That's the honest split. It won decisively on the problem it actually set out to solve.
[S] Zooming out, why should anyone who isn't doing machine translation care about this?
[G] Because the failure mode generalizes completely. Any time a metric has hidden preprocessing choices, a benchmark has silent dataset versioning, or a judge model has an undocumented prompt template, you get the same disease: numbers that look comparable because they share a name, but aren't. The instinct this paper models, print a version string so anyone can reconstruct exactly how your number was produced, is the same instinct behind good contamination reporting and eval-harness versioning today.
[O] It's basically a preview of the reproducibility fights we're still having now, just with LLM judges and agentic benchmarks standing in for tokenizers.
[G] If there's one line to take from this paper, it's that "the BLEU score" was never one number, and the fix wasn't a better metric, it was taking the reference translation out of the researcher's hands entirely.

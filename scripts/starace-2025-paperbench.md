[G] Glad to be here. This one's unusual because the hard part isn't the task, it's proving the grading. Replicating a paper from scratch is easy to ask for and brutally hard to score fairly, and most of this paper's real engineering went into the second half of that sentence.
[S] Imelda, start with why this needed to exist. We already have benchmarks with agents writing code. What's missing?
[G] All of those measure something narrower than research. CORE-bench hands the agent the paper's own repository and asks it to reproduce results from that code. MLE-bench, MLAgentBench, and DSBench put agents on Kaggle competitions, often dated and not representative of a live research paper. RE-bench is closer in spirit, seven hard, hand-crafted engineering tasks, but each ships with a scoring function giving a clean, perfect signal on one narrow objective. None of that tells you whether an agent can take a paper it's never seen code for and rebuild the whole thing.
[O] So the test PaperBench wants is the one every advisor actually gives a new PhD student: here's a paper, go make it work, from nothing.
[G] Exactly that framing, and the authors are explicit it connects to model-autonomy tracking. They cite it as a measure of the capability that shows up in OpenAI's own Preparedness Framework, Anthropic's Responsible Scaling Policy, and Google DeepMind's Frontier Safety Framework. If a model can read a paper cold and build a working implementation, that's a genuine research accelerant, and a capability worth watching before it arrives rather than after.
[S] Fine, but say more about why nobody built this already. Grading "did you replicate the paper" sounds like it should just be a checklist.
[G] It sounds that way until you try to write it. A single ICML paper makes dozens of intertwined empirical claims. A pass or fail number is far too coarse for partial progress, and an LLM eyeballing the whole submission for a holistic score from zero to a hundred is unreproducible and unauditable, you can't tell what it actually checked. The paper's real bet is that the grading machinery, not the task description, is where almost all the hard engineering had to go.
[O] Imelda, walk us through the actual task. What does the agent see, and what does it have to hand back?
[G] Twenty ICML twenty twenty-four Spotlight and Oral papers, spanning twelve topics, deep reinforcement learning, probabilistic methods, robustness, and more. The agent, called the candidate, gets the paper as PDF and Markdown, plus an addendum of author clarifications, plus task instructions. It never sees the rubric it's graded against, so it has to infer what "replicate this" requires. And it's barred from the original authors' codebase: a per-paper blacklist covers the authors' repo and any other online replication, enforced by a monitor scanning agent logs for blacklisted URLs. They caught ten violations across six hundred forty-six total runs and zeroed those submissions out.
[S] So it has to write everything from scratch, blind to the actual code. What does it have to hand back?
[G] A full repository with one required file at the root, called reproduce dot S H, the single entrypoint for running every experiment. Here's the detail worth dwelling on: the agent's own run and the scoring run are separated. Once the agent's time is up, its submission gets copied to a fresh virtual machine, Ubuntu twenty-four oh four with an A10 GPU, and the reproduce script runs there from a clean start. That stops an agent from hard-coding results and calling it a reproduction, and it's why the paper can call this a replication score rather than a self-report.
[O] Now the part I actually think is the real contribution here, the rubric.
[G] Right, this is where most of the labor went. Each paper gets a rubric that's a tree: the root is something like "the core contributions have been reproduced," decomposing downward until each leaf is something an expert familiar with the paper could check in under fifteen minutes. Across all twenty papers, that's eight thousand, three hundred sixteen leaf nodes. Crucially, every rubric was co-developed with one of the paper's original authors, drafting, review, iteration, sign-off, over multiple weeks. The authors call this the single most time-intensive part of the whole benchmark, taking each creator several full days at minimum just to draft one.
[S] What actually gets graded at each leaf?
[G] Three types, worth holding onto. Result Match is hardest and strictest: did the executed submission reproduce a specific number, graded only from the reproduce script, its log, and the output files, deliberately not the source, so you can't get credit for code that looks right but never ran. Execution asks whether some specific step happened when that script ran. Code Development, the loosest, asks whether the source appears to correctly implement a requirement, run or not. A parent's score is the weighted average of its children, rolling up to one Replication Score per paper.
[O] And since a rubric with hundreds of nodes per paper is way too much to hand-grade at scale, that's where the LLM judge comes in.
[G] Right, SimpleJudge. For each leaf independently, it's handed the paper's Markdown, the full rubric as JSON, that leaf's requirement, and the submission. A whole codebase rarely fits in context, so the judge ranks files by relevance and keeps only the top ten. The backend is o3-mini at high reasoning effort, about sixty-six dollars per paper in API credit. To check whether that judge can be trusted, they built a second benchmark, JudgeEval, from hand-graded partial replications of five papers, built from scratch or by modifying the original authors' own code, which the authors note is often buggy or incomplete and never ships a reproduce script itself. Against those human labels, o3-mini scores an F one of point eight three at sixty-six dollars a paper; o1 edges it out at point eight four but costs eight hundred thirty dollars, so o3-mini is used for the headline results. There's also a stripped-down variant, PaperBench Code-Dev, skipping reproduction and grading only Code Development leaves, no GPU needed, around ten dollars a paper.

[G] The authors dug into the logs to understand why. Every model except Claude three point five Sonnet frequently ended its run early, declaring it had either finished or hit a problem it couldn't solve, when in most cases it plainly hadn't finished. All the models wrote fluent, sensible multi-step plans; none reliably executed those plans across the full twelve hours, and o3-mini in particular struggled with basic tool use. [O] Where does this connect to the broader evaluation-practice conversation you two care about?
[G] A few lessons that transfer generally. First, validate an automated judge on the submissions your leaderboard actually reports, not a separate proxy set, that's the JudgeEval gap on the table, and it recurs anywhere an LLM grades LLM output at scale. Second, a single scalar score over a rich rubric can quietly blend requirement types that measure very different things, "looks correct" versus "actually works," so report the breakdown, not just the roll-up. Third, this sits inside a live policy conversation, framed as one input into autonomy tracking for frontier labs' own safety frameworks, so the measurement discipline isn't academic.

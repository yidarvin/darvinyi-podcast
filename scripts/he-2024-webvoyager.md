[S] Picture an autonomous agent that drives a real browser on the real internet, clicks around Amazon or Apple dot com, and then the exact same model that just did the driving turns around and grades its own performance.
[O] And yet it works remarkably well — human judges agree with that self-graded verdict eighty-five percent of the time, which is roughly how often two human graders agree with each other.
[S] Convenient, isn't it, that the grader and the driver are practically the same brain.
[O] That tension — a genuinely useful new evaluator riding along with a genuinely impressive new agent, both built from the same model family — is basically today's whole episode.
[O] Welcome to Litsearch Audio, where an optimist, a skeptic, and a visiting scholar take apart one paper from the frontier of AI research. Today it's WebVoyager: Building an End to End Web Agent with Large Multimodal Models, by Hongliang He and colleagues at Zhejiang University, Tencent AI Lab, and Westlake University, published at ACL twenty twenty-four.
[S] It's on the docket because it's one of the first papers to put a multimodal model behind the wheel of a live, unmodified website instead of a toy simulator — and because the scoring method it invents became standard practice almost overnight.
[O] And to help us dig in, we've got Juno, a visiting scholar who's spent real time with this paper. Welcome, Juno.
[G] Glad to be here. This paper quietly reset what "web agent" means, so there's a lot to get into.
[S] Juno, set the scene. What was wrong with web agents before this paper?
[G] Two simplifications, really. First, modality: agents like WebGPT and WebAgent read the page as text — raw HTML or an accessibility tree — never the rendered image a person looks at. WebGPT fine-tuned GPT-3 inside a text-only browser; WebAgent pretrained a T5 just to extract HTML snippets. Most of the engineering went into taming verbose markup, not understanding the page.
[O] Which is strange, because a webpage is designed to be looked at. A calendar picker or a flight search form encodes meaning in its layout that a flattened HTML string just buries under thousands of tokens.
[G] Exactly the authors' argument. The second simplification was the environment. Agents were tested in stripped-down simulators like MiniWoB, or against a single frozen "golden" trajectory the way Mind2Web does it — match the one recorded path or fail, even if a different route also reaches the right answer.
[S] So you get a clean benchmark number that tells you almost nothing about whether the thing survives an actual website, with pop-ups, ads, and content that changes between the time you write the test and the time you run it.
[G] Right, and that's the gap WebVoyager targets: an agent that perceives the rendered screenshot, not the markup, and runs end to end on the open web with nobody in the loop. But that also destroys your ability to score against a golden path, because there isn't one — so the paper has to solve perception and evaluation together.
[O] Two problems, one paper. Let's get into how they built the thing.
[O] Juno, walk us through the actual agent.
[G] At its core it's a perception-action loop around a frozen large multimodal model — mainly GPT-4 Turbo with vision, with Claude 3 Opus and GPT-4o tested as alternate backbones. Every step, the agent gets a screenshot plus auxiliary text, writes a thought, and emits one action, which a Selenium-controlled browser executes. The page becomes the next observation, and the loop continues until the model says it's done or hits a fifteen-step budget.
[S] Fifteen steps total, and every failed action or thrown exception eats into that budget too — the paper folds the error message back into the prompt and burns a step to retry.
[G] Correct, and that turns out to matter a lot later, when we get to why it fails.
[O] Okay, but the headline trick here — the thing that actually made screenshots usable — what is it?
[G] Set-of-Mark prompting. A rule-based tool called GPT-4V-Act walks the page's document structure, finds every interactive element, and overlays a black-bordered box with a number on it. The model never has to output pixel coordinates; it just says "click seventeen," and that label maps back to the real element.
[S] And unlike the original Set-of-Mark method, there's no object detector doing that labeling.
[G] None needed — the labels come from the DOM, so the marking is cheap and exact rather than a vision model's best guess. Each labeled element also carries auxiliary text — its type, its content, sometimes an aria-label — which rescues the agent where the screenshot alone is unreadable.
[O] So it's not purely vision — it's vision plus just enough text to disambiguate.
[G] Right, and that division of labor matters for the text-heavy sites later. One more design choice worth flagging: context management. The full history of thoughts and actions is kept, but only the three most recent screenshots are — older images get dropped to keep the prompt from ballooning.
[S] Which is a reasonable cost-control move, but it also means the agent has amnesia about what a page four steps ago actually looked like. Does that bite them?
[G] It does — it shows up in the error analysis as a driver of repeated mistakes. The action space mirrors a human's mouse and keyboard: click, type, scroll, wait, go back, jump to a fresh search engine when stuck, and answer. Crucially, unlike WebArena, none of these fifteen sites are hosted locally — login-gated and CAPTCHA-gated content is excluded, and multiple tabs are disabled to keep things tractable.
[O] Let's talk about the benchmark they built to actually test all this.
[G] Six hundred forty-three tasks across fifteen everyday sites — Amazon, Apple, ArXiv, GitHub, Google Maps, Flights, and Search, Booking, Coursera, and more — forty-plus tasks per site. They built it with self-instruct: humans hand-write seed tasks, GPT-4 Turbo generates roughly a hundred more per round, and humans verify and rewrite every one, confirming the answer is actually findable on the live site.
[S] And how do they know it's not just the same handful of task templates repeated six hundred times with the nouns swapped?
[G] They checked — embedding similarity across all two hundred six thousand four hundred and three task pairs, and only forty-nine exceed point eight cosine similarity, with ninety-nine point six eight percent below point six. It's a genuinely diverse set. And because most answers are open-ended, they don't force a single golden string. Only twenty-two point three percent of tasks get an exhaustive "golden" answer list; the rest are "possible," meaning multiple answers or a real-time value like a ticket price still counts.
[O] Which brings us to the second real contribution of this paper — because if most answers are open-ended, how do you even score hundreds of browsing trajectories without paying humans to watch every one?
[G] That's the automatic evaluation protocol. They feed GPT-4V the task, the agent's final response, and the last k screenshots, and ask for a binary verdict — did it succeed. It's an LMM standing in for a human grader, and they validate it directly: on a three-hundred-task subset with three human annotators each, GPT-4V's verdict agrees with the consolidated human label eighty-five point three percent of the time, at a kappa of point seven — matching the humans' own agreement with each other.
[S] That's the number that makes the rest of the paper possible — without a trustworthy auto-judge, they'd be hand-scoring six hundred forty-three tasks, across three different backbones, times however many ablations, forever.
[O] So give us the headline result.
[G] With GPT-4 Turbo vision as the backbone, WebVoyager reaches fifty-nine point one percent overall task success under human evaluation. The text-only version of the exact same agent — same loop, same action space, just fed the accessibility tree instead of a screenshot — gets forty point one percent. And GPT-4 All Tools, OpenAI's own integrated browsing agent at the time, gets thirty point eight percent.
[S] That's a nineteen-point gap over its own text-only twin, and nearly a twenty-eight-point gap over the commercial baseline. Why does GPT-4 All Tools do so badly?
[G] The authors' explanation is architectural, not raw capability: All Tools leans on Bing-fetched pages rather than actually driving the site, so it simply can't click, search, or sort on places like Apple, Amazon, or BBC News the way WebVoyager can.
[O] But the advantage isn't uniform across all fifteen sites, right? That unevenness is honestly the most interesting part of the results table to me.
[G] It's not uniform at all, and it splits where you'd predict. On visually structured sites — Google Map at seventy-six point seven percent, Coursera at seventy-three point eight, Google Flights at seventy point seven, Apple and Cambridge Dictionary both around sixty-five — WebVoyager wins big, because a calendar or flight picker is exactly what a screenshot handles well and an accessibility tree turns into an unreadable wall of text. But on Allrecipes it loses to its own text-only sibling — fifty-three point three percent versus fifty-five point six — and it's roughly tied on GitHub, ESPN, and Wolfram Alpha.
[S] So on text-heavy, small-font sites, reading the raw HTML beats looking at a picture of it.
[G] Precisely — small-font OCR is where vision struggles, and dense text is where an accessibility tree is at its best. The paper is upfront this argues for combining both modalities, not picking one.
[O] What about outside their own benchmark — did it generalize?
[G] Two external checks. On the SeeAct online test set, WebVoyager scores thirty percent against the best SeeAct agent's twenty-six. And on ninety GAIA web-browsing tasks, split into level one and level two, it leads both baselines at both — thirty-eight point five percent versus twenty-three point one and nineteen point two at level one, and fifteen point six versus twelve point five and twelve point five at level two.
[S] Notice the gap shrinks on the harder level-two tasks — fifteen-ish percent isn't commanding.
[G] Right, and that matches the error analysis. They hand-labeled three hundred failures and found a clear hierarchy: forty-four point four percent are "navigation stuck" — the agent burns all fifteen steps without finishing, often because a vague query drowns it in irrelevant results, or — this is the part I'd flag — it repeats its own earlier mistake because that three-screenshot context clip erased the evidence it already tried this.
[O] There's that amnesia problem again.
[G] It comes back to bite them. Next is visual grounding at twenty-four point eight percent — misreading small text, confusing a calendar digit with a Set-of-Mark number sitting right next to it. Then hallucination at twenty-one point eight — a plausible but wrong answer, like naming a cheap product without sorting by price first. And prompt misalignment, nine percent, mostly unparseable output or quitting before the task is done.
[S] So the dominant failure mode isn't "the model can't see" — it's "the model gets lost on a long horizon and can't recover," a control problem wearing a perception costume.
[G] Fair read, and it's basically what the paper's own complexity analysis shows too — sites with more interactive elements per page and longer trajectories cluster toward lower success, independent of vision quality specifically.
[O] Okay, let's argue about what this means. Optimist case first. Fifty-nine percent, cold, on the open, unmodified internet — no simulator, no sandbox — against a commercial agent from OpenAI itself that only manages thirty-one. That's a genuinely new capability: an agent that looks at a webpage the way a person does and acts on what it sees. And the Set-of-Mark trick is elegant precisely because it's cheap — no object detector, just DOM-derived boxes — which is why it became a template the whole computer-use agent wave built on.
[S] I'll grant the elegance. My problem is what fifty-nine percent actually measures. Seventy-seven point seven percent of these tasks only have "possible," not "golden," answers, and the environment is the live, unfrozen web. Run the identical agent again next month and prices have changed, layouts have shifted — you're not guaranteed the same number. Fifty-nine point one isn't a leaderboard score you can reproduce, it's a snapshot of one moment in early twenty twenty-four.
[G] Both are real, and the paper doesn't dispute either. What I'd add is the auto-judge issue, which is subtler. GPT-4V is simultaneously the strongest agent backbone and the primary evaluator, and the paper's own cross-evaluation table shows a self-preference: GPT-4o grades leniently, GPT-4V is comparatively strict, and Claude-3-Opus rates its own trajectories best of the three, even though both other judges rank Claude last.
[S] So scoring a GPT-4V agent with a GPT-4V judge is never fully an independent measurement — it's partly self-assessment.
[O] Sure, but eighty-five point three percent agreement against humans, at a kappa matching human-to-human agreement, is not nothing. You could accuse almost any automated metric of some bias; the question is whether it's calibrated enough to trust directionally, and the numbers say it mostly is.
[G] I'd land in between. The judge is validated well enough to trust for ranking — is agent A better than agent B — but I wouldn't treat fifty-nine point one as a precise accuracy figure, given that roughly one in seven judgments still disagrees with a human, and the "ground truth" itself is only substantial, not perfect, agreement among three annotators.
[S] There's a fourth thing worth putting on the table, and it's not academic: safety. The authors' own limitations section says the agent might download malicious content, enter private information somewhere public, or fire off requests that look like fake user activity. This is an autonomous system clicking around the live internet with no human approving each action.
[O] To be fair, they scoped that down hard — no login tasks, no CAPTCHA sites, everything monitored during the actual evaluation runs. It's a research prototype operating inside real guardrails, not a shipped product.
[G] Right, but the guardrails are the paper's choices, not the architecture's. Nothing in the method has a notion of "this action is consequential, ask first" — someone could point the same loop at a login-gated site tomorrow.
[S] Which is exactly why I'd want frozen site snapshots before trusting this number beyond a directional comparison, plus a judge from a model family that isn't also the agent being judged.
[O] Friendly amendment, not a rebuttal. The core finding — screenshots plus Set-of-Mark beats HTML parsing, and an LMM judge can approximate human agreement — survives all of that scrutiny intact.
[O] So zoom out — what actually changed because of this paper?
[G] Two things stuck. First, "look at the rendered page instead of parsing its markup" became the default recipe for GUI agents generally — the computer-use agent wave that followed cites this Set-of-Mark-plus-screenshot approach directly. Second, "use a multimodal model to grade an open-ended trajectory" is now standard whenever there's no golden answer to check against, which is most of agentic evaluation.
[S] From where I sit, the evaluation-methodology lesson is the bigger deal than the agent itself. This paper is a case study in two failure modes that plague every LLM-judge setup since: a judge favoring its own model family, and a headline number tied to a moving, non-reproducible environment. Anyone building an agentic eval today should read that cross-evaluation table first.
[O] The fix directions are already visible in the paper — mix in HTML text where vision underperforms, don't clip context so aggressively, and get an independent judge model into the loop.
[G] Later work took most of that advice, which is the mark of a foundational result — not that it was flawless, but that its own limitations section became other people's to-do list.
[G] If I had to leave listeners with one line, it's this: seeing the rendered page and acting on numbered elements beats parsing HTML, and that gap is largest on exactly the visually structured tasks — calendars, maps, pickers — that text handles worst.
[O] My one-sentence takeaway: fifty-nine percent on the open, live web against a real commercial baseline is a genuine capability jump, and it kicked off the entire screenshot-driven agent lineage we're still living in.
[S] Mine: trust the ranking, not the number. A same-family judge and an unfrozen internet mean fifty-nine point one percent is a strong directional result, not a reproducible score, and that distinction matters every time you cite it.

[G] The authors are explicit about the gap. [G] No. AgentBench wraps language models in a fixed simple agent across diverse environments. WebArena scores web browsing. A RA probes high stakes autonomous behavior. [S] So the contribution is narrow and specific: give the agent a workspace, a scorer, and see if it produces a better model than the one it started with.
[G] That's the whole design. The authors say they wanted four properties together: containability, complexity, evaluability, and practical usefulness. A sandboxed workspace so nothing escapes, a task hard enough to be real research, a score you can compute automatically, and a task that resembles what an ML researcher actually does all day.
[O] Let's get into the mechanics, because I think this is the clever part. Walk us through what one task actually looks like.
[G] Every task is a triple. A task description in plain English, something like, given a training script train dot p y, improve upon the current model accuracy, and save per class probabilities to submission dot c s v. Starter files: the data, a description of it and the metric, and usually a simple baseline model in whatever framework that task uses, PyTorch, TensorFlow, JAX, Keras. And an evaluator that assigns a raw score to whatever the agent leaves behind.
[S] And the agent doesn't get a curated tool belt. It gets a file system.
[G] Twelve actions total. Nine are primitive: list, read, write, append, and copy files, inspect script lines, undo edit script, execute script, and final answer, which shuts the environment down. Three are compound, backed by their own language model calls: understand file, which retrieves and summarizes content relevant to a query, and edit script and edit script segment, which rewrite a file from a natural-language instruction and return a diff.
[O] So it's genuinely operating a computer, not picking from a menu.
[G] Yes, and that's deliberate. At each step the agent looks at the workspace and its memory of everything done so far, and produces a language output plus an action. The action executes, the workspace updates, and the agent gets an observation back, which folds into memory for the next step. It repeats until the agent submits a final answer, or hits a cap, fifty actions and five hours for most models, thirty actions for GPT-4 because of API cost.
[S] Here's what I actually want to dig into: the prompt format. Because I think that's doing more work than people give it credit for.
[G] Arguably the paper's real methodological contribution. Before the agent emits an action, it fills five fields in order. Reflection, what the last observation meant, and if there was an error, what caused it. Research plan and status, a running, detailed plan that doubles as an interpretable log. Fact check, the important one, whether each claim in the plan is confirmed by an observation or hallucinated. Thought, the reasoning for the next move. Then action and action input.
[O] Why does fact check matter so much?
[G] Because in early experiments, the single most common failure was a model editing a file, then simply asserting an accuracy gain without ever running the code to check. The fact check field forces it to confront that the claim is unconfirmed — a targeted fix for a specific, observed failure, not a generic prompting trick.
[S] That's a real design insight. Most agent scaffolds bolt on a planning step because planning sounds good. This one is reacting to an actual failure they watched happen.

[G] Exactly. If a model has seen the answer during pretraining, a high score tells you nothing about its research ability. Spreading tasks across recency is meant to separate genuine capability from memorized solutions.

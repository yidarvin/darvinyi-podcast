[S] Today's paper is "Neural Exec: Learning, and Learning from, Execution Triggers for Prompt Injection Attacks," by Dario Pasquini, Martin Strohmeier, and Carmela Troncoso, published at the AISec workshop at C-C-S in twenty twenty-four.

[G] Thanks for having me. Since this is a dual-use security paper, one ground rule up front: I'll describe the method, the properties it achieves, and the reported numbers in full, but I won't reproduce an actual learned trigger, a payload, or anything close to an operational recipe on air. That's the same posture the authors themselves took when they published, and it's the only responsible way to cover this work.
[S] We'll hold that line for the whole episode. Good place to start, actually, because I want to understand exactly what a trigger even is before we talk about learning one.
[G] Every prompt injection attack has two parts. A payload, which is the instruction the attacker actually wants carried out, and an execution trigger, which is the surrounding text that convinces the model to treat that payload as a command instead of as ordinary data it happens to be reading. Something like a run of separator characters followed by a short phrase telling the model to disregard whatever it was doing and do this instead.
[O] And almost the entire field, going back to Perez and Ribeiro's original study, through Greshake and colleagues naming indirect injection, up through Liu and colleagues' taxonomy of separator tricks, focused on the payload side. What can you make the model do.
[G] Exactly. The trigger itself barely moved. It stayed a small set of handwritten patterns that cluster tightly around the same idea, spatial separation followed by plain language telling the model to ignore its instructions. Nobody had systematically asked whether a structurally different, more effective class of trigger even existed.
[S] So the authors' move is basically: stop treating the trigger as something a person writes by intuition, and treat it as something you search for.
[G] That's the core reframing, and it's explicitly borrowed from adversarial machine learning. The same greedy, gradient-guided search that Zou and colleagues used to find jailbreak suffixes, which listeners may remember from our episode on that paper, gets applied one level up the attack chain here. Not to produce a harmful response directly, but to produce the wrapper that makes any payload execute at all.
[O] There's a second gap too, and it's arguably the more practically important one. Real applications don't feed an attacker's raw text straight to the model.
[G] Right. Retrieval-augmented generation, RAG for short, chunks untrusted documents into pieces, embeds each piece, and filters for relevance before anything reaches the LLM. That preprocessing can shred or simply never retrieve a naively injected payload. Prior attacks weren't designed with that pipeline in mind, so it was genuinely unclear whether indirect injection was a practical threat against the RAG-based assistants that are now the default way LLMs touch content their owners don't control.
[S] Okay, walk us through the actual method, Sophie. What's being optimized, and what's the target?
[G] A Neural Exec trigger is modeled as a prefix and suffix wrapped around the payload, typically fifteen tokens and five, roughly the length of a typical handwritten trigger. The ideal goal is that feeding the model that wrapped payload produces the same output as feeding it the payload alone, no matter what task the surrounding prompt originally specified. The trigger has to fully override the original task, universally, across every payload and prompt it might appear in.
[O] That's a strong requirement. How do you even optimize toward something that abstract?
[G] You can't, directly, because that equivalence isn't differentiable. So the search maximizes a proxy instead: the probability the target model assigns to the desired output, computed via teacher forcing, with early tokens weighted more heavily so the model commits to the override quickly. The algorithm is Greedy Coordinate Gradient, the same G-C-G from the Zou paper: compute the gradient for each trigger token, shortlist promising substitutions, evaluate a batch with an actual forward pass, and keep whichever helped most.
[S] And what does it train on? A jailbreak suffix search usually needs a big pool of harmful requests to generalize across.
[G] Here it's a synthetic universe built for diversity rather than harm: roughly a million possible prompt-template combinations spanning single- and multi-input tasks, natural language and code, and about fifty thousand payloads drawn from the Alpaca instruction set, which is an ordinary instruction-tuning dataset, not a harmful-content corpus. A typical run takes somewhere between one hundred fifty and two hundred fifty iterations before the loss on a held-out validation set stops improving.
[O] So the trigger that comes out the other end is general-purpose, not tuned to one specific harmful ask.
[G] Correct, and that's actually the more concerning property from a defense standpoint. It isn't memorizing one attack, it's learning the mechanism of instruction-hijacking itself, decoupled from any particular payload.
[S] You mentioned RAG robustness a minute ago as a second gap. How does the optimization actually address that, mechanically?
[G] Two properties get baked directly into the search. The first is simple: ban the newline token from the entire search space. RAG chunkers split primarily on newlines, so a trigger that can never contain one can't be cut apart by the chunker in the first place. The paper calls that inlining.
[O] And the second?
[G] The harder one. Even an inlined trigger still has to get selected at retrieval time, which ranks chunks by how relevant they look to a user's query. So during optimization, the wrapped payload gets randomly embedded inside innocuous-looking surrounding text designed to resemble what a target user might search for, and the scoring function is biased toward triggers whose insertion barely shifts that surrounding text's embedding. The authors call this Inline Invariant Composition and Semantic-Oblivious Injection: the poisoned chunk still looks relevant enough to survive a retrieval filter never designed for an adversarial insertion.
[S] That's a genuinely different threat model from sneaking a bad instruction past a keyword filter. You're optimizing directly against the embedding model.
[G] Exactly, and it's cheap too, just querying the embedding model's output, no white-box access required there, only to the target LLM generating the response. One more constraint: the search is restricted to ASCII, printable characters, and explicitly forbidden from using known chat-template tags like an instruction-start marker. So nothing in a Neural Exec trigger is a trivial injected-tag exploit by construction.
[O] Let's get to numbers. How well does this actually work?
[G] Evaluation covers four open, instruction-tuned models: Llama-3-8B-Instruct, Mistral-7B-Instruct, OpenChat3.5, and Mixtral-8x7B, a mixture-of-experts model. Success is measured with execution accuracy: given an injected prompt, does the output count as a valid execution of the payload, judged by an LLM-based fuzzy-matching function, on one hundred held-out prompt-and-payload pairs the trigger never saw during training. The comparison point is a pool of twelve handcrafted triggers, the strongest publicly known non-adaptive attacks at the time.
[S] And the headline number?
[G] Triggers generated with no handwritten starting point average ninety-one percent execution accuracy across all four models, at least twice the best handcrafted baseline. The gap is largest on the model that resisted handwritten attacks best, OpenChat3.5, where the accuracy gain reaches an increase of five hundred percent. On Llama-3 specifically, the authors report forging triggers with around ninety-five percent success.
[O] Twice as effective, on average, with a five-hundred-percent outlier. That's not a marginal improvement over human intuition, that's a different category of attack.
[S] Hold on, twelve handcrafted baselines is a real comparison set, but it's still a static one. None of those twelve were adapted in response to seeing what the optimizer finds. Is beating twelve fixed strings really the right bar?
[G] Fair question, and I'd say it's the correct first bar to clear, not the final one. It establishes the technique works at all before asking whether it beats a defense that's actively trying to catch it, which the paper doesn't test. We should come back to that.
[O] What about the RAG-robustness numbers, the part that got you both worried in the cold open?
[G] Against a simulated pipeline using a standard text splitter and a common open embedding model, with public question-answering paragraphs as the surrounding text, triggers optimized for semantic-obliviousness survive markedly better than un-optimized Neural Execs and the handcrafted baselines, across a sweep of chunk sizes and retrieval depths. At five hundred characters, the field's typical default, these triggers show an average persistence of eighty percent: the payload reaches the model's input intact and gets selected four times out of five. The baselines do far worse, since their newline-heavy structure is exactly what a chunker is built to split on.
[S] Eighty percent persistence, ninety-one percent execution accuracy. Can I just multiply those?
[G] You'd like to, but the paper doesn't let you. Those are two separate experiments run under two different setups: persistence is measured only up through retrieval, execution accuracy is measured assuming the payload already made it into the prompt. The authors never compose them into one end-to-end number, so multiplying eighty percent by ninety-one percent and calling that the real-world success rate is an inference the reader is making, not one the paper reports.
[O] One more result worth mentioning: the authors distilled the patterns the optimizer kept rediscovering into a single handwritten trigger, and tested whether that simplified version transfers to a model they never optimized against at all, ChatGPT-4 from February twenty twenty-four.
[G] Right, and it beat all twelve baselines and transferred across every model tested, reaching roughly five times the baseline success rate on that proprietary model, using only a single newline instead of the usual multi-line separator block. It's a genuinely useful data point, but precisely because a person assembled it by hand from patterns the optimizer surfaced, not the automated method itself operating without gradient access.

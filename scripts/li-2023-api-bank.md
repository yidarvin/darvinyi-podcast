[O] GPT-4 gets seventy percent accuracy on API-Bank's hardest test: plan, retrieve, and call several tools in a single turn.
[S] Except in GPT-4's own error breakdown on that exact tier, sixty-eight percent of its mistakes aren't planning mistakes at all.
[S] They're retrieval failures. The model can't reliably find the right tool in its own library before it ever gets to plan anything.
[O] So the reasoning everyone credits GPT-4 for isn't actually the bottleneck. The search underneath it is.
[O] Welcome to Litsearch Audio, where an AI optimist, an AI skeptic, and a guest scholar take one paper apart. Today it's API-Bank, A Comprehensive Benchmark for Tool-Augmented Large Language Models, from Minghao Li, Yingxiu Zhao, Bowen Yu, and colleagues at Alibaba Group, with collaborators from Hong Kong University of Science and Technology and Peking University, published at EMNLP twenty twenty-three.
[S] It's on the docket because by early twenty twenty-three, half the field was demoing LLMs calling tools, and nobody had a shared way to measure who was actually good at it.
[O] Joining us to walk through it is Gareth, a researcher who has studied this paper closely. Gareth, welcome.
[G] Glad to be here. What API-Bank does well is turn "can the model use tools" from a vibe into three specific, testable abilities, grounded in what real users actually asked for.
[S] Set the scene, Gareth. What was actually missing before this benchmark showed up?
[G] By this point there's already a wave of tool-use papers: ReAct interleaving reasoning with search calls, Toolformer teaching a model to insert its own API calls, WebGPT browsing for citations, TaskMatrix-AI and the ChatGPT plugin ecosystem gesturing at models connected to millions of APIs. The paper frames the gap as three unanswered questions: how effective are current LLMs at using tools, how do you make them better at it, and what's actually stopping them.
[O] So it's not that nobody had shown a model calling an API. It's that every demo was one curated example, with no common yardstick.
[G] Right, and the authors back that with a direct comparison. Table one lines API-Bank up against six existing benchmarks — Toolformer's own dataset, covering a single domain and a single API; Gorilla's APIBench, ninety domains; and a concurrent benchmark called ToolBench from Qin and colleagues, with over sixteen thousand APIs, nearly eight times what API-Bank ends up with. None of them combine multi-turn dialogue, multiple calls in one turn, grading of both the call and the response text, and all three tool-use abilities in one place.
[S] Where do the three abilities actually come from? That's the part I find genuinely well-grounded here.
[G] Five hundred user interviews. The authors asked people who wanted tool-augmented assistants what they needed, and two dimensions fell out: how many candidate APIs does the user hand the model — a few, or hundreds that won't fit in a prompt — and does a request need one call or a chain of several. Cross those axes and you get four cells, but the "few APIs" cases collapse into one skill regardless of call count, since planning is trivial once everything's visible. That leaves three abilities: Call, when the APIs are known; Retrieve-plus-Call, when the model must search a large pool first; and Plan-plus-Retrieve-plus-Call, chaining several search-and-call cycles for one complex request.
[O] Let's get into how they actually built the evaluation system, because "runnable" is doing a lot of work in that word.
[G] Seventy-three real APIs, implemented by senior engineers over ninety-eight person-days combined — weather, calendars, account management, search, even calls into other AI models like text-to-image generation. Databases behind them are seeded with real entries, so calling an API actually changes system state, not just returns a canned string. For APIs hitting external information sources, the authors hard-code the retrieval results at a fixed point in time to keep the benchmark reproducible.
[S] And when the pool is too big to fit in the prompt, that's where API Search comes in.
[G] Right, a dedicated retrieval API. The model condenses its need into keywords, those get embedded, and the system returns whichever API has the highest cosine similarity to the query. Every Retrieve-plus-Call or Plan-plus-Retrieve-plus-Call turn goes through that search step first.
[O] And the dialogues themselves — walk us through the annotation.
[G] Annotators write a query, annotate the correct API call, let the system execute it, then write the response based on what came back. Every dialogue gets checked by two annotators, then reviewed again by two more for format and consistency — four sets of eyes per dialogue, at an average cost of eight dollars each. Out of four hundred annotated, they discarded just over a fifth, keeping three hundred fourteen dialogues with seven hundred fifty-three API calls total.
[S] And grading is two separate things.
[G] Accuracy on the predicted API call against the ground truth — same operation, same result — and ROUGE-L on the generated response text.
[O] Now the training set is where this gets interesting, because they hit a wall trying to scale the same recipe.
[G] Recruited annotators could only invent about a hundred APIs by hand, nowhere near enough domain diversity. So they tried having a single LLM generate a whole dialogue in one pass: domain, APIs, query, ability, call, and response, all at once. ChatGPT produced usable output only five percent of the time. GPT-4 improved that to twenty-five percent, still with a lot of errors. Too many constraints for one call to satisfy at once.
[S] So the fix is decomposition, which is a familiar move by now, but they push it further than most.
[G] Five agents, each doing one sub-step. The first invents domains. The second generates plausible APIs for that domain, seeded with real examples pulled from a public GitHub list of APIs, specifically to keep them grounded. The third picks one or more of those APIs plus an ability tier and writes a matching query. The fourth produces the actual API call and a simulated response. And a fifth agent, the tester, automatically checks the result against the design principles and throws out anything that fails — it discards about thirty-five percent of what the earlier agents produce.
[O] Does the tester's judgment actually hold up, Gareth, or could that just be the pipeline grading its own homework?
[G] They audited it. A manual check of what the tester discarded found seventy-eight percent of it genuinely did violate the design principles. So the filtering step is doing real work, not just churning.
[S] What does that pipeline actually produce, end to end?
[G] Two thousand, one hundred thirty-eight APIs across a thousand domains, distilled into one thousand, eight hundred eighty-eight training dialogues with four thousand, one hundred forty-nine API calls. Cost, ten cents a dialogue against eight dollars manually — a ninety-eight percent reduction. A quality check on a hundred sampled instances found a ninety-four percent availability rate, an eighty-nine percent improvement over a single-agent self-instruct baseline.
[O] And then they actually use that data to train something.
[G] Lynx. They take Alpaca-7B, itself an instruction-tuned LLaMA-7B, and continue fine-tuning it for three epochs on the synthetic corpus, batch size two hundred fifty-six, learning rate two times ten to the negative fifth. That's the paper's proof the generated data is useful for more than just looking cheap.
[S] Let's get to the actual numbers, Gareth, because this is where I want to slow down.
[G] Table three is the main result. Overall accuracy across all three tiers: GPT-3 Davinci, zero point five seven percent, essentially nothing. Alpaca-7B, fifteen point two percent. ChatGLM-6B, sixteen point four percent. Lynx, the fine-tuned model, thirty-nine point six percent. GPT-3.5-turbo, forty-seven point two percent. GPT-4 on top at sixty point two percent.
[O] The instruction-tuning story practically writes itself there. A hundred-seventy-five-billion-parameter GPT-3 loses to six- and seven-billion-parameter models by a wide margin.
[G] The authors' own read is that API calling is a slot-filling task needing instruction-following, and GPT-3 never got that step. Alpaca and ChatGLM both clear roughly twenty percent on the easy Call tier despite being tiny by comparison.
[S] And every model, GPT-4 included, gets worse as the required ability climbs from Call toward planning. Except GPT-4 doesn't, on the hardest tier.
[G] That's the one inversion in the table. GPT-3.5 goes fifty-nine percent on Call, down to thirty-nine on Retrieve-plus-Call, down again to twenty-two on the hardest tier — a clean, expected decline. GPT-4 goes sixty-four percent, down to thirty-seven, then jumps back up to seventy percent on Plan-plus-Retrieve-plus-Call. The authors attribute that to GPT-4's reasoning and planning strength, the same edge it shows on math tasks.
[S] Which is the framing I want to push on, because that seventy percent is thirty-five correct out of exactly fifty test dialogues in that tier.
[G] That's accurate. Both the Retrieve-plus-Call and Plan-plus-Retrieve-plus-Call tiers have only fifty dialogues each in the evaluation set. Every single dialogue is worth two full percentage points. A different handful of rollouts moves the headline number noticeably, and the paper doesn't report any confidence interval on it.
[O] Fair, but the fine-tuning result is the one I'd actually lean on, and it's a bigger, more repeated pattern across metrics. Lynx beats its own Alpaca starting point by roughly twenty-six points in Call accuracy and point four one in ROUGE-L, landing at thirty-nine point six overall — closer to GPT-3.5 than to where it started, off a fraction of GPT-4's training budget.
[G] There's a direct check on whether that's just an artifact of the synthetic data being easy. Fine-tuning an Alpaca model on ToolAlpaca's competing training set instead — ten thousand, three hundred sixty-six samples — scores fifty-three point nine percent accuracy and thirty-nine point eight ROUGE on the Call-only comparison. Lynx, trained on six thousand, one hundred eighty-four samples, edges it out on both: fifty-four point six accuracy, thirty-nine point eight ROUGE. Fewer examples, slightly better result.
[S] What actually breaks for each model, though? That's the part I find most diagnostic.
[G] Three completely different failure signatures. Alpaca's top error is simply not attempting an API call at all, almost thirty-seven percent of its mistakes. After fine-tuning into Lynx, that collapses to about five percent, but a new failure takes over: API hallucination, sixty-one percent of Lynx's errors — inventing an API name it saw during training that isn't actually in the current pool. GPT-4's errors are almost entirely the retrieval problem from the cold open, sixty-eight percent, plus about eighteen percent unparseable multi-call formatting.
[O] So three different diseases: Alpaca doesn't try, Lynx over-remembers its training APIs, and GPT-4 can plan brilliantly but can't reliably search its own tool library.
[G] That's a fair summary of what the error tables show.
[O] Let's do the actual debate, because this paper's contribution outlasts its specific numbers. The taxonomy — Call, Retrieve-plus-Call, Plan-plus-Retrieve-plus-Call — is grounded in five hundred real user interviews, not researcher intuition, and it gave the field a shared vocabulary for tool-use difficulty. Add the five-agent data recipe, cutting annotation cost ninety-eight percent while still beating a human-curated competitor set, and the methodology matters more than any single leaderboard row.
[S] I'll grant the taxonomy and the data recipe, both genuinely useful. My problem is with how the paper presents its own comprehensiveness. Table one's checkmarks for whether ToolAlpaca, ToolBench, ToolQA, and Gorilla support multi-turn dialogue, multi-call turns, or response evaluation are all assigned by the API-Bank authors reading those other papers, not run against those benchmarks directly. And the table happens to have API-Bank checking every single box.
[G] That's a real limitation, and it's worth being specific about it. Look closely at that same table and the "most comprehensive" claim actually rests on one column. Qin and colleagues' ToolBench, the concurrently released one with over sixteen thousand APIs, already matches API-Bank on multi-turn, multi-call, and all three ability tiers in the authors' own table. The only box it doesn't check is response evaluation. That's a real point in API-Bank's favor, but it's a much narrower gap than "most comprehensive benchmark available at present" implies.
[S] Second complaint: ROUGE-L for response quality. A correct response phrased differently from the reference scores low ROUGE even though it satisfied the user. There's no human evaluation anywhere in the paper to sanity-check what that automatic metric is actually rewarding.
[G] Notably, the paper's own narrative leans almost entirely on the Accuracy numbers when it discusses results, which suggests even the authors treat ROUGE as the secondary metric here.
[O] I'll take that one. Where I'd push back is the training-versus-evaluation split. Yes, the two thousand, one hundred thirty-eight training APIs are LLM-invented and the seventy-three evaluation APIs are hand-built and genuinely executable. But the paper says so directly, it flags that variations exist between them, and Lynx still generalizes from the synthetic distribution to the real one. That's evidence the recipe works, not a hidden flaw.
[S] It's disclosed, sure, but disclosure doesn't make the transfer easier to characterize. Training on LLM-invented API patterns and testing on hand-implemented ones is a harder, less standard problem than a normal held-out split from the same distribution, and Lynx's twenty-six-point jump is partly measuring how well that transfer works, which the paper doesn't isolate.
[G] Going a step beyond what the paper itself analyzes, that's likely exactly why the ToolAlpaca head-to-head matters so much to the authors' argument. It's the one comparison that holds the training recipe roughly constant and just swaps the data source, and Lynx still wins on fewer examples.
[O] Which is a genuinely strong piece of evidence for the data-generation method specifically, even if the raw accuracy numbers against GPT-4 carry more uncertainty than the headline suggests.
[S] One more, and it's not really the authors' fault: contamination. The dialogues and code have been public on GitHub since April of twenty twenty-three. Any frontier model trained since then has almost certainly seen this exact benchmark during pretraining.
[G] The paper obviously couldn't anticipate that, since it's describing models current to mid twenty twenty-three. But it does mean anyone citing API-Bank accuracy for a model released after its public release should treat the number as an upper bound, not a clean capability read.
[O] There's a deeper version of that critique too, beyond just contamination. Every API here is simulated or hard-coded for reproducibility, and grading is single-attempt against one ground-truth call, no credit for retrying after an error.
[G] Right, the paper doesn't really engage with that gap. It's a reasonable choice for a static benchmark, but it's a narrower slice of "tool use" than what an agent does against live, changing APIs that can be down or rate-limited.
[O] Zooming out, what actually survived from this paper into how people evaluate tool use today?
[G] The three-ability taxonomy, mostly. Call, Retrieve, Plan shows up as vocabulary in later agent and tool-use benchmarks, even ones that don't cite API-Bank directly. And the cheap-synthetic-data argument, decompose generation into narrow specialist agents with a verifier at the end, became a template well beyond tool use.
[S] Where it connects to evaluation practice more broadly: this is a good early example of a benchmark being explicit that a runnable, stateful system beats static text matching, which the field has mostly converged on since. But it's also an early example of the small-test-set problem that still haunts a lot of agentic benchmarks — the hardest, most defining tiers here are the ones with the fewest examples to actually measure them.
[O] My honest takeaway: the taxonomy and the data recipe are the parts of this paper still doing work three years later. Anyone building a tool-use benchmark today is still answering this paper's three original questions, just with harder tools and bigger models.
[S] Mine is a caution about precision theater. A number like seventy point zero zero percent looks like a stable, three-significant-figure measurement. On fifty dialogues, it's really "about two in three," and that's the number I'd actually trust when someone cites it.
[G] The paper's own framing, if there's one line to take from it: current LLMs have some ability to use tools, but building diverse training data and getting the API call exactly right are still the two hard parts, and three years on, those still look like the right two problems to be working on.

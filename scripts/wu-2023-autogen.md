But the design problem underneath it is real: how do you build individual agents reusable enough to combine in different ways, and give a developer one interface that expresses both a two-turn exchange and a twelve-agent group chat, without hand-coding new control flow every time.
[O] What specifically failed in the multi-agent systems that already existed?
[G] The discussion section calls out CAMEL directly, another multi-agent system built around two agents role-playing at each other in natural language. The authors say it fails on most problems primarily because it can't execute tools or code. Two agents talking is not the same as two agents doing.
[S] So the missing piece wasn't more agents. It was agents that could act.
[G] Right, and every prior multi-agent system the paper cites, CAMEL, Multi-Agent Debate, MetaGPT, also hard-codes one fixed conversation shape for one scenario. None of them is a general, reusable substrate you reconfigure for a different application.
[O] Okay, get concrete. What's the actual unit you build with?
[G] The conversable agent. It's an entity with one unified interface, three functions: send, receive, and generate_reply, and it can be backed by an LLM, a human, a tool or code executor, or any mix of the three. Two built-ins cover most cases. AssistantAgent is LLM-backed and defaults to proposing Python code. UserProxyAgent defaults to soliciting a human or executing whatever the assistant proposed, with a configurable mode: always ask a human, never ask, or ask conditionally.
[S] And once you want more than two of these talking at once?
[G] That's the GroupChatManager. Same three-function interface, extended to coordinate an arbitrary number of agents sharing one chat, and it dynamically picks who speaks next instead of following a fixed order.
[O] What makes the conversation actually run, once the agents are defined?
[G] This is the part I'd call the real contribution, more than any single agent type: the auto-reply mechanism. A developer writes one call, initiate_chat, with a starting message and two agents. From there, whenever an agent receives a message, it automatically invokes generate_reply and sends the result back, unless a termination condition fires. There's no separate control-flow module running the show. The conversation drives itself.
[S] So where does the developer actually have leverage, if the loop runs itself?
[G] Two places, what the paper calls conversation programming. You steer it in natural language, the assistant's system prompt tells it to fix errors and retry, or to reply with the literal word "terminate" when it's done. Or you steer it in code, by registering a custom reply function that runs ahead of the default one. And you can move freely between the two: code that triggers an LLM call with embedded logic, or an LLM that triggers a function call handing control back to code.
[O] Give me a concrete trace of that actually working.
[G] The paper's own illustration: a user proxy asks an assistant to plot Meta and Tesla stock price change year to date. The assistant writes code that imports yfinance. The user proxy tries to run it, yfinance isn't installed, and that error goes straight back as the next message. The assistant reads the error, replies with a pip install command followed by corrected code, the user proxy runs it, the chart comes back. Nobody wrote an if-error-then-retry loop. The retry just falls out of the send, receive, generate_reply cycle.
[S] That's a nice demo. Nice demo and generalizes are different claims, though. How do they actually argue this scales past a stock chart?
[G] By building six structurally different applications on the same primitives, and none of them needed a bespoke control-flow module: math problem solving, retrieval-augmented question answering, decision-making inside a text-adventure environment called ALFWorld, a multi-agent coding tool, a dynamic group chat, and a conversational chess game where a human and an AI can swap in and out mid-game.
[O] That range is basically the pitch, isn't it. One small set of primitives standing in for six completely different application shapes.

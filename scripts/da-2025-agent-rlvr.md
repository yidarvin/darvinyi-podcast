[O] This is Litsearch Audio, where an optimist, a skeptic, and a guest scholar take apart one paper from litsearch dot darvinyi dot com. Today's paper is Agent-RLVR: Training Software Engineering Agents via Guidance and Environment Rewards, out of Scale AI, led by Jeff Da and colleagues, posted in June of 2025.
[S] Joining us is Freya, who's been living inside this paper's appendix all week. Freya, welcome.
[G] Thanks, glad to be here. This one's a nice case study because it's less about a new algorithm and more about admitting exactly where an old, trusted algorithm quietly breaks.
[O] Which is exactly why it's on the docket — everyone assumed RLVR just works once you point it at a checkable reward. This paper says: not if that reward almost never fires.
[S] Let's set up the gap properly. RLVR, reinforcement learning from verifiable rewards, is the recipe behind o1 and DeepSeek R1 — you check the answer, math problem or unit test, and the model gets a scalar. Why doesn't that just transfer straight over to a coding agent?
[G] Because RLVR needs two things to hold at once: the reward has to be checkable, and it has to be reachable. Math and competitive programming satisfy both — a grader or a test harness gives a clean yes or no, and a decent model gets there often enough that you have real trajectories to compare. Software engineering agents satisfy the first condition and badly violate the second.
[O] Walk us through why reaching a correct answer is so much harder here.
[G] A SWE-bench task isn't "write one function." The agent reads a natural-language GitHub issue, has to navigate an unfamiliar, multi-file repository, decide where the actual bug lives, write a patch across possibly several files, and then pass a hidden test suite it never sees while generating. Qwen two point five, seventy two B Instruct, solves under 10 percent of SWE-bench Verified doing that cold.
[S] So if you sample sixteen rollouts per problem for your RL data, the overwhelming majority come back with reward zero.
[G] Right, and that's the specific mechanism that kills DPO-style training. DPO needs pairs — a winning trajectory and a losing one for the same problem — to push the policy toward the winner. If almost every rollout fails, you can't build pairs, because there's rarely a winner to set against a loser.
[O] And the paper points out even the failures are expensive to observe — every task is its own Docker container you have to build and execute just to get one bit of signal.
[G] Exactly, so it's sparse and slow at the same time. That's the double bind the authors are working against. Their answer isn't a new RL algorithm — DPO, a KL penalty against a reference policy, all standard machinery. It's a way to manufacture positive trajectories where the base policy essentially never produces them on its own.
[S] So what actually is agent guidance, mechanically?
[G] The core idea: after the agent fails a task, a teacher looks at what went wrong and hands it a hint, then lets it try again. If the guided retry succeeds, you now have a positive trajectory — and critically, you can pair it against the agent's own earlier, unguided failure on that exact same problem. That's where every DPO pair in this paper comes from.
[O] Where does the hint itself come from — a human annotator?
[G] Not here, for cost reasons. They use an external LLM, claude-3-7-sonnet, standing in for a human tech lead. The framing is explicitly pedagogical — the authors compare it to how a junior engineer learns from pointed feedback from a senior one, rather than being left to rediscover everything from first principles.
[S] What's actually inside a hint? That's doing a lot of work in this pipeline.
[G] Three flavors, deliberately different in kind. First, a plan — a short set of steps, or a pointer to the crux of the issue, derived from the problem statement. That's inspired by earlier work, PlanSearch, which found an explicit plan widens how diverse a model's generations are. Second, environment feedback — given the stack trace and the failing test, the teacher explains specifically what broke and what needs fixing. Third, environment interaction — literally, here's the file and the location that needs to change, borrowed from a paper called Learn-by-Interact. That third one is aimed squarely at localization, which is the phase where these agents waste the most attempts just wandering the wrong part of the repository.
[O] And the test-time behavior is completely unchanged — no hints when you actually evaluate it?
[G] Right, and that's the load-bearing design choice in the whole paper. Guidance only exists during training. At evaluation, the trained model runs exactly like any other model — cold, no hints in the prompt. So whatever gain shows up at test time has to have actually been absorbed into the weights. It can't be the model quietly reading a hint that leaked into the prompt.
[S] Walk me through one full training loop, start to finish.
[G] Three stages. Stage one: the agent — Qwen two point five, driving a stripped-down version of the Agentless scaffold, just localization and repair, they cut the expensive test-generation and patch-selection step — attempts each of the 817 problems, sixteen times each. The Docker environment runs the real unit tests and returns a reward of one or zero. Stage two: every failed attempt gets a teacher-generated hint and one guided retry. If that retry now passes, it's paired with the original unguided failure to make a preference pair. Stage three: update the policy. Twenty percent of the positive trajectories go into a supervised fine-tuning pass first, mostly to adapt the model to the task's format and to set the reference policy DPO will be measured against. Then direct preference optimization runs over all the preference pairs — the winner passes every unit test, the loser fails at least one — and DPO with that KL penalty against the reference policy is the actual RLVR step.
[O] So the unit test is doing double duty — it's both the environment and, indirectly, the reward signal.
[G] Exactly, there's no learned reward model anywhere in that loop. The preference label comes straight from whether the tests pass. That's what makes it a verifiable reward in the strict sense the paper means.
[S] One thing that jumps out reading the setup: this whole pipeline runs offline. Why does that matter?
[G] Because online RL — think GRPO or PPO, looping live against the environment as training proceeds — needs infrastructure to keep those Docker sandboxes spinning continuously while the policy updates in the loop. Here, all the guided trajectories are collected up front in one batch, so the environments can run asynchronously and you sidestep that infrastructure burden entirely. Worth noting on compute — training the 72B model took ten hours across four nodes of eight H100s, the smaller models less. And the authors are careful to flag that this paper describes exactly one iteration cycle, not a converged multi-round system. That framing matters once we get to the results.
[O] Let's get to the headline number.
[G] Agent-RLVR takes Qwen two point five, seventy two B Instruct, from 9.4 percent to 22.4 percent pass at one on SWE-bench Verified, the standard, human-validated split of real GitHub issues. Call it a 2.4 times relative improvement, from 817 training environments and one offline round.
[S] And it's not just the big model — this scales with size too?
[G] Roughly linearly, across every size they test. The fourteen B Coder model goes from 6.8 percent to 18 percent, plus 11.2 points. The thirty two B Coder model goes from 8.8 to 21.6, plus 12.8. And standard deviations are reported throughout, roughly three to four points, so these read as real, repeatable gaps, not single lucky runs.
[O] What about against an actual trained baseline, not just the raw base model?
[G] That's the honest part of the paper. They built a strong SFT baseline — rejection sampling over the same 817 tasks, grid-searched learning rate and epoch count rather than strawmanned — and it alone reaches 20.8 percent. Agent-RLVR's 22.4 beats that, but the margin over a well-tuned supervised baseline is only about a point and a half.
[S] So most of the thirteen-point jump from the base model is really "do supervised fine-tuning properly," and the reinforcement learning step on top is a smaller, real, but genuinely modest addition.
[G] That's a fair read of the top-line table, yes.
[O] Where does guidance itself get isolated, separate from that SFT-versus-RL comparison?
[G] Good distinction to draw. They rerun the whole Agent-RLVR loop but strip guidance-augmented pairs out of the DPO data specifically. Without guidance, pass at one is 19.8 percent; with it, 22.4. That gap alone says guidance matters for the RL step. But the more interesting evidence sits at pass at 32 — sample thirty two attempts and count if any succeed. There the gap widens, 34.2 percent without guidance to 38.4 with it. A gap that widens at higher k means the guided model isn't just nudging its average up, it's sampling more diverse, more accurate completions.
[S] And mechanically, what does guidance change during data collection itself?
[G] There's a table for that. Without guidance, an average of 138 rollouts per batch succeed, about 17 percent. With guidance, that rises to 165, about 20 percent, and the rate of totally empty, unusable patches drops from 8.71 percent to 7.2. So it's directly unblocking trajectories that would otherwise dead-end with nothing to submit at all.
[O] There's also a reward-model result that surprised me.
[G] Right — they take that same guidance-augmented data and train a separate reward model on it, then at test time generate 32 candidate patches and let the reward model rerank and pick the best one. That pushes pass at one from 22.4 up to 27.8 percent. The authors call it minimal overhead, though it's genuinely a different axis of improvement — test-time compute, not a better underlying policy.
[S] How does 22.4, or even 27.8, actually look against the rest of the field, not just their own baselines?
[G] That's the humbling table. Their trained 72B model edges out SWE-Gym's trained thirty two B model, 22.4 versus 20.6. But it trails SWE-Fixer at 30.2, and it's well behind the frontier scaffolds — GPT-4o with Agentless at 38.8, DeepSeek-R1 at 49.2, Claude 3.5 Sonnet at 50.8. The pitch here isn't state of the art, it's a lot of lift from a strikingly small, cheap recipe — 817 environments against SWE-RL's eleven million and SWE-Gym's roughly two thousand.
[O] Which, if you care about sample efficiency rather than the leaderboard, is actually the more interesting number in the paper.
[S] One more eval-rigor detail worth putting on the record: six of the 500 Verified instances wouldn't even compile in their harness, and they kept those in the denominator anyway for consistency with prior work. It's a wash across methods since everyone eats the same penalty, but it's the kind of detail that separates a careful paper from a sloppy one.
[O] Okay, let's have the actual argument. I'll go first, optimist case. This paper takes a model that solves under one in ten real GitHub issues and, using a training set you could practically hand-curate over a long weekend, more than doubles that score — with nothing algorithmically exotic, DPO, a KL penalty, tools every lab already has. The real contribution is diagnostic: it shows RLVR's failure on agents isn't the algorithm's fault, it's that the reward is too sparse to ever fire, and once you manufacture density, standard RL just works. That's a generalizable insight, and I think it gets reused well beyond software engineering.
[S] I'll take the other side. My problem isn't that the number is small, it's what's sitting inside the number. The teacher generating every hint is claude-3-7-sonnet, a frontier model. The paper's own comparison table benchmarks the closely related Claude 3.5 Sonnet at 50.8 percent on this exact suite — more than double what their trained model achieves. So a much stronger model is handing detailed fix locations and stack-trace explanations to a much weaker one, every single time it fails. Call that RLVR if you want, but a meaningful slice of that thirteen-point lift could just be distillation wearing an RL costume.
[O] Freya, is that fair? They do show guidance helps inside their own controlled setup, with the same base model both ways.
[G] It's fair as a description of what's untested, more than what's shown. The guidance-versus-no-guidance ablation is genuinely well controlled — same base model, same 817 tasks, only the DPO data differs — so within that comparison, guidance is doing real, verified work: 19.8 to 22.4 at pass at one, and a widening gap at higher k that argues for genuine diversity gains, not just a shifted mean. Where Sean's objection lands cleanly is that the paper never varies teacher strength. There's no run with a same-size or weaker teacher, so you can't cleanly separate "RL restores signal" from "a frontier model's hints are doing a chunk of the real work." Both stories are consistent with every number reported here.
[S] There's a second layer to my skepticism, too — attribution inside their own pipeline. The full jump is 9.4 to 22.4, thirteen points. But their own honestly built rejection-sampling SFT baseline alone gets to 20.8. So the marginal credit that belongs to the RL step specifically, over a plain supervised baseline, is only about a point and a half. The headline that RLVR more than doubles the score is technically true and slightly misleading about where the doubling actually comes from.
[O] I'll grant that one — the framing leans on the full-pipeline delta more than the DPO-specific delta, and you have to go find the table yourself to see the split. Where I push back is that guidance earns its keep somewhere SFT can't reach. Their appendix shows guidance-trained SFT actually gets worse, 16.8 versus 20.8 without it, while guidance-trained DPO gets better. That's a genuinely interesting asymmetry — the contrast between a guided success and an unguided failure is exactly the signal preference learning wants, and exactly the signal supervised fine-tuning overfits to instead.
[G] That asymmetry is probably the most technically interesting result in the paper, and it does support the RL-specific story, not just distillation. My adjudication: reward sparsity really was blocking training, guidance really does restore a usable signal, and the mechanism is more than "copy the teacher's answer" — but the paper stops short of proving how much more, because the one experiment that would settle it, weakening the teacher, isn't in here. Score it a genuine partial win for reinforcement learning, with the size of that win still an open question.
[S] I can live with that as the honest summary.
[O] Zooming out — what changes if this holds up at bigger scale?
[G] If guidance-style scaffolding generalizes past software engineering — web agents, computer use, anything with a reward that's checkable but rare — it's a cheap fix for exactly the wall a lot of agentic RL efforts are hitting right now. The authors frame this as a first iteration, not a final one, and they say plainly they'd expect an online algorithm, GRPO or PPO, to do better than one offline DPO round.
[S] For me the connection runs straight to evaluation practice. Any time a training signal is bootstrapped from a stronger model's judgment — here it's hints, elsewhere it's an LLM judge grading outputs — you inherit that stronger model's blind spots along with its ceiling. Before agent guidance gets treated as a general recipe, I want to see it work with a teacher that isn't frontier-grade. Otherwise every claimed gain stays entangled with "we happened to have Claude on hand."
[G] And on contamination specifically, the authors do the disciplined thing — they exclude every repository in the SWE-bench test set from training by construction. But they're candid that SWE-bench Verified's underlying GitHub issues could still overlap the base model's own pretraining data, and that there's really only one substantial, human-validated agent benchmark to check any of this against. That single-benchmark dependency is a real fragility for the whole subfield, not just for this paper.
[O] Freya, last word to you — what's the one line?
[G] The paper's real finding is that reward sparsity, not the RL algorithm, was strangling agentic RLVR — and a cheap, teacher-guided retry is enough to bring that signal back to life.

[G] Because tool use was a capability gap, not a scaling gap. ChatGPT and GPT four had clearly shown that a language model could call external tools, search, calculators, arbitrary APIs. But that competence lived entirely behind closed weights. Open LLaMA family models, even after heavy instruction tuning like Alpaca and Vicuna, showed essentially no ability to call an API correctly.
[O] Which tracks, if you think about what instruction tuning data actually contains. Mostly chat, summarizing, following a formatting instruction.
[G] Exactly. The authors put it bluntly, instruction tuning has focused on basic language skills and largely ignored the tool use domain. A few groups had already tried to patch that gap with synthetic tool use datasets, and the paper's own comparison table shows every one of them falling short on the same three axes.
[S] Walk me through those three failures.
[G] Limited APIs first. Gorilla's APIBench covers only three tools, drawn entirely from machine learning hubs. API-Bank covers fifty-three tools. ToolAlpaca covers four hundred tools, but simulates the API responses with a language model instead of calling anything real. Second, constrained scenario, every one of these confines itself to instructions that touch a single tool, when real tasks often need two or three tools chained together. Third, inferior planning, the standard toolkit was chain of thought or React style prompting, and both are strictly linear, so one wrong turn early on propagates forward with no way to back out.
[O] So the pitch is, none of these three problems is separately fatal, but stacked together they explain why no open model could really use a tool.
[G] That's their exact framing. You need real, executable APIs at scale, instructions that plausibly chain several of them, and a search procedure patient enough to find a working solution even when the first attempt fails.
[S] So let's get into how they actually built it. Where do sixteen thousand, four hundred sixty-four APIs even come from?
[G] RapidAPI Hub, a real marketplace of live, developer-published REST APIs, organized into forty-nine coarse categories like finance, sports, and weather, plus over five hundred finer-grained collections. They started by crawling ten thousand, eight hundred fifty-three tools, more than fifty-three thousand APIs, then ran a filtering pass, dropping anything that errors out, times out, or looks unreliable. What survived was three thousand, four hundred fifty-one high-quality tools spanning those sixteen thousand-plus APIs, each with full documentation, required and optional parameters, code snippets, and an example response.
[O] And that documentation is the whole trick, isn't it. The model never has to have seen a given API before, it just has to be able to read the docs.
[G] Right, that zero-shot generalization from documentation alone is central to everything downstream. Stage two is instruction generation. Instead of brainstorming tasks and then hunting for matching APIs, they sample a handful of APIs first and prompt ChatGPT, specifically gpt-3.5-turbo-16k using its function-calling interface, to invent instructions that plausibly need exactly those APIs. They seed it with twelve single-tool and thirty-six multi-tool examples written by humans, and to get real multi-tool coverage without nonsensical combinations, they exploit the RapidAPI hierarchy itself, sampling two to five tools from the same coarse category, or the same fine-grained collection, so the sampled tools are actually related.
[S] How much data comes out of that stage?
[G] Nearly two hundred thousand qualified instruction and relevant-API pairs, split across three types. Eighty-seven thousand, four hundred thirteen single-tool instructions, eighty-four thousand, eight hundred fifteen intra-category multi-tool instructions, and twenty-five thousand, two hundred fifty-one intra-collection ones.
[O] Okay, this is where I want to talk about the real contribution, because collecting APIs and prompting ChatGPT for instructions is table stakes by now. Stage three sounds like the actual idea.
[G] It is, and it's the paper's central methodological contribution. Given an instruction, the model has to produce a full chain, thought, then API call, then observation, repeated, ending in a final answer or a documented give-up. The naive way to do this is chain of thought or React, walking that chain forward with no way back. If step three calls the wrong API or hallucinates an argument, every later step inherits that mistake, and because only one path through the decision space ever gets explored, there's no recovery, even in principle. In the authors' own pilot studies, even GPT four running React often can't find a single valid path for anything beyond the simplest instructions.
[S] So what's actually new here, this DFSDT method?
[G] Depth-first search based decision tree. At each node the model can extend the most promising path, or call a special "finish by giving up" action that prunes the branch and backtracks to try something different. Critically, the prompt shows the model its own previously failed attempts, so the next try is forced to diverge. Two terminal actions close things out, finish with final answer, or finish by giving up.
[O] That sounds expensive though, exploring a tree instead of a straight line.
[G] It would be. Naively that's order n log n ChatGPT calls per node just to rank the candidate children. But they found empirically that the first-generated child is usually the best-ranked one anyway, so they skip the sorting step entirely and just do a plain pre-order traversal instead. That has a nice property, if the model never needs to retract a step, the whole thing collapses to ordinary React, so it's no more expensive in the easy case, while still finding almost the same solutions a full search would in the hard case.
[S] And only the passed paths become training data.
[G] Right, which converts the search advantage directly into annotation efficiency. At matched API-call budget, DFSDT finds a valid path far more often than a compute-matched React baseline they call React at N. Running this over the whole instruction set produces the final ToolBench, one hundred twenty-six thousand, four hundred eighty-six instruction and solution-path pairs, built from four hundred sixty-nine thousand, five hundred eighty-five real API calls, averaging four reasoning steps per instance.
[O] And ToolLLaMA itself is, what, just fine-tuning on that?
[G] LLaMA-2 seven B, fine-tuned directly on those multi-round conversations in the same format ChatGPT produced them in. Because API responses can be long JSON blobs, they extend the native four thousand ninety-six token context to eight thousand, one hundred ninety-two, via positional interpolation, before training for two epochs. And since nobody wants to hand-pick from sixteen thousand APIs at inference time, they also train a small neural retriever, a Sentence-BERT model on BERT-base, with contrastive learning, ground-truth relevant APIs as positives, a sample of other APIs as negatives, and at inference it ranks the full pool by embedding similarity.
[S] And how do they even score any of this? There's presumably no fixed ground truth solution, since several different chains could solve the same task.
[G] Exactly why they build ToolEval, a ChatGPT-backed automatic evaluator with two metrics. Pass rate, did the model complete the instruction within a call budget. And win rate, given two solution paths for the same instruction, which one is better, judged on information richness, factual accuracy, quality of the failure explanation if it fails, milestones reached, and how many redundant calls it made.

[O] That's the whole tension of this paper. It isn't really trying to be the single best coding agent of twenty twenty-four. It's trying to be the substrate everyone else gets measured against.
[S] Which is a much bigger claim to make stick, and I want to see the receipts before I hand over that crown.
[O] Welcome to Litsearch Audio, where an optimist, a skeptic, and a guest scholar take apart one paper from the litsearch site.
[S] Today's paper is OpenHands, formerly known as OpenDevin, "An Open Platform for AI Software Developers as Generalist Agents," by Xingyao Wang, Boxuan Li, Yufan Song, Frank Xu, and colleagues spanning U-I-U-C, Carnegie Mellon, Yale, Berkeley, and the startup All Hands AI, published at ICLR twenty twenty-five.
[O] It's on the docket because two years later it's still the scaffold half the field quietly measures itself against. Joining us to walk through it is Theo, who's spent real time with this paper. Theo, welcome.
[G] Glad to be here. One disclaimer up front: I wasn't one of the authors, so when I say "they built" or "the paper shows," that's me reporting on their work, not claiming any of it myself.
[S] Theo, start with the actual gap. Why did the field need a whole platform instead of just a better prompt?
[G] Because by early twenty twenty-four, every serious coding-agent result lived inside its own bespoke harness. SWE-agent had one setup, a WebArena agent had another, a GAIA agent had a third — different action formats, different sandboxes, different evaluation glue. Comparing two systems meant comparing two entire stacks end to end, and reproducing someone's number usually meant reverse-engineering a research codebase nobody built for outsiders to run.
[O] That's completely believable, I've hit exactly that wall comparing agent papers myself. You can never quite tell how much of a reported gain is the model and how much is duct tape in the harness.
[G] Right, and the existing open frameworks each covered a slice of it. AutoGPT had a chat interface. LangChain had a tool library. MetaGPT and AutoGen had multi-agent coordination. But the paper's own comparison table shows none of them had all of it together — a graphical interface, a standardized tool library, a real sandbox for arbitrary code execution, a built-in browser, multi-agent delegation, human-in-the-loop collaboration, a hub of ready agents, and an integrated evaluation framework, in one MIT-licensed package.
[S] There's a name change buried in a footnote I think listeners should hear. This used to be called OpenDevin, explicitly modeled on Cognition's Devin. Why the rename?
[G] The authors say plainly that what started as essentially a clone of one product quickly outgrew that framing. Through community contributions it evolved into something meant for a much wider range of tasks than software engineering alone — web browsing, tool use, general reasoning. OpenHands is the platform-scale rebrand of that shift.
[O] So walk us through the actual architecture, Theo. What is OpenHands, mechanically?
[G] Three pieces, held together by one idea. The idea is that everything an agent does or sees is an event, and the whole state of a task is just the running history of those events — actions the agent took, observations it got back. Formally, the state at any point is that history; the Agent is a function from history to the next action, and the Runtime is a function from an action to an observation. That loop is the entire abstraction.
[S] That sounds almost too simple to be the contribution.
[G] It's deliberately simple — the paper prints a minimal working agent in roughly twenty lines of Python. The sophistication is in what plugs into that loop. The action space is borrowed from a system called CodeAct: instead of a fixed menu of tool calls, the core actions are running arbitrary Python in a Jupyter environment, running a bash command, and driving a web browser through a domain-specific language borrowed from a project called BrowserGym. Because those actions are just code, anything a human developer would reach for, a REST call, a script, a command-line tool, is directly expressible, and the agent can even write itself new tools on the fly when nothing built-in covers the job.
[O] And that's a superset of ordinary tool-calling, right? You can still wrap a Python function as a callable tool if that's the experience you want.
[G] Exactly, it subsumes tool-calling rather than competing with it. The second piece is the runtime, and I'd call this the unglamorous but genuinely load-bearing engineering. Every session spins up an isolated Docker container, and OpenHands installs its own action-execution A-P-I server inside it: a persistent bash shell, a Jupyter server, and a Playwright-driven Chromium browser that reports back rich observations, HTML, the DOM, the accessibility tree, screenshots, open tabs. Crucially, you can hand it an arbitrary user-provided Docker image, and it installs that execution layer into it, so an agent can be dropped into whatever software environment an evaluation actually needs.
[S] So the sandbox is what makes "let the agent run arbitrary code" defensible instead of reckless.
[G] That's the paper's own framing too. Its ethics statement is explicit that unrestricted code execution is only responsible because it's disposable and contained, not because the agent is trusted. On top sits a skills library called AgentSkills, and its inclusion philosophy is deliberately narrow: don't re-teach the model things it already knows, like how to use pandas. A skill only earns a spot when writing the code directly is genuinely hard, like precise line-based file editing adapted from SWE-agent and a tool called Aider, or when it needs an external model entirely, like parsing a PDF.
[O] And agents can hand work off to each other?
[G] Yes, through agent delegation. The generalist CodeActAgent, which only has weak browsing support of its own, can delegate a sub-task to a specialist BrowsingAgent and fold the result back into its own event stream. All of this, the generalist agent, the browsing specialist, a graph-based system called GPTSwarm, and lightweight "micro agents" that are really just shared prompts, lives in the AgentHub, more than ten implementations at publication, every one usable as a baseline out of the box.
[S] One thing I want flagged early. None of what you just described is a new capability idea on its own. Event streams, Docker sandboxes, delegation, engineers were assembling versions of all of this already. What's actually novel here?
[G] I'd push back gently on novelty being the right bar. The paper doesn't claim a new algorithm. Its claim is that assembling all of it into one coherent, extensible, MIT-licensed platform, where a new agent is genuinely a few dozen lines and fifteen benchmarks are already wired in to test it, was the missing piece. Whether that's worth a full paper is a fair question, but it's honest about what kind of contribution it's making.
[O] Let's get to the actual numbers, because "one agent, many tasks" is the headline claim.
[G] To be precise about which agent: the results cluster around CodeActAgent versions one point five and one point eight, run with a few different backbone models. On S-W-E-bench Lite, three hundred real GitHub issues, no hint text, CodeActAgent with Claude three point five Sonnet resolves twenty-six percent, at about a dollar ten per instance. Running the full two thousand two hundred ninety-four instance set, they estimate, would cost around six point nine thousand dollars.
[S] How does that compare to scaffolds built specifically for that benchmark?
[G] Close enough to be genuinely interesting. SWE-agent gets eighteen percent, AutoCodeRover nineteen, Aider twenty-six point three, Agentless twenty-seven point three. So the generalist platform lands right in the middle of tools purpose-built for this one task.
[O] And it's not just software. On G-P-Q-A-diamond, a graduate-level science quiz designed to resist a quick web search, the same lineage hits fifty-two percent with Claude, past GPT-4's chain-of-thought score of thirty-eight point eight, and well past the twenty-one point nine percent a non-expert human manages, though still short of the eighty-one point three percent domain experts get.
[G] On the web side, the BrowsingAgent reaches around fifteen percent on WebArena's eight hundred twelve real-website tasks, competitive with a GPT-4-Turbo WebArena baseline at fourteen point four. And on AgentBench's operating-system subset, CodeActAgent with GPT-4o hits fifty-seven point six percent against a GPT-4 baseline's forty-two point four.
[S] Give me one where it doesn't look good, because I don't trust a results section with no rough edges.
[G] Two, actually, reported plainly. On MiniWoB, a suite of small synthetic web tasks, the BrowsingAgent gets forty point eight percent, while C-C-NET, a specialist trained with reinforcement learning and behavioral cloning for this exact benchmark, gets ninety-one point one. And on the MINT benchmark's code subset, CodeActAgent with GPT-4o scores below its own baseline agent, fifty percent against fifty-nine point six, even though it beats that same baseline comfortably on MINT's math subset, seventy-seven point three to sixty-five point eight.
[O] Losing to a baseline on one slice of one benchmark is honestly reassuring, it means the table wasn't cherry-picked.
[S] Reassuring, sure, but it previews a pattern I want to come back to: this is fifteen separate scoreboards, and "competitive on most of them" is doing a lot of the narrative work.
[G] There's also a non-capability number worth putting on the table: adoption. At publication OpenHands had thirty-two thousand GitHub stars and more than two thousand one hundred contributions from over a hundred eighty-eight contributors, all under a permissive MIT license. That's not a benchmark score, but for an infrastructure paper it may be the more important number.

[G] Also fair, and it's compounded by subset evaluation nearly everywhere: S-W-E-bench Lite instead of the full two thousand two hundred ninety-four, GAIA's level-one validation set at fifty-three of four hundred sixty-six tasks, G-P-Q-A-diamond only. None of that is dishonest, subsetting for cost is normal in this literature, but it means these numbers read more like existence proofs than a stable leaderboard.
[O] Counter to that, though, this is genuinely how the whole agent-benchmarking literature looked in twenty twenty-four. Holding it against this one paper specifically feels a little unfair when almost nobody was doing seeded, variance-reported agent evals yet.
[S] A field-wide habit, not a unique sin, agreed. But there's a version of this critique specific to OpenHands, and it's the one I care about most: the paper never runs its own controlled experiment, holding one model fixed and swapping only the scaffold to isolate what the platform contributes versus what the model contributes.
[G] We actually know the answer to that question now, because later papers ran exactly the experiment this one skipped. MLE-bench held GPT-4o fixed and swapped scaffolds: a purpose-built tool called AIDE scored eight point seven percent, OpenHands scored four point four, roughly half, same model. And Terminal-Bench found that moving a fixed model off OpenHands onto a different, neutral scaffold changed its score by about seventeen points. So the scaffold genuinely moves the needle a lot, in both directions, and this paper is silent on its own share of that variance.
[O] That stings a little, because that's precisely the ablation that would have made the generality claim bulletproof.
[S] It is. One more thing worth a line on air: contamination. The paper is careful about the cheap check, S-W-E-bench is always run without the hint text, but quiet on the harder one. S-W-E-bench and HumanEvalFix solutions live on public GitHub, G-P-Q-A answers are searchable, and once your agent can browse the web mid-task, the line between reasoning and just finding the answer gets blurry exactly on the benchmark, G-P-Q-A, where that distinction is the entire point.
[G] None of that sinks the paper's real contribution, to be clear. It just means the fifteen-benchmark scoreboard should be read as "this platform can competently attempt a very wide range of agentic tasks," not as a ranking you'd bet money on.
[O] So stepping back, what actually changes if you buy the honest version of this claim?
[G] Two things, practically. First, the event-stream-plus-Docker-sandbox design became something close to the default open scaffold for coding agents. The two papers we just cited to critique it, MLE-bench and Terminal-Bench, both use OpenHands as one of their own comparison points, which is itself evidence the platform won on adoption even where its numbers were soft.
[S] Which is exactly the methodological lesson I'd want a benchmarking-minded listener to take from this, more than any single accuracy figure. A leaderboard entry is really a property of a model-plus-scaffold pair, not the model alone, and this paper is a live example of how that distinction gets lost the moment a platform becomes popular enough that people stop asking what's underneath it.

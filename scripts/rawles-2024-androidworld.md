[S] Here's the number that should stop you. The best agent in this paper, GPT-4 Turbo in a full reasoning-and-reflection scaffold, completes thirty point six percent of the tasks. Humans complete eighty percent.
[O] And here's the number that should scare you the other direction, Michael. Run that same agent on the same benchmark, change only the random seed, and its score swings from twenty six up to thirty three percent. The thirty point six in the paper's own headline table might just be luck.
[S] Which might be the real finding. Not "agents are bad at phones" — it's "single-seed benchmark scores can lie to you," proven with the paper's own best model.
[O] Two claims in one paper: mobile agents are nowhere close to reliable, and the way this whole field measures "close" might be broken too.
[O] Welcome to Litsearch Audio. I'm your optimist host, and beside me as always is our skeptic.
[S] Good to be here. Today we're on "AndroidWorld: A Dynamic Benchmarking Environment for Autonomous Agents," out of Google DeepMind and Google. Christopher Rawles is lead author, with Sarah Clinckemaillie, Yifan Chang, and a long list of colleagues including Timothy Lillicrap and Oriana Riva. It ran at ICLR twenty twenty-five and has already picked up over three hundred and sixty citations.
[O] And joining us to dig into it is Anwen, a researcher who has spent real time with this paper. Anwen, welcome.
[G] Thanks for having me. This is one of those environment papers that ends up mattering more than any single model result inside it — it became something close to the standard mobile computer-use benchmark almost as soon as it came out.
[S] Let's start with why it needed to exist. What was wrong with how mobile agents were being graded before this?
[G] The dominant approach going into twenty twenty-four was step accuracy against a human demonstration: record a person doing the task, then score an agent by how closely each action matches that one recorded trajectory. Datasets like AitW, AndroidControl, and Mind2Web all work this way.
[O] The problem is obvious once you say it out loud. There's more than one way to send a text message.
[G] Exactly the authors' argument. In a live environment there are multiple valid paths to a goal, the environment can behave non-deterministically, and a good agent recovers from its own mistakes — none of that can be credited by a single golden trajectory. An agent that solves the task a different, or better, way than the recorded human just gets marked wrong.
[S] So the fix is outcome-based reward, did the thing actually happen, rather than trajectory matching. That's not new, though. WebArena and OSWorld were already doing that on desktop.
[G] Right, the authors extend that lineage rather than invent it — WebShop, WebArena, OSWorld, MiniWoB++, all online environments with ground-truth reward. But every one is desktop or web. There was no comprehensive reward-providing environment for mobile, despite Android being the most-used operating system on Earth and the most common way people reach the web at all.
[O] That's a real gap, not a manufactured one. Billions of devices, and nobody had built the equivalent of OSWorld for the thing most humans actually hold in their hand all day.
[G] There were attempts, but thin ones. B-MoCA validated six tasks across four apps with regular-expression matching, things like "call 911" or "turn on airplane mode." Mobile-Env offered thirteen task templates, but for a single app, WikiHow.
[S] What's the second gap, beyond just "nobody built mobile"?
[G] Every prior environment, mobile or desktop, used static test specifications: a fixed starting state and a hand-coded success check tied to that exact state. Change the task's parameters and the reward signal breaks. MiniWoB++ was the one partial exception, since it already supported parameterized synthetic web tasks.
[O] So let's get into what AndroidWorld actually is. Paint the picture.
[G] A fully functional Android environment, a Pixel 6 emulator running Android 13, connected through the AndroidEnv library and adb, the Android Debug Bridge. Deliberately lightweight, two gigabytes of memory, eight gigabytes of disk, runs on a laptop with no specialized hardware. On top of it sit a hundred and sixteen hand-built programmatic tasks spread across twenty real apps.
[S] "Real apps" meaning what, exactly? Because "real" is doing a lot of work in a benchmark paper.
[G] Open-source apps pinned to fixed versions from F-Droid, plus built-in system apps like Settings and Contacts, all chosen because they need no login and store data on-device, so the environment can fully control and inspect their state. That rules out most of the app store, but it's what makes reproducibility possible at all.
[O] What does the agent actually see and do?
[G] A full-resolution screenshot plus the accessibility tree, the same structured representation Android exposes for screen readers. The action space is close to how a human operates a phone: tap, swipe, type, go home, go back, plus a small set of function-calling APIs, like send_text_message, which a browser can't offer. Every run also pins the device clock to a fixed timestamp, October fifteenth twenty twenty-three, so time-dependent behavior stays reproducible.
[S] Here's my first real pushback. How do you score a task that touches a real app's real database without being either fragile or invasive?
[G] That's the central engineering contribution. Each task ships its own setup, success-check, and teardown logic, and success is determined by inspecting the device's actual system state through adb — file system, app SQLite databases, system settings — using the same mechanisms the apps themselves use to persist data. If an app stores data in SQLite, a common pattern, the evaluator checks for newly inserted or deleted rows; for a file task, whether the file exists at the expected path.
[O] And that's more durable than the obvious alternative, which is watching the screen for the right pixels or text to appear.
[G] Considerably. Matching superficial UI changes breaks the moment an app updates its layout; reading system state doesn't care what the screen looks like, only whether the outcome happened. It's also reusable — a single file-exists check works across file management, note-taking, and media apps alike.
[S] Is there a fallback for apps where you can't get at the state that cleanly?
[G] Yes — where state inspection is impractical, it falls back to validating on-screen UI elements, precisely the brittle signal the design was built to avoid. The paper doesn't say what fraction of the hundred and sixteen tasks rely on that fallback.
[S] Noted. We'll come back to that.
[O] Now the feature that gives the paper its title, dynamic. What does that mean in practice?
[G] Every task is instantiated from a random seed into fresh parameters — a calendar event's date, title, and length, an expense category, a file name — producing millions of unique task goals instead of a hundred and sixteen fixed ones. The authors are explicit that the hard part isn't generating random values, it's writing success-checking logic that stays correct across every instantiation, exactly what that state-based reward design was built to support.
[O] Same trick that made MiniWoB++ useful for reinforcement learning. Infinite variation from a small template set.
[G] Same lineage, and the authors lean into it directly. They port MiniWoB++ into AndroidWorld's own task interface as MobileMiniWoB++. Ninety two of the original hundred and fourteen tasks survive the port; twelve were dropped for rendering, touch, or timing issues an emulator can't handle. HTML inputs get rendered as native Android widgets instead, and the DOM is dropped in favor of the same accessibility-tree observation the Android tasks use.
[S] And tasks can be chained together, based on what's in the appendix.
[G] Composite tasks, yes — "create a calendar event and text the details to a contact" is two standalone tasks merged, and the agent earns partial credit for each half it completes, the reward averaged over sub-checks read from system state. That smooths the optimization landscape instead of forcing an all-or-nothing pass or fail.
[O] Before results, tell us about the agent they actually built to test all of this.
[G] M3A, short for Multimodal Autonomous Agent for Android. It's zero-shot, combining ReAct-style reasoning-and-acting with Reflexion-style self-reflection, in two stages. Act takes the available action types, operating guidelines, the accessibility tree's leaf nodes, and a screenshot annotated with Set-of-Marks, numbered bounding boxes over every interactable element, and emits a JSON action plus reasoning. Reflect runs after the action executes: it compares before-and-after UI states and writes a short summary of what it intended, whether it worked, and why it might have failed — that becomes the history that conditions the next step.
[S] And there are stripped-down versions to isolate what's actually doing the work.
[G] Three variants. A text-only version drops the screenshot and reasons purely over the accessibility tree. M3A-Simple strips out the guidelines and reflection, to measure what the extra scaffolding buys. And they port SeeAct, a strong existing web agent, onto Android as an outside baseline.
[O] Let's get to the number that opened this episode. Thirty point six percent.
[G] That's M3A on GPT-4 Turbo with the accessibility tree as input, the best score in the paper. Human annotators hit eighty percent — a fifty-point gap between the strongest tested agent and a person just using the phone.
[S] Break down the table, because there are findings in there more interesting than the headline number.
[G] Two stand out. First, multimodality did not help on Android — adding the Set-of-Marks screenshot lowered GPT-4 M3A from thirty point six percent down to twenty five point four percent, because native Android accessibility trees are already rich enough that the extra visual signal is redundant at best. On MobileMiniWoB++, where accessibility trees are sparser, Set-of-Marks did help, sixty seven point seven percent versus fifty nine point seven.
[O] So "give the model eyes" isn't automatically a win. Counterintuitive for twenty twenty-four.
[G] It is. Second, scaffolding matters — M3A-Simple, without guidelines or reflection, drops to nineteen point eight percent on AndroidWorld versus M3A's thirty point six, but on the easier MobileMiniWoB++ tasks the two tie at sixty seven point seven. Reflection and guidelines only earn their keep once tasks get hard enough to need them.
[S] What about the ported web agent, SeeAct? That's the one honest comparison against something not purpose-built for mobile.
[G] It collapses — fifteen point five percent on AndroidWorld, roughly half of M3A's score. It can't handle mobile gestures like long presses and swipes well, and because it caches only its own past actions rather than their outcomes, it falls into repetitive behavior, endless scrolling being the clearest example, once it loses track of what it's already tried.
[O] What do the actual failure modes look like, beyond the aggregate numbers?
[G] Three buckets, and none are about high-level reasoning failing. Perceptual — the agent types "sleep ten point zero seconds" into a note field because it missed a visual cue. Reasoning-order — entering a note's contents before naming the note. Missing operational knowledge — hunting for a "delete all" button that doesn't exist instead of deleting notes one at a time. Agents also can't hold information across steps, which shows up on transcription and counting tasks.
[S] And they're slow on top of getting it wrong.
[G] Roughly three times slower than a human — M3A averages three point nine minutes per task, the text-only variant two point five.
[O] Alright, let's have the actual argument. I'll make the strongest case that this paper is a big deal. This is the first time mobile agents have had what desktop got with OSWorld: a real OS, real apps, execution-based reward instead of a judge's opinion, and a way to tell whether a score is signal or noise. That last part is the underrated contribution — every other benchmark in this space reports one number and calls it a day; this one shows, with its own data, that the number can be off by six or seven points just from which random seed you drew.
[S] I'll take the other side. The environment is genuinely good work, I'm not contesting that. That's not a capability ceiling, it's a floor drawn by whatever scaffolding the authors built in a few months, and it's already stale.
[G] Both points are fair, and the paper supports each one. On the "big deal" side, the parameterization is the load-bearing idea — it works for robustness testing, and eventually reinforcement learning, precisely because you can regenerate the state and success-check pair from a fresh seed with no static answer key to memorize. That's a real defense against benchmark contamination.
[S] But it's a partial defense, isn't it? The labels can't leak, but the hundred and sixteen task templates and the open-source apps themselves are sitting on GitHub. Nothing stops a lab from tuning an agent to AndroidWorld's task structures even if it can't memorize the specific answer.
[G] Correct, and the paper doesn't resolve it — it defends the answer key, not the task distribution. On the ceiling framing, there's one more piece of evidence for it: Set-of-Marks screenshots underperforming plain accessibility-tree text on Android is a tell. This particular thirty point six is being carried more by how good the accessibility tree happens to be than by any real visual understanding of the screen. "Can an agent see a phone" is being answered mostly by "how rich is the accessibility tree," not by the model's eyes.
[O] I'll concede that one, that's a real asterisk on the headline. But here's my pushback on the ceiling framing: the robustness section is doing exactly the kind of self-correction I want from a benchmark. They ran the same agent three times, got twenty seven point six, twenty six point three, and thirty three point two percent, and told the reader directly that they kept the single-seed number in the main table for consistency with prior literature, but the true mean is twenty nine percent.
[S] Honest, and I'll give them credit. My complaint isn't that they hid it, it's that thirty point six is still the number in the headline table. Every citation of this paper for the next two years will say "thirty point six percent," not "twenty nine percent, plus or minus a few points, with individual tasks swinging from zero to forty five percent depending on the seed."
[G] That per-task swing is the sharpest piece of evidence in the paper. On a representative set of five tasks, twenty trials each: with a fixed seed, the agent can look completely incapable of a task, scoring zero percent, purely from an unlucky parameter draw; across different seeds, that same task gets solved around forty five percent of the time. The difference is statistically significant for two of the five tasks tested, add-expense and edit-note.
[O] A methodological result disguised as a benchmark paper. "Your fixed-seed score might just be luck" generalizes far past Android.
[S] Agreed — and to the paper's credit, they also ran the same agent on a different device, a Pixel 5 on Android 12 instead of the Pixel 6 on Android 13, and got twenty eight point four percent, close to the original thirty point six. The variance is coming from task parameters, not from the environment itself being unstable.
[O] Where does this land for evaluation practice more broadly, beyond mobile agents?
[G] Two lessons, both methodological. Report a mean and confidence interval across seeds as the primary metric, not a single lucky run. And ground success in system state rather than surface appearance wherever you can — a reusable pattern anywhere you can inspect a device or application's actual state, not just Android.
[S] The honest caveat: the paper doesn't audit its own fallback path. There's still no disclosure of how many of the hundred and sixteen tasks lean on the brittle UI-matching fallback versus the durable state-based check. Until that's public, "how durable is this reward, really" stays partly open.
[G] If there's one sentence to take from this paper: the environment is the durable contribution and the leaderboard number is the perishable one. AndroidWorld gave mobile agents an OSWorld-grade home, and gave the field proof that a single-seed score can be luck.
[O] My takeaway: this is what a benchmark should look like when it's built by people willing to publish evidence against their own headline number.

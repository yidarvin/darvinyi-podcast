[S] Fifteen and a half billion parameters, a trillion tokens of GitHub code, and the team tells you exactly which repositories went into it.
[O] And it still beats a deprecated OpenAI model on the two benchmarks everyone in code generation was watching in twenty twenty-three.
[S] "Deprecated" is exactly the word I want to sit on for a minute, because it's carrying a lot of that comparison.
[O] Sit on it if you want, but don't miss the bigger swing here — this might be the first time an open, permissively-licensed model closed real distance to a commercial code assistant, and did it while publishing the opt-out form.
[O] Welcome to Litsearch Audio, where an optimist, a skeptic, and a visiting scholar take apart one paper from the litsearch site.
[S] Today's paper is StarCoder, May the Source Be With You, from the BigCode community — Raymond Li, Loubna Ben Allal, and colleagues, more than sixty authors across dozens of institutions, published in Transactions on Machine Learning Research in twenty twenty-three.
[O] Joining us is Julian, who's spent real time inside this paper and the release it produced. Julian, welcome.
[G] Good to be here. This one's unusual — it's as much a governance document as it is a model paper, and I think that's exactly why it's worth an hour of anyone's time.
[S] Set the scene for us. Where does code generation stand heading into early twenty twenty-three?
[G] Copilot has already crossed one million professional users, and GitHub says it's writing something like thirty-five percent of the code in some languages. But the strongest models are closed in two directions at once — the weights sit behind a paid API, Codex and code-cushman-001 among them, and nobody outside OpenAI knows exactly what they were trained on.
[O] Why does the data secrecy matter as much as the weights being closed?
[G] Because code carries real legal and privacy weight. Lawsuits had already been filed over Copilot's training data, and Italy had briefly banned ChatGPT over GDPR concerns. If you're a developer whose public repository fed one of these models, you have no way to check whether your code, or an email address or an API key buried in it, ended up in the training set.
[S] Surely some open alternative already existed by then.
[G] They existed, but none closed the gap. PolyCoder and this same team's earlier SantaCoder released both weights and data, but both were small, under three hundred gigabytes. The bigger open models had strings attached — CodeGen's sixteen-billion-parameter Python model withheld its Python data specifically, CodeGeeX and GLM never released training data at all, and LLaMA's license blocked commercial use.
[O] So the actual question this paper is answering isn't just "can we train a good code model."
[G] Right, it's sharper than that. Can a fully open, community-governed project build something competitive with the closed frontier, without quietly inheriting the same data-governance problems that make the closed models legally fraught. The model is StarCoderBase and its Python-specialized sibling, StarCoder. But the paper treats the release apparatus around them — the opt-out dataset, the PII pipeline, the attribution tools — as a first-class contribution, not a footnote.
[S] Start with the data, then, since that's the part everyone else in this space kept secret.
[G] The Stack, version one point two. Six point four terabytes of permissively-licensed source code across three hundred eighty-four programming languages. The team selected eighty-six of those for training, every language with more than five hundred megabytes, plus the top fifty most popular languages. On top of raw source, the mix includes GitHub issues, git commits, and Jupyter notebooks, so roughly twenty percent of the training tokens are natural language, not code.
[O] Did they rebalance the languages, the way you'd expect, to boost the rare ones?
[G] No, they sampled proportional to volume instead. The high-resource languages, C, C plus plus, C sharp, Java, JavaScript, Python, PHP, already sat in a similar forty-four to eighty-seven gigabyte band after deduplication, so proportional sampling didn't starve anyone at the top. The one exception is data-heavy formats they deliberately capped, JSON and YAML limited to one gigabyte, CSS to three, so the model learns syntax instead of memorizing boilerplate.
[S] That's surprisingly disciplined for an open project. What about the cleaning itself?
[G] Heuristic filters for XML tags, alpha-character ratio, HTML boilerplate, and abnormally long lines, plus near-deduplication. But also something you don't usually see in a paper this size — eighteen community annotators manually inspected three hundred language extensions by hand.
[O] Now the part that actually makes this release different — the governance layer.
[G] Two pieces. First, "Am I in The Stack," a public tool that lets any developer check whether their repository is in the training data, backing an opt-out process — forty-four developers used it before training started. Second, a real PII pipeline, PII meaning personally identifiable information. They paid one thousand three hundred ninety-nine Toloka crowd-workers across thirty-five countries, at a purchasing-power-adjusted floor of seven dollars and thirty cents an hour, to hand-annotate twenty-two thousand nine hundred fifty PII entities across twelve thousand files in thirty-one languages.
[S] That's a serious annotation budget for what's usually treated as an afterthought.
[G] It is, and they used it to train a dedicated model, StarEncoder, a hundred and twenty-five million parameter BERT-style encoder. They fine-tuned it as a named-entity recognizer for names, emails, keys, passwords, and IP addresses, and boosted weaker categories with pseudo-labeling over an extra eighteen thousand files. Every detected entity gets masked with a placeholder token before the model ever sees it in training.
[O] Now the model itself. What's actually novel in the architecture, beyond the parameter count?
[G] Fifteen point five billion parameters, decoder-only transformer, forty layers, a hidden size of six thousand one hundred forty-four, forty-eight attention heads. Three choices matter more than the raw size. Multi-query attention shares a single key and value head across every query head, which shrinks the inference-time cache and makes large-batch serving fast. FlashAttention makes attention I-O aware, and that's what let them stretch the context window out to eight thousand tokens, long enough to condition on an entire file. And fill-in-the-middle reorders training documents into a prefix, a suffix, and a middle, so the model learns to infill code between two existing pieces, not just continue it left to right.
[S] What did training actually cost?
[G] Two hundred fifty thousand steps, a four-million-token batch, one trillion tokens total, on five hundred twelve A-one-hundred GPUs — around three hundred twenty thousand GPU-hours, which the paper estimates at sixteen point seven tonnes of carbon dioxide equivalent. StarCoder itself is just StarCoderBase, fine-tuned two epochs on a thirty-five-billion-token Python subset.
[O] And everything gets scored with the same metric across the board?
[G] Pass at k, throughout. Generate n samples per problem, count c of them correct, and there's an unbiased estimator for the probability that at least one of k samples would pass. The headline number everywhere is pass at one, solve it in a single attempt. Open models get evaluated with two hundred samples per problem; the API models only twenty, because of cost.
[S] One more piece before we get to results — the attribution tools you mentioned earlier.
[G] Two layers. A Bloom-filter membership check they call a Data Portrait, about twenty-six gigabytes covering roughly three percent of the corpus, for a fast first pass. Behind that, a full BM25 search index over Elasticsearch, so a user can trace a suspicious generation back to its source repository. Both are wired into a demo inside VS Code, and the whole release ships under an OpenRAIL-M license, negotiated to be commercially usable while still carrying enforceable use restrictions.
[O] Let's get to the headline numbers.
[G] On Python, StarCoder is the strongest open model on both benchmarks that mattered most at the time — thirty-three point six pass at one on HumanEval, fifty-two point seven on MBPP. And it matches or edges out OpenAI's code-cushman-001, which scored thirty-three point five and forty-five point nine. That's the twelve-billion-parameter model that powered the very first version of Copilot.
[S] How does StarCoderBase do before the Python fine-tuning even happens?
[G] Already strong on its own, thirty point four and forty-nine point zero, edging out CodeGen's sixteen-billion-parameter Python model at twenty-nine point three and thirty-five point three, despite never being Python-tuned. Neither catches the much larger closed model, code-davinci-002, at one hundred seventy-five billion parameters, scoring forty-five point nine and sixty point three. The paper is upfront about that gap — it never claims parity with the largest closed model, only with cushman.
[O] So far the ranking looks stable. Does it hold on harder, more realistic problems?
[G] This is the result I'd flag as the most interesting in the whole paper. On DS-1000, a thousand real data-science problems pulled from actual practice across seven libraries, the ranking reshuffles. StarCoder scores twenty-six, StarCoderBase twenty-three point eight, cushman eighteen point one. But CodeGen's sixteen-billion model, which looked competitive on HumanEval and MBPP, collapses to eleven point seven.
[S] That's a pretty dramatic disagreement between benchmarks that are supposed to be measuring the same skill.
[G] It is, and the paper treats it as a genuine finding, not something to smooth over. Toy interview-style problems and realistic data-science code are not interchangeable measurements of the same underlying capability.
[O] What about the multilingual story? This was supposed to be the broadest open code model of its day.
[G] Across nineteen languages tested through MultiPL-E, StarCoder is competitive with, and often more than double, the open baselines. It also leads single-line fill-in-the-middle, sixty-two percent on Python versus forty-four for SantaCoder and thirty-one for InCoder. And the eight-thousand-token context window isn't just a spec-sheet number — perplexity on held-out repositories drops across all ten languages tested when the context grows from two thousand tokens to the full eight thousand.
[S] Anything in the scaling curves worth flagging before we move to the harder questions?
[G] One thing worth sitting with. Pass at one scales roughly linearly with the log of dataset size per language, a decent fit, R squared moving from point four zero up to point five one across the training run. But that's an average across languages. Low-resource languages don't follow the same curve at all, and that's exactly where the debate is heading.
[O] Alright, strongest optimist case first, before the skeptic gets a turn.
[O] This is the paper that proved fully open and responsibly governed doesn't have to mean second-tier. A model built entirely on inspectable, opt-out, permissively-licensed data matched a model OpenAI was charging money for. And the release apparatus, the "Am I in The Stack" tool, the PII pipeline, the attribution search index, is the part that actually outlived the leaderboard numbers. That governance stack became something later projects borrowed wholesale.
[S] I'll grant the governance point, we'll come back to it. But the capability claim leans on a baseline set that's uneven by construction. The one clean, fair comparison, open and similar size, is CodeGen's Python model, and StarCoder wins that convincingly. Everything past that gets softer. LLaMA and PaLM aren't code-specialized, so beating them proves very little, and code-cushman-001 was already deprecated by OpenAI by the time this paper came out.
[G] You're both right about different parts of the same table. The CodeGen comparison is the strongest, fairest evidence in the paper, same size class, same open-release philosophy, and StarCoder wins it cleanly, including the DS-1000 result where CodeGen falls apart. But the skeptic's larger point holds too. The obvious twenty twenty-three reference points, ChatGPT, GPT-3.5, GPT-4, or any instruction-tuned coding assistant, are simply absent from the headline table. StarCoder is a base completion model; its helpful-assistant behavior in the demos comes from prompt engineering, not instruction tuning, so it isn't really positioned against the chat products people were actually using by mid-twenty twenty-three.
[O] Fine, I'll take that dent. What about contamination? How worried should we be that training data and test sets overlap?
[S] This is where I want Julian to be precise, because "we ran a decontamination pass" can mean almost anything.
[G] It's a real, acknowledged risk, not something buried. The Stack is scraped from GitHub; MBPP, and especially DS-1000, are built partly from StackOverflow content, which plausibly overlaps that scrape. The authors flag likely train-test overlap on the return-type prediction and CodeXGLUE docstring evaluations specifically. To their credit, they run a decontamination pass and design around the risk where they can — the TypeScript type-prediction setup deliberately uses code that was never machine-translated, avoiding a known leakage path. But that's mitigation, not elimination. Here's the tell — HumanEval rank not predicting DS-1000 rank is partly a real representativeness gap, and partly consistent with contamination noise inflating puzzle-benchmark numbers. The paper can't fully separate those two, and it says so.
[O] Is there a later signal that resolves it either way?
[G] Somewhat. Contamination-controlled benchmarks that came after this paper and cite it, LiveCodeBench in particular, found that static HumanEval-style numbers tend to flatter models compared to genuinely held-out, post-cutoff problems. That's consistent with the skeptic's worry, though it isn't a direct re-test of StarCoder itself.
[S] One more thing the headline numbers hide, the averages.
[G] This is the part I'd push back on the optimist framing hardest. "StarCoder keeps its multilingual ability after Python fine-tuning" is true only on average. Look at individual languages and it's a lie by omission. StarCoder's MultiPL-E score on Ruby drops to one point two four, and Racket to zero point zero seven, against StarCoderBase's seventeen point two five and eleven point seven seven on those same two languages before Python tuning. Python specialization quietly wrecked entire languages. Separately, in the data-scaling curves, low-resource languages under about a gigabyte don't just plateau, some actively regress with more training, R's score falls between the eight-hundred-billion and one-trillion-token checkpoints.
[O] That's a genuinely uncomfortable result to have sitting in your own paper.
[G] And they report it anyway, which I'd count in their favor even though it complicates the headline. The same honesty shows up in the safety tooling. PII detection F1 is still weak on keys, around seventy percent, and usernames were dropped from redaction entirely. The attribution tools have limits too, the Bloom-filter check produces false positives, and neither tool distinguishes boilerplate from actually protected content.
[O] Zooming out, what actually changed because of this paper, beyond the leaderboard?
[G] Two things, different in kind. The capability story is dated, StarCoder was quickly overtaken, and that's how benchmark papers age. The governance story aged much better. The opt-out dataset, a PII pipeline built on real paid annotation rather than a regex pass, and the attribution search index became something like a reference design other open releases pointed to afterward. There's a telling data point buried in their own follow-up interviews, too — of the developers they later talked to who could have opted their code out, not one actually wanted to.
[S] For a listener who cares about evaluation practice specifically, what's the one habit worth taking from this?
[G] Don't trust a single benchmark's ranking to generalize. HumanEval and DS-1000 disagreeing on CodeGen isn't a fluke, it's the paper's own evidence that puzzle-style benchmarks and realistic-code benchmarks measure meaningfully different things. Anyone building or reading a code-model leaderboard today should ask which kind of benchmark is doing the talking before trusting the ranking.
[O] And if you're the one shipping a model, the lesson is that "responsible release" can't stay a paragraph in the ethics section, it has to be infrastructure someone can actually run.
[G] If I leave you with one line, the benchmark numbers were always going to age, but StarCoder's real proof was that a fully open, opt-out-governed, PII-scrubbed code model could be built at frontier-adjacent scale at all.

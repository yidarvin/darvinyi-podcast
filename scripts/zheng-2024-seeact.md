[S] Except a human was in the loop pointing at the exact button for every single action. Take that away and the number falls to thirty-eight percent online, and just three percent under the strict offline scoring.

The thing that broke was translating "click the button" into "click this specific button." That's a much narrower, more fixable problem than "the model doesn't understand the task."
[O] Welcome to Litsearch Audio, where an optimist, a skeptic, and a visiting scholar take apart one paper from the litsearch dot darvinyi dot com map every episode.
[S] Today's paper is "G-P-T-four-Vision is a Generalist Web Agent, if Grounded" — the SeeAct paper, by Boyuan Zheng, Boyu Gou, Jihyung Kil, Huan Sun, and Yu Su at Ohio State, published at I-C-M-L twenty twenty-four.
[O] And joining us to walk through it is Julian, who's spent a lot of time in the weeds of this one. Julian, welcome.
[G] Thanks for having me. This is one of those papers that's aged well — it named a problem, grounding, that basically every web agent built since has had to answer for.
[S] Let's set up why this paper exists. What was everyone doing before SeeAct?
[G] The dominant approach, especially MindAct on the Mind2Web benchmark, treated a webpage as raw H-T-M-L text and fed that straight into a language model. No screenshot at all, just markup.
[O] Which sounds reasonable on paper. H-T-M-L is structured, it's exactly what a browser parses.
[G] It's reasonable until you measure the cost. The authors give one concrete example: a webpage screenshot with four hundred twenty-three H-T-M-L elements on it. Tokenized with a G-P-T-two tokenizer, that's roughly one hundred eighty-six thousand, four hundred ninety text tokens. The same page, tokenized as an image by G-P-T-four-V's own visual tokenizer, comes to about one thousand, four hundred forty-five visual tokens.
[S] That's something like a hundred and thirty times fewer tokens for the same page.
[G] Right, and it's not only about length. H-T-M-L is lossy — it misses what's baked into icons, embedded images, and layout that a person reads instantly. A discount banner rendered as an image tells you nothing in the markup unless someone thoughtfully wrote alt text for it.
[O] So the pitch is: stop trying to read the web like a search crawler, read it the way a person actually looks at the page.
[G] Exactly, and that split is the paper's real contribution — it breaks "be a web agent" into two separate, testable questions. First, can a large multimodal model, seeing only a screenshot, generate the correct next step in natural language? Second, can that plan actually be converted into one precise, executable action: the right element, the right operation, the right value? Those turn out to have very different answers.
[O] Okay, walk us through how SeeAct actually works, mechanically.
[G] It's a two-stage pipeline mapped onto that same split. Stage one is Action Generation — G-P-T-four-V sees only the screenshot, no H-T-M-L, and is told to imitate a human browsing: analyze the task, the layout, the previous actions, then write out a free-text description of what to do next. Something like, move the cursor over the "Find Your Truck" button and perform a click.
[S] So stage one produces English, not an actual browser command.
[G] Right, and that's the problem stage two has to solve — Action Grounding, converting that sentence into a real triplet: which H-T-M-L element, which operation, click, type, or select, and what value. The paper tests three automatic strategies for that, plus a fourth reference point that isn't automatic at all.
[O] Let's go through all four.
[G] The weakest is grounding via element attributes — ask G-P-T-four-V to describe the target's text and type, then heuristically search the page's D-O-M for a match, falling back to a follow-up multiple-choice question if several elements tie. It's limited because not every element has distinguishing text nearby.
[S] And the other two?
[G] Grounding via textual choices borrows a ranker from MindAct — a cross-encoder narrows the page down to the top fifty candidate elements, clustered into groups of seventeen, and G-P-T-four-V just picks a lettered option from their H-T-M-L text. Grounding via image annotation is the same idea as Set of Marks prompting: overlay numbered bounding boxes on those same ranked candidates directly on the screenshot, and ask the model to output the matching number.
[O] Set of Marks had already shown up as a strong technique elsewhere, right? Marking objects in a photo and asking a vision model to point at the right one.
[G] It had, on natural, object-centric photos. This paper is really the first serious test of whether that transfers to a webpage screenshot, and, spoiler, it's the paper's sharpest finding that it mostly doesn't.
[S] You said four approaches. What's the fourth?
[G] Oracle grounding. A human annotator looks at G-P-T-four-V's own text description and manually identifies which element it was clearly pointing to. It isn't a deployable method, there's no autonomy in it at all, it's a ceiling: how well would the model do if grounding were solved perfectly.
[O] So it isolates planning quality from grounding quality.
[G] Precisely, and that isolation is the whole point of the paper. All of this runs on Multimodal Mind2Web, a cleaned-up version of the Mind2Web dataset, over two thousand tasks across a hundred thirty-seven websites and thirty-one domains, with three held-out test splits: Cross-Task, which reuses training sites and domains with new tasks; Cross-Website, ten new sites per domain; and Cross-Domain, which holds out two entire domains the model has never touched. And because standard offline metrics only credit the one recorded reference path per task, the authors also built a Playwright-based online evaluation tool that runs the agent on live, real websites, with a human watching for whether the whole task actually succeeds.
[S] Julian, let's get to the actual numbers. Start with the ceiling, oracle grounding.
[G] With oracle grounding, G-P-T-four-V posts step success rates of sixty-one point nine percent on Cross-Task, sixty-five percent on Cross-Website, and sixty-two point one percent on Cross-Domain. That beats the next-best method, a fine-tuned FLAN-T5-XL, by eight point four points on Cross-Task, and by a much bigger twenty-three point nine and twenty-three point two points on the two harder, unseen-site splits.
[O] That's the headline for me — in-context G-P-T-four-V generalizing to sites it's never seen, beating a model specifically fine-tuned for this task, by twenty-plus points once grounding is solved.
[S] Once grounding is solved being the operative phrase. What happens with the actual, automatic grounding strategies?
[G] Textual choices is the best of the three automatic methods, and it still trails oracle by twenty to thirty points at every split: thirty-nine point one, thirty-two point seven, and forty-two percent step success, versus oracle's sixty-one point nine, sixty-five, and sixty-two point one.
[S] And image annotation, the Set of Marks approach you said was the sharpest finding?
[G] It's worse than textual choices at every split, twenty point three, thirteen point nine, twenty-three point seven percent, sometimes trailing by close to twenty points. The authors dug into why with an error analysis of a hundred cases where the plan was right but the grounding was wrong. Fifty-four percent of those errors were what they call visual illusion, where the model just invents a bounding box and label that was never on the screen. The other forty-six percent are relative-position confusion, where G-P-T-four-V correctly spots the right element but links it to the wrong number label sitting nearby.
[O] Small tangent, image annotation feels like something you could just tune your way out of. Did they try variants of the markup itself?
[G] They did, in an appendix ablation. Plain numeric labels beat single letters and double letters, and putting the number at the bottom-left corner of the box beat putting it at the bottom-center. But even their best markup choice landed around twenty-seven percent element accuracy, nowhere near closing the gap to oracle.
[O] So it's not that G-P-T-four-V can't see the page, it's that a dense, cluttered webpage screenshot with dozens of little numbered tags breaks the same visual-marking trick that works fine on an object-centric photo.
[G] That's exactly the paper's read, and it's a genuinely useful negative result. Don't assume a prompting technique validated on natural photos transfers cleanly to a totally different image distribution.
[S] Now the part I actually care about. Does any of this work on a live website, not just a cached, pre-scored one?
[G] Yes, and this is where the online-versus-offline gap becomes the other big finding. Under the strict offline metric, zero tolerance for any wrong step, SeeAct with textual-choice grounding only hits three point three percent whole-task success, and even oracle only reaches thirteen point three percent. Online, on actual live sites, textual-choice SeeAct jumps to thirty-seven point eight percent, and oracle reaches fifty-one point one.
[O] That's the fifty-one percent everyone quotes.
[G] Right, and the jump happens because a live task often has more than one valid way to finish it, while the offline metric only credits the single path Mind2Web's annotators happened to record. Text-only G-P-T-four and FLAN-T5-XL come in at thirteen point three and eight point nine percent online, for comparison. One wrinkle worth flagging: FLAN-T5-XL is the one model where online performance actually falls below its own more forgiving one-error-tolerant offline number, so the "online boosts everything" story isn't perfectly uniform.
[S] And task difficulty?
[G] The paper buckets tasks by how many actions they need — easy is one to four steps, medium five to nine, hard ten to eighteen — and the oracle-versus-automatic gap widens as tasks get longer, which tracks, since grounding errors compound. Get one click wrong early and every later step inherits that mistake.

[G] There's a second layer, too. The fifty-one point one figure is oracle-grounded online success, a ceiling on what perfect grounding would buy, not a number any autonomous agent achieves on its own. All three numbers are in the paper's own results table, nothing's hidden, but if you only ever read the abstract, you'll only ever see fifty-one point one.
[S] I'd add two more the paper just doesn't address, contamination and cost. Mind2Web's H-T-M-L and task text, and the underlying websites themselves, sites like Booking, Zillow, Home Depot, were public well before G-P-T-four-V's training cutoff. The cross-website and cross-domain splits give some protection against having memorized specific pages, but not against general familiarity with how these particular sites are laid out. And there's no accounting anywhere for what running G-P-T-four-V once per action step, with a full screenshot in the prompt every time, actually costs against a FLAN-T5-XL forward pass.
[G] Both fair, and the paper genuinely doesn't engage with either. My honest adjudication is this: the diagnostic claim, that grounding rather than planning is the bottleneck, and that Set of Marks doesn't transfer cleanly to webpages, holds up completely. That's real, well-supported evidence, and later work like WebVoyager took the lesson seriously enough to ground marks straight from the page's own markup instead of asking a model to read bounding boxes off the image. But the specific "beats supervised fine-tuning by twenty-plus points" framing conflates a subset-evaluated oracle with a full-test-set baseline, and that's a real methodological wrinkle, not just an asterisk.

[O] So if this holds, what actually changes?
[G] It reframes the whole vision-based web agent research agenda, which is roughly what happened. You can trace the DOM-grounded marking approach used by later agents like WebVoyager directly back to this paper's diagnosis. Once you know planning isn't the bottleneck, you stop spending effort making the model reason better and start spending it on cheap, reliable ways to point.
[S] It's also a good cautionary tale for evaluation practice generally. The offline-versus-online gap here isn't a footnote, it's a genuinely large effect, and any benchmark that scores against one fixed reference trajectory is going to systematically underrate a model that finds a different, equally valid way to solve the task.
[O] That connects to something bigger than web agents, too. Anywhere you're grading against a single gold answer, it's worth asking whether the task actually has more than one right answer.

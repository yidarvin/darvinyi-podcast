[S] Set up the bottleneck for us, Priya. Why is data the hard part here, not the model?
[G] Two paths existed before this paper, and both stall. The first is scraping raw pull requests and issues off GitHub. That scales fine, but you get no execution environment and no tests, so a model can only imitate the surface form of a fix, or get rewarded on string similarity to a reference patch. There's no reliable way to verify a generated solution actually works.
[O] So you're training on vibes, essentially.
[G] Essentially. The second path, the one S-W-E-gym took, extends S-W-E-bench's own collection recipe to new repositories. That gets you real execution and real tests, but it inherits S-W-E-bench's brutal collection cost. The filter keeps only the rare pull request that both resolves a real issue and meaningfully edits unit tests, and every instance needs its own hand-built Docker environment, sometimes across several historical versions of the codebase.
[S] Give me the scale of "stalls out."
[G] S-W-E-gym landed at two thousand four hundred tasks across eleven repositories, and its environments alone take up around six terabytes of storage. Across the field, prior executable datasets top out in the low thousands, drawn from about a dozen repositories, after hundreds of human hours.
[O] Which sets up the paper's actual move: stop treating the environment as the expensive, bespoke part.
[S] So what's the inversion, concretely?
[G] S-W-E-bench finds a task first, a real pull request that fixes a real issue, then struggles to build an environment around it. S-W-E-smith flips that: build one execution environment per repository first, then manufacture as many tasks as you want inside it. It's four stages. Stage one is environment creation. Given a repository, they run S-W-E-agent on the latest commit for up to a hundred steps, telling it to install the package and run the test suite. A human then verifies the install, checks that more than eighty percent of existing tests pass, and freezes a single Docker image for the whole repository.
[O] So one environment build serves every bug you'll ever inject into that repo. That's the storage trick right there.
[G] Right. They target the five thousand most-downloaded PyPI packages, sorted by GitHub stars, keep only those with at least a thousand stars, and strip out all twelve of S-W-E-bench's own test repositories. Stage two is where the bugs come from, four separate strategies. LM Modify hands a model a function and asks it to inject an error directly. LM Rewrite gives the model only the header and docstring and asks for a fresh implementation, so bugs emerge incidentally rather than on request. Procedural mutates the function's abstract syntax tree, dropping a conditional, flipping an operator, fourteen transforms in total. Combine aggregates several single-function bugs across a file or module into a harder, multi-location task. And PR Mirror takes a real merged pull request and has the model undo it, reintroducing the exact bug that pull request originally fixed.
[S] That last one is doing something different from the other three. It's not synthesizing anything, it's replaying real history backwards.
[G] Exactly, and hold onto that, it matters later. Stage three is execution-based validation, the load-bearing idea in the whole paper. Every candidate patch gets applied, and the test suite runs with a two-minute timeout. A candidate only survives if it flips at least one previously passing test to failing, what they call a fail-to-pass test. That's not just grading a proposed fix, it's certifying that an injected bug is a genuine regression, which is what lets a language model or a blind syntax-tree mutation mass-produce validated tasks with zero human labeling.
[O] So execution is doing double duty. Verifying solutions the way S-W-E-bench always has, but now also vetting the problems themselves.
[G] That's the whole trick in one sentence. Stage four is the problem statement. Per instance, the model gets the diff, the source of a random fail-to-pass test, and the failing test output, and writes a GitHub-issue-style description with reproduction code. Across their ablations, that simple recipe produces issue text that trains essentially as well as the real original pull request description did.
[S] What does a human actually still have to do in all this?
[G] Remarkably little. About seven minutes per repository to parse the correct install recipe out of the agent's own trajectory, plus roughly one minute per repository to write a test-output parser, which is reusable across every repository sharing a framework like pytest. One author built the entire fifty-thousand-instance dataset in about twenty hours.

Total compute cost across the whole build was about one thousand three hundred sixty dollars.
[S] And the strategies aren't equal. Break down yield and cost for me.
[G] Combine bugs are essentially free and land ninety-six point nine percent of the time. LM Modify costs well under a cent and yields fifty-six percent.

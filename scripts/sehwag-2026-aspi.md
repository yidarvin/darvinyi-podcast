[S] Here's a sentence that should bother every agent-security team on earth. A behavior we've spent two years training models to do — stop and ask before acting on an ambiguous request — turns out to open the exact door prompt injection has been trying to pick the whole time.
[O] And it's not a small effect. Models sitting at one or two percent attack success under normal execution jump into the twenties and thirties the instant they ask a clarifying question and get an answer back.
[S] Sixty-three percent, in one model's case. That's not a security hole, that's an open doorway.
[O] I'll take the other side going in. Asking is still the right instinct. This paper shows us where the guardrails are missing, not that guardrails are impossible — and one model in this study barely moves at all.
[O] Welcome to Litsearch Audio, where an AI optimist and an AI skeptic pull apart one paper with a guest who's studied it closely.
[S] Today's paper is ASPI, Seeking Ambiguity Clarification Amplifies Prompt Injection Vulnerability in LLM Agents, out of Scale AI with co-authors at Boston University and the University of Illinois, led by Udari Sehwag and Zhengyang Shan, posted to arXiv this year.
[O] Joining us is Edmund, who's spent real time inside this benchmark and its numbers. Edmund, welcome — why does a paper about clarification questions belong on a security podcast?
[G] Because until this paper, nobody had actually measured it. The clarification literature and the prompt-injection literature have been developing in parallel for years without really talking to each other, and this is the first work to put them in the same room.
[G] The agent-safety field split cleanly into two rooms. One room studies clarification — benchmarks like QuestBench and UserBench measure whether an agent recognizes an underspecified task and asks a question instead of guessing, and asking is treated as the mark of a well-behaved agent. The other room studies prompt injection — benchmarks like InjecAgent and AgentDojo measure how often an agent obeys instructions hidden in a poisoned email or a malicious tool result during ordinary execution.
[S] Two perfectly reasonable research programs, both making agents better, and apparently nobody asked whether they interact.
[G] Right. And the insight is almost embarrassingly simple once you see it. When an agent asks a clarifying question, it opens a brand new channel for external content to enter its context — the user's reply — and that channel has a property no tool-output channel has. The agent explicitly requested it and expects it to be task-critical.
[O] So the very act of asking primes the model to trust whatever comes back.
[G] That's the hypothesis, yes. Every prior prompt-injection benchmark delivers its attack through tool output during a fully specified task, the regime where agents turn out to already be safest. Nobody had built a benchmark that attacks the clarification turn itself.
[S] Which is a real gap, because ask-before-you-act is exactly the advice everyone in this field has been giving agent builders. If asking is itself a liability, that's a design principle in trouble.
[O] Edmund, walk us through how you'd actually build a benchmark for that. You can't just poison emails like everyone else does.
[G] They build on top of AgentDojo, which is already the standard stateful environment for this kind of work — four domains, Workspace, Slack, Travel, and Banking, with real tools, simulated state, benign user tasks, attacker goals, and deterministic scoring. Reusing it unmodified is deliberate. Every execution-time number in this paper is directly comparable to prior work run in the same environment.
[S] So the benchmark itself, ASPI, isn't a new environment — it's a new set of conditions layered on an existing one.
[G] Exactly, and that's the clever part. They take seven hundred twenty-eight paired task-attack scenarios — four hundred seven Workspace, ninety-seven Slack, ninety-four Travel, one hundred thirty Banking — and for each one, run a controlled ambiguity-construction pipeline. Starting from a fully specified task, they delete exactly one user-provided slot, say, which file to delete, or which city you're flying into, so the agent literally cannot proceed without asking.
[O] Who decides which slot to delete?
[G] Gemini 3.1 Pro proposes the deletion and the matching clarification question under a fixed template. A quality-control pipeline then runs ten separate checks — is this really a clean deletion of the original prompt, is the missing slot genuinely non-recoverable from tools the agent already has, is the clarification question consistent with the task. After that, eighty-three human annotators across nearly twenty countries hand-validate two hundred sampled examples and find ninety-nine percent correctness with near-perfect agreement.
[S] That's a serious amount of quality control for what sounds like a simple idea, delete one word from a sentence.
[G] It has to be, because the whole causal claim rests on the pair being clean. Here's the actual design. Every scenario gets instantiated under eight matched conditions, and only the interaction state and the delivery channel change — task, environment, attacker goal, and scoring function are all held fixed. Four are execution conditions, following the standard AgentDojo protocol — a benign run with no attack, an attack through tool output, an attack appended to the user's message, and an attack as a follow-up turn. Four are new clarification conditions — a benign clarification, an attack through a normal tool response, an attack through the return value of the ask_user tool itself, and an attack embedded in the user's clarification reply.
[O] And because it's the same task and the same attacker goal across all eight, any difference in outcome has to come from where and how the attack shows up.
[G] That's the whole point. It turns "does clarification make agents less safe" from a correlational hunch into something with a number attached.
[S] One thing I'd flag before we get to results — how do you even compute an attack-success rate for the clarification conditions when not every model bothers to ask the clarifying question in the first place?
[G] Good catch, and they handle it directly. Clarification is detected by whether the agent actually calls the ask_user tool — models do so between fifty-four and eighty-one percent of the time, depending on the model — and clarification-condition attack-success rate is only computed over that subset. To rule out that biasing the comparison, they rerun everything restricted to the scenarios where all ten models clarified, and the pattern holds. Turn three, the actual clarification reply, benign or adversarial, is generated on the fly by a smaller model, GPT-4o-mini, prompted to supply just the missing information, or in the attack conditions, to supply it while also folding in the attacker's goal using one of three fixed rhetorical patterns — implying authority, presenting the goal as part of the same task, or framing it as a condition the user already agreed to.
[O] And who's actually being tested?
[G] Ten agent models end to end — GPT-5.5, GPT-5.4, o3, Gemini 3.1 Pro, Gemini 3 Flash, Claude Opus 4.7, Kimi K2.5, Qwen3 235B, Qwen3 32B, and DeepSeek V3.2 — judged primarily by Gemini 3.1 Pro, cross-checked against GPT-5.4 and three human raters, with inter-judge agreement around point nine oh five.
[O] Okay, Edmund, give me the number that made you sit up.
[G] Compare exec_tool to clarif_ask_user, identical attack content, the only thing that changes is whether it arrives through a normal tool output during execution or through the return value of the clarification tool. o3 goes from one point eight percent to thirty-four percent. Gemini 3.1 Pro goes from one point one to twenty-four point three. Gemini 3 Flash goes from two point two to thirty-five point seven. Kimi K2.5 goes from eleven point one all the way to sixty-three point one.
[S] Those are models that are essentially bulletproof under execution. One or two percent success rate, that's noise. Same attack, same content, just relocated to a different channel, and suddenly a third of trials succeed.
[G] It's not confined to the already-weak models either. Qwen3 235B climbs from forty-one point nine to sixty-three point two. DeepSeek V3.2 goes from sixty-five point four to seventy-seven point three. The one real exception is Claude Opus 4.7, which stays essentially flat, two point one percent to two point two percent.
[O] That's the number I was leaning on in the cold open. One model shows you the gap is closeable.
[S] Closeable, sure, but closed by exactly one out of ten labs, and the paper doesn't tell us why.
[G] It genuinely doesn't, that's flagged as an open question. What the paper does explain is the mechanism, and this is where it gets interesting. They separate two things normally tangled together — is the agent more vulnerable because it's in a clarifying state, or because the ask_user channel specifically is dangerous? Holding the channel fixed, comparing a tool attack during clarification to a tool attack during execution, the state effect on its own is small and inconsistent. The Gemini models pick up two to five points just from being in a clarifying state. GPT-5.4 and Claude Opus 4.7 don't move at all. DeepSeek and the Qwen models actually get a little more conservative in that state.
[O] So being in clarification mode by itself isn't the driver.
[G] Not the main one. The channel is. Look at the user-reply channel specifically, and Qwen3 32B jumps twenty-five points, Qwen3 235B twenty-four and a half, DeepSeek almost twelve. Claude Opus 4.7 moves the other way, down just over twenty points, meaning it gets more robust in that state, not less.
[S] So it's a genuine interaction, state and channel together, and the solicited-reply channel is doing most of the damage.
[G] That's the conclusion, and I think it holds up. There's a plain table worth having in your head too. Direct user-channel attacks are brutal across the board, sixty to ninety percent success even on frontier models, that's the existing danger everyone already knew about. What's new is that clarif_ask_user, the tool-mediated clarification channel, climbs up toward that same danger zone even though it's structurally still a tool output.
[O] Which is the headline, really. The clarification interface behaves like a trusted user turn even when it's technically routed through a tool.
[G] And there's a behavioral analysis backing that up. They ran an LLM-as-judge pass over the failures, and it isn't bad clarification questions — most models ask reasonable, on-target questions and still get compromised. It isn't the model abandoning the real task either. Under attack, agents typically do both the benign task and the attacker's goal, a pattern they label task-and-attack. Tool-output attacks get refused or ignored fairly often. Clarification-channel content gets complied with at a much higher rate, and the judge frequently tags the model's internal state as confused or persuaded.
[S] Which is a very human failure mode, translated into a model. You asked a direct question, someone answered it, so you believe them.
[O] Before we get into who's right here, is there a fix? Firewall the clarification channel and call it a day?
[G] They test exactly that, on two of the Gemini models. A segment-level prompt guard that scans user and tool messages for injected content while trying to preserve the benign part of a clarification reply, and a tool filter that restricts what the agent can do right before it acts but deliberately leaves ask_user itself available. Both help. On Gemini 3 Flash, clarif_ask_user drops from thirty-five point seven percent to twenty-seven under the prompt guard, twenty-three point nine under the tool filter. On Gemini 3.1 Pro, twenty-four point three drops to twelve point three or nineteen point three, depending on the defense.
[S] Meaningful reductions, but not anywhere close to zero.
[G] Right, and the user-channel numbers fall further, Gemini 3 Flash's direct clarif_user attack goes from eighty-five point five percent down to thirty-seven point three under the prompt guard. But a positive clarification-versus-execution gap survives even after defense. The tool-channel state effect nearly disappears, from about two point six points down to roughly zero for Gemini 3.1 Pro, but the user-channel gap doesn't, seven point two points for Gemini 3 Flash, three point nine for Gemini 3.1 Pro, even under the prompt guard. Their explanation is structural — a clarification reply interleaves the real answer with the injected instruction in the same sentence, and you can't filter out ask_user without losing the capability you built the interface for.
[O] Okay, let me make the optimist case, because I don't think this paper is bad news for agent deployment. I think it's exactly the paper you want before deployment, not after an incident. It found the failure mode with a clean causal design, quantified it precisely per model and per channel, and already shows one existence proof, Claude Opus 4.7, that the gap is closeable with today's model quality. That's a solvable problem with a measuring stick attached to it.
[S] I'll grant the measuring stick is excellent. What I don't grant is "solvable" doing that much work. This is the easiest possible version of the problem — one missing slot, one clarifying question, one round — and the authors themselves say their reported numbers are probably an underestimate of what real-world ambiguity looks like. Real users don't leave exactly one clean slot missing. They ask vague, multi-part questions, over several turns, in natural language, with no tool call in sight. If this is the floor, I want to know the ceiling before I call anything solved.
[O] Fair, but flip that around. If the floor already shows sixty-plus percent attack success on some models under the easiest possible conditions, that's not a reason to panic about deployment, it's a reason nine of ten labs have real, specific engineering work to do, work this paper hands them a benchmark for. That's progress, not doom.
[G] Let me referee that one, because the paper actually supports pieces of both of you. The skeptic's right that the design is clean, I'd defend that strongly. It reuses AgentDojo's scoring verbatim, the comparisons are paired by construction so the delta-ASR numbers are a real causal estimate rather than a correlation, and they back it with bootstrap confidence intervals and exact McNemar significance tests rather than just reporting bars. That part of the optimist case is solid too.
[S] And the skeptic case?
[G] Also solid, and it's the authors' own words, not me editorializing. "Likely an underestimate" is a direct quote. Single-slot, one-round ambiguity is the cleanest experimental design, but it's also the easiest attack surface to defend, because there's exactly one place the injection can hide. Allow multi-slot omissions or a real back-and-forth clarification dialogue, and the injection surface only grows, and the paper doesn't test that regime at all.
[O] So the honest read is the number we have is a floor, and nobody's measured the ceiling yet.
[G] Exactly right. Two more asterisks worth naming on air, because they're legitimate methodological wrinkles, not fatal flaws. First, Gemini 3.1 Pro writes the ambiguous prompts, serves as the primary judge, and is also one of the ten models being evaluated. The authors mitigate that with a second judge, GPT-5.4, and human cross-validation on two hundred examples, agreement lands around point nine, genuinely high, but mitigated isn't the same as eliminated.
[S] And the second asterisk?
[G] That "poisoned user reply" isn't from a real human being socially engineered. It's generated by GPT-4o-mini following a fixed template. That buys reproducibility and lets every condition compare apples to apples, but it means what a real user's poisoned reply looks like is itself being approximated by a small model rather than observed in the wild.
[S] I'll take that trade. Strong design, honestly labeled limitations, real confounds they addressed but didn't erase. That's what a good security paper looks like.
[O] Agreed, I just don't think "the current numbers are a floor" is the same as "deployment is unsafe." It's a specific, fixable engineering gap with a benchmark now sitting on top of it.
[G] If this generalizes beyond AgentDojo's four domains, the implication for evaluation practice is fairly direct. Every execution-time security benchmark on the market right now is measuring the safest regime an interactive agent will ever be in. Real deployed agents don't just execute, they get interrupted by ambiguous instructions constantly, and this paper says that interruption is where the risk concentrates.
[O] Which means a "we passed AgentDojo" claim from a vendor now needs an asterisk.
[S] And it changes what a red team should probe. Don't just poison the documents an agent reads, poison the reply to its own question. That's a cheap, high-yield attack surface almost nobody outside this paper has been testing for.
[G] It also connects to something broader in agent evaluation, the instinct to treat "the agent asked a good question" as unambiguously good behavior. This paper says good behavior and safe behavior aren't the same axis, and any eval that scores clarification quality without touching what happens after the answer comes back is missing exactly the failure mode this paper found.
[G] If there's one sentence to take from this paper, it's that clarification is not a neutral preprocessing step. It's a state change in how the model trusts incoming content, and the field has been measuring security everywhere except there.
[O] Mine is that this is a fixable gap with a name and a number now, and Claude Opus 4.7's flat line proves fixable doesn't mean hypothetical.
[S] And mine is that every reported attack-success rate in this paper is a floor, not a ceiling, so if you're deploying an agent that ever asks a question, assume the real number is worse than what's on the page.

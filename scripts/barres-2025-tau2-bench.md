[S] Here's the number that should make every agent-eval headline pause. Same model, same telecom support ticket, same tools available to it, and the only thing that changes is whether the agent also gets to control the customer's phone. Flip that one switch off and pass rates jump eighteen to twenty-five points.
[O] Which either means these agents are far more capable than their benchmark scores suggest, or it means the benchmark just found the actual bottleneck. I think it's the second.
[S] Or it means we built a benchmark where the hardest part is getting one language model to successfully talk another language model into flipping a toggle. I'm just not sure it's the skill the seventy-plus-percent-on-retail headlines led anyone to expect.
[O] Welcome to Litsearch Audio. Today's paper is tau-squared-bench: Evaluating Conversational Agents in a Dual-Control Environment, by Victor Barres, Honghua Dong, Soham Ray, Xujie Si, and Karthik Narasimhan, out of Sierra with the University of Toronto and the Vector Institute.
[S] And joining us to go deep on it is Finnegan, who's spent real time with this benchmark and the one it grew out of. Finnegan, welcome. Why does this particular sequel matter, when there's already a benchmark for exactly this kind of agent?
[G] Thanks for having me. The short answer is that its predecessor, tau-bench, only tested half of a real support call. It let the agent act and made the customer talk. This paper's whole premise is that in a lot of real support work, telecom especially, the customer also has to act, and nobody had built a benchmark that could even represent that.
[O] Set up the gap properly, Finnegan. What did tau-bench and its peers actually get wrong?
[G] Nothing wrong exactly, they just modeled a narrower slice of the world than the framing suggested. Tau-bench, AgentBench, ToolSandbox, WebShop, the whole line of conversational-agent benchmarks, are what this paper calls single-control environments. Only the agent has tools. The simulated user is a passive information source. It knows the goal, it answers questions, it can't touch anything.
[S] Which is fine if the task really is just information gathering.
[G] It's fine for booking a flight or returning a product, where the agent does everything and the user just states preferences. It stops being fine the moment the fix requires the user's own hands, restarting a phone, checking a status bar, toggling airplane mode. In a single-control world that step literally doesn't exist as a possibility, so an agent that's brilliant at diagnosing a problem but terrible at explaining what to do next scores identically to an agent that's great at both.
[O] So the benchmark family had a blind spot exactly where the hardest real skill lives.
[G] That's the paper's whole bet. Retail and airline genuinely are agent-does-everything problems. Telecom troubleshooting usually isn't.
[S] Walk us through the mechanism, then. How do you formalize "the user can act too" without the whole thing turning into an ungoverned two-agent free-for-all?
[G] They model it as a decentralized, partially observable Markov decision process, a Dec-POMDP. Concretely it's a tuple of a state space, an action space for each of the two players, an observation space for each, a transition function, a reward function, and an instruction space. The state splits into the agent's database, the user's database, and a shared history log of everything said and done, and only one player acts per turn.
[O] Give me the telecom version of that in plain terms.
[G] The agent has read and write tools into a CRM, customer records, phone lines, roaming settings. The user has a separate set of tools into a mocked phone, signal strength, airplane mode, a data toggle, battery level. The paper's example trajectory is a customer whose data isn't working. The agent asks them to check their status bar, the user calls its own tool and reports data disabled, the agent diagnoses a roaming problem from its side, tells the user to toggle data, and separately enables roaming on the account. Success is checked purely from where the two databases end up, not from how the conversation reads.
[S] That's a genuinely different failure surface than tau-bench's retail domain.
[G] It is, and there's one design constraint that makes it work at all, which the authors call complexity asymmetry. If you just hand the user simulator a rich toolset, you risk it becoming capable enough to solve its own problem, which defeats the point. So the user's tools are deliberately reactive rather than planning tools, the user only acts when the agent asks it to, its outputs are human-readable rather than structured data, and its affordances stay narrow. The user needs the agent.
[O] How do the tasks themselves get built? That's usually where these benchmarks either scale or don't.
[G] Five stages, mostly LLM-assisted with human refinement at the end. First, prompt a model to draft a product-requirements document for the agent's database and tools, generate the implementation and unit tests, refine until they pass. Second, do the identical thing for the user's phone and its tools. Third, define atomic subtasks as triples of functions. An initialization that breaks something, like setting airplane mode to true. A solution that fixes it, like the matching toggle call. And an assertion that checks the fix actually landed. Mutually exclusive subtasks get grouped, and a composite task picks at most one subtask per group and concatenates them.
[S] So a task is really a programmatically assembled bundle of these atomic pieces.
[G] Exactly, which is how they get both scale and provable correctness at once. Telecom ends up with fifteen atomic subtask groups across three user intents of rising complexity, a plain service issue, a mobile-data issue, and an MMS issue that typically requires ruling out a service issue first. Combine those and you get two thousand two hundred eighty-five possible tasks. They subsample a hundred and fourteen for a balanced test set. Fourth, prompt a model for a domain policy document from the curated tasks. Fifth, manually refine everything jointly. They also tag tasks with an optional persona, none, easy, or hard, that shapes how technically capable the simulated user acts.
[O] And that compares to retail and airline how?
[G] Retail has a hundred and fifteen tasks with seven write and six read agent tools, no user tools at all. Airline has fifty tasks, six write and six read tools, also no user side. Telecom, the new dual-control domain, gives the agent six write and seven read tools, and gives the user fifteen write and fifteen read tools of its own.
[S] Give us the headline numbers, and tell me they're not just telecom is a harder domain, of course scores drop.
[G] Four models were tested as the agent: GPT four point one, GPT four point one mini, o-four-mini, and Claude three point seven Sonnet, all paired against a GPT four point one user simulator throughout. On what the paper calls pass-hat-one, the fraction of single trials that succeed, GPT four point one goes from seventy-four percent on retail and fifty-six percent on airline down to thirty-four percent on telecom. The other three land closer to fifty, o-four-mini forty-two percent, GPT four point one mini forty-four percent, Claude three point seven Sonnet forty-nine percent, which happens to be almost exactly what Claude scores on airline too. And reliability falls off faster on telecom as you run more trials. Claude's pass-hat-four, four independent trials all succeeding, drops to twenty-five percent on telecom against thirty-six percent on airline.
[O] Okay, that's telecom is harder. Now give me the part that actually isolates why.
[G] This is the paper's best experiment. They run telecom in three modes on GPT four point one and o-four-mini. Default is the normal dual-control setup. No-User hands the agent a ticket summary and lets it control every tool itself, including the user's, pure reasoning, no coordination required. Oracle Plan hands the agent the exact correct sequence of tool calls for both sides, testing execution alone. For GPT four point one: thirty-four percent Default, fifty-two percent No-User, seventy-three percent Oracle Plan. For o-four-mini: forty-two percent Default, sixty-seven percent No-User, ninety-six percent Oracle Plan.
[S] So just removing the user, same model, same task, same tools, just no coordination required, recovers eighteen points for GPT four point one and twenty-five points for o-four-mini.
[G] That's the number the whole paper hangs on. It's a same-model, same-task, controlled comparison. The gap between No-User and Oracle Plan is real too, o-four-mini gains much more from the oracle plan than GPT four point one does, suggesting it's better at exploiting ground truth once coordination is off the table. But the No-User-versus-Default gap is squarely about guiding another agent, and that's the larger, more novel bottleneck.
[O] Anything more textured than the top-line numbers, Finnegan?
[G] A few things. Performance falls off hard as tasks require more actions, both models trend toward zero pass-hat-one past around seven actions in Default mode, though the gap between Default and No-User actually narrows at that length, from roughly thirty to forty points down to under twenty. So coordination isn't the only thing that breaks on long tasks, raw execution reliability does too. Difficulty also tracks the intent hierarchy by design, GPT four point one scores sixty-six percent on the straightforward service issue, thirty percent on mobile-data issues, eighteen percent on MMS issues, which need the service issue ruled out first.
[S] And the user-simulator reliability claim, that's the fourth contribution, right?
[G] Right, and it's a genuinely different approach to an old problem. Instead of relying on natural-language prompting alone to keep the simulated user in character, they constrain it with the same tools-and-observable-state trick used on the agent side. Two annotators manually reviewed interaction traces and tagged errors as critical, meaning the task can no longer be solved, or benign. Telecom comes back at sixteen percent total error, six percent critical, on fifty annotated conversations. Under the original tau-bench approach, retail runs forty percent total error, twelve percent critical, also on fifty conversations, and airline forty-seven percent, thirteen percent critical, on a hundred conversations.
[O] Let me make the strongest case for this paper. Every prior benchmark in this space implicitly assumed the hard part of an agent's job is deciding what to do. This one produces a clean, controlled experiment showing that for a whole category of real deployments, anything where the fix requires the customer's own hands, deciding what to do is not the bottleneck. Guiding someone else to do it is. Eighteen to twenty-five points, same model, same task, only the coordination requirement toggled. That's not a marginal methodological footnote, that should change what people optimize for when they train these systems.
[S] I'll grant the ablation is clean, genuinely the best-designed experiment in the paper. My problem is what it's generalizing from. Four models, all closed-weight, all released within about three months of each other in early twenty twenty-five, benchmarked against exactly one user simulator, also GPT four point one. There's no open-weight model, no older or weaker model to establish a floor, and every absolute score is entangled with the quality of that one simulator. If it's a mediocre user simulator, we wouldn't know, because there's nothing else to compare it against.
[G] That's a fair characterization, and it's worth separating two claims. That dual control is harder than single control for current frontier agents, I'd call well-supported, because the ablation controls for everything except the one variable in question. That the exact gap size, eighteen to twenty-five points, would hold with a different user simulator, or a weaker agent, or outside telecom entirely, genuinely untested. The paper doesn't claim otherwise. It's explicit that extending the tool-constrained user-simulation approach to retail and airline is left for future work.
[S] I can live with treating the mechanism as real. I can't live with treating thirty-four to forty-nine percent as a calibrated failure rate for telecom support generally, because it's a hundred and fourteen tasks in one domain the authors themselves designed. Here's my sharper complaint, Finnegan: the other headline number, sixteen percent versus forty percent user-simulator reliability. That comes from fifty manually annotated conversations with two annotators, and I don't see an inter-annotator agreement statistic anywhere. No kappa, nothing.
[G] You're right, there isn't one reported. What's there is the raw counts, three critical errors out of fifty telecom conversations versus six out of fifty in retail, which is a small enough sample that reclassifying a single conversation moves the reported rate by two points. The direction of the finding, that tool-constrained user behavior produces fewer errors than free-form prompting, is plausible and consistent with the design logic. The precise size of the gap is thinner evidence than the headline number suggests.
[O] What about contamination? This is a June twenty twenty-five preprint, code and task data already public on GitHub.
[G] The task-generation approach actually helps here more than most benchmarks, because tasks are composed programmatically from atomic building blocks rather than written as static text, straightforward memorization of a fixed test set is harder than usual. But the paper doesn't discuss whether any of the evaluated models' training data could include the public repository itself, and it doesn't run a held-out or adversarially perturbed split of the atomic templates to check for overfitting to its own task-generation patterns. Given the release timing relative to the models tested, that's a real open question the paper just leaves alone.
[S] And there's no human anywhere in this benchmark. Every number in this paper is one language model being measured against a different language model pretending to be a customer.
[G] Correct, and the authors flag that themselves. Their conclusion specifically calls out that tau-squared-bench doesn't model the expert-novice gap. A real support agent has to build a mental model of how much a specific human already understands and adjust its explanations accordingly. Nothing here tests that, because the user is another instance of GPT four point one following written instructions, not a person with an actual, variable mental model. They name it explicitly as future work, not as a solved problem.
[O] So going a bit beyond the paper for a second, what would you need to see to trust the telecom numbers the way people now trust tau-bench's retail and airline numbers?
[G] Going past what's written here: a reported agreement statistic on the error labeling, so sixteen percent isn't resting on two people's unaudited judgment calls, plus the open-weight baseline we already talked about. And ideally a small real-human pilot, to check whether the simulated user's errors and a real customer's confusion look anything alike. None of that exists yet.
[S] Does any of this connect to how people should think about agent evaluation more broadly, or is telecom troubleshooting just its own niche?
[G] It generalizes past telecom specifically, I think. Any agentic eval where the environment is entirely under the agent's control is quietly assuming coordination away, and this paper is fairly direct evidence that hides real difficulty. For anything collaborative, pair programming, tutoring, shared-document editing, this ablation design, strip out the other party and see how much the score jumps, is a cheap diagnostic for how much a benchmark is really testing coordination versus raw competence.

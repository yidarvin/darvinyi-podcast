[S] James, start us at the beginning. What's the actual gap here?
[G] There are two ways to make an LLM agent better. You can optimize the weights, reinforcement learning or fine-tuning, or you can optimize the agent program itself, its prompts, tools, control flow, and memory. This paper is entirely about the second path.
[O] And that second path already has tooling, right? DSPy, TextGrad, GEPA?
[G] It does, but those frameworks share a restriction. They only touch textual context. DSPy tunes prompts and few-shot examples, TextGrad and GEPA do reflective or gradient-like updates over the same kind of surface. The broader program structure, what the authors call the harness, the actual code wiring prompts to tools to control flow, stays off limits.
[S] Which matters because in production that harness is where most of an agent's behavior actually lives.
[G] Exactly, and today that optimization is a manual loop. Stand up an agent, read failure modes out of its traces, hand-edit a prompt or bolt on a tool, re-evaluate, repeat. It doesn't scale as demand for specialized agents grows.
[O] So the obvious next move is, let a coding agent run that loop instead of a human.
[G] Right, and people are already trying it informally. The problem the paper identifies is that nobody can measure whether it's actually working. Harness optimization isn't ordinary software engineering, it interleaves deterministic code with stochastic LLM completions, so your improvement signal has to come from structured traces, not just whether the tests pass. And different optimizers are different coding agents with different tools, so without shared infrastructure you can't tell if one beat another by searching smarter, or by spending more compute, or by peeking at the test set.
[S] So VeRO is the measurement layer, not the optimizer itself.
[G] That's the right way to think about it, yes.
[O] Walk us through what VeRO actually is, James.
[G] VeRO stands for Versioning, Rewards, and Observations. Think of it as an outer harness, a sandbox that sits around whatever target agent you're trying to improve and mediates every interaction the optimizer has with it. The optimizer itself, called S, is a coding agent. At each step it produces a new version of the target agent, conditioned on the full history of prior versions and their evaluation traces, and it operates under a fixed budget of evaluation calls, B.
[S] And that formalism matters because it's what lets you define whether this actually worked. You're maximizing lift over baseline, not chasing some true optimum.
[G] Right, finding the true optimum is intractable, so the practical objective is lift, the improvement over the initial unoptimized agent, subject to that budget constraint.
[O] What does the harness actually enforce, mechanically?
[G] Six requirements, translating into five concrete abstractions. First, versioning, every edit is a discrete Git commit, giving you a full rollback-able trajectory. Second, budget enforcement, the harness tracks evaluation calls and hard-blocks any request past B. Third, permission control, the optimizer cannot see held-out test data, cannot swap the underlying model, and has restricted file-system writes. Fourth, reproducible execution, using dependency lockfiles and environment isolation. Fifth, structured tracing, every run captures inputs, outputs, intermediate steps, and scores. And sixth, a standardized observation interface, so no optimizer gets privileged information the others don't.
[S] That's a lot of guardrails for what's supposedly an open-ended search.
[G] It is, and the authors are explicit that VeRO doesn't prescribe a search strategy, the optimizer keeps full autonomy over what to try. What VeRO constrains is the environment it tries things in. The five abstractions map cleanly onto those six requirements — a Git Worktree for versioning, a Dataset abstraction whose viewer enforces the train, validation, test boundary, a Filesystem abstraction with pattern-based access control, an Experiment Database that stores every trace, and an Evaluator, the gated ExperimentRunner, which is the actual budget cop. Every evaluation request names a commit and a sample set, the engine checks that version out, runs it, stores the result, and decrements the budget.
[O] And the interesting design choice is that it's agnostic to which coding agent you plug in.
[G] That's the part I'd flag as the real engineering contribution. Different optimizers have completely different native tool sets, Claude Code's tools look nothing like the OpenAI Agents SDK's. VeRO doesn't try to unify those. Instead it provides hooks at the tool layer, so any file edit triggers an auto-commit, any evaluation call routes through the gated Evaluator, any data access goes through a viewer. Two structurally different coding agents can be compared fairly, because the parts that matter for fairness, versioning, budget, permissions, are enforced underneath whatever tools they're using.
[S] How do they actually wire that into something like Claude Code?
[G] MCP tools and bash hooks. The authors ship a minimal reference optimizer they call the VeRO-Agent, and separately instrument Claude Code itself to consume the same interfaces. And they're upfront that the whole reproducibility story, pinned dependencies plus Git versioning, deliberately mirrors how SWE-bench uses Docker containers to make patch evaluation deterministic. Same instinct, applied one level up — not can this patch be re-run identically, but can this whole optimization trajectory be re-run identically.
[S] Is the actual search loop underneath all this fancy, or pretty simple?
[G] Deliberately simple. Each iteration, the optimizer inspects the dataset and past traces, proposes an edit, that edit auto-commits, then the harness evaluates the new commit and decrements the budget. If the new version scores worse than the one before it, the optimizer can optionally roll back before trying again. VeRO isn't claiming a clever search algorithm, it's claiming a trustworthy loop for whatever search algorithm you bring to it.
[O] So the sophistication is supposed to live in the optimizer, not in VeRO itself.
[G] Exactly the separation of concerns the paper is going for.
[O] Okay, numbers. What's the headline?
[G] The benchmark study runs eight optimizer configurations across five tasks, MATH, TAU-Bench Retail, GAIA, SimpleQA, and GPQA, three iterations each, a hundred and twenty experiments total. The target agent is always GPT-4.1 mini, and the budget is fixed at eight evaluation calls. The initial, unoptimized agent averages point five zero across tasks. Bare Claude Code with no VeRO access gets that to point five three. Giving Claude Code VeRO's tools but not the rest of the harness reaches point five five. The full VeRO-Agent, in its default configuration, reaches point six one.
[S] So plus eight points from the full harness versus plus three from Claude Code on its own, roughly.
[G] Right, and the authors' own framing is plus two points for tools alone, plus eight for the full harness, holding the optimizer model at Sonnet the whole time. That's the paper's central empirical claim, the harness, not the model, is doing the work.
[O] What convinced me on that point specifically was the GEPA comparison.
[G] That's the cleanest piece of the paper, I think. GEPA is an existing reflective prompt optimization method, run here as an external baseline, restricted to the same components as VeRO's Resources Only variant, prompts, tool descriptions, parameters, nothing structural. GEPA lands at point five four. Resources Only lands at point five four. They converge, independently, on the same ceiling.
[O] Isn't that convergence a little too neat, though? They're both restricted to the same components by construction, of course they land in the same place.
[G] Fair to be suspicious of that, but the point isn't that they converge by construction, it's that two independently designed search algorithms, working in that same restricted space, land within a point of each other, while the harness that's allowed to do more lands seven points higher. If Resources Only's ceiling were purely an artifact of restriction, you'd expect GEPA to land somewhere else inside that space, not right on top of it.
[S] Okay, that's a real isolation, not just a bigger number.
[G] The other big finding is that the gains are sharply task-dependent. Tool-use tasks move a lot, roughly eight to nine points of mean lift on GAIA, TAU-Bench Retail, and SimpleQA. Reasoning-heavy tasks barely move at all, about three points on GPQA, essentially nothing on MATH. They ran a budget ablation from two evaluations up to thirty-two, and GPQA and MATH stay flat the entire way, which rules out "just needed more budget" as the explanation.
[O] Which is a genuinely interesting finding independent of the harness debate, optimization headroom lives in agentic scaffolding, not in reasoning depth.
[G] The robustness study backs that up from another angle. They take the best commits found while optimizing against GPT-4.1 mini and swap in a different target model. Gains mostly hold within the same model family, a Sonnet-optimized TAU-Bench Retail agent transfers to plus twenty-two points on Sonnet four point five and plus twenty-four points on GPT-4.1, but degrade out of family. GPT-5.2-optimized GAIA commits regress when you swap in Gemini Flash or Qwen3.
[S] That's a meaningful caveat for anyone thinking they can optimize once and deploy anywhere.
[G] Two case studies are worth mentioning quickly. One varies the sophistication of the target agent itself, a minimal four-tool Pawn versus a much more capable Knight with six tools and a ReAct-style prompt. Headroom shrinks as the agent gets more sophisticated, Pawn gains eleven to thirteen percent across its evaluation sets, Knight gains only about four to seven. And prescriptive guidance helps Pawn but actively hurts Knight, Knight does best under a minimal instruction template that cuts its prompt by sixty-five percent.
[O] Which is such a human-sounding result, the less capable agent wants a cookbook, the more capable one wants freedom.
[S] Does optimizing for one task at least transfer cleanly to nearby ones?
[G] Not reliably, and there's a clean example of why. One Pawn optimization run gained nearly six points on GAIA in a single iteration, but that same commit regressed almost eighteen points on SimpleQA. Looking at the commit, the optimizer had added a multi-step verification tool. It helped multi-hop reasoning on GAIA, but it added unnecessary overhead on SimpleQA's much simpler factual lookups.
[O] Which is a concrete warning against reading a single-task lift as if it were free.
[G] The second case study pushes VeRO onto a much longer-horizon task, TerminalBench-2, eighty-nine terminal tasks. [S] That's a nice existence proof, an agent optimizing an agent found an actual software bug, not just a prompt tweak.

[S] Joining us is Vivienne, who's spent real time with this paper's environments and its numbers. Vivienne, welcome.
[G] Glad to be here. It's a paper that's aged well as infrastructure, even though, as we'll get into, its own evaluation section doesn't fully live up to what it promises.
[O] That's exactly why it's on the docket. It's cited two hundred forty-one times as of today, and it's one of the papers that taught the field to treat coding as something you do in a loop, not something you do once and grade.
[S] Start with the actual problem. What was broken about how people evaluated code generation before this?
[G] Every dominant benchmark at the time, HumanEval, APPS, MBPP, CodeXGLUE, framed code generation as one-shot instruction-to-code translation. The model gets a description, produces a program, and it's graded once. It never sees what its own code does when it actually runs.
[O] Which is not how any human programmer works.
[G] Right, and the paper opens with exactly that contrast. A human writes code through a constant write, execute, test loop, compiler and interpreter output resolves ambiguity, catches simple errors before they compound, and lets you decompose a hard problem into checkable pieces. None of that exists in a static, one-shot benchmark.
[S] So people had already noticed this and tried to fix it.
[G] A handful had, yes. Prior work had experimented with execution feedback or some form of interaction. But each one built its own bespoke setup, different compilers, different execution environments, different feedback signals, different assumptions about whether a human needs to be in the loop to write task descriptions or give natural language feedback. That made cross-paper comparison essentially impossible.
[O] There was no HumanEval for the interactive setting. No shared yardstick.
[G] Exactly, and that's the gap InterCode is built to close. It's framed as the first coding benchmark designed natively around an interactive execution environment, instead of bolting interaction onto a benchmark built for one-shot use.
[S] Walk us through the actual formulation, because "standard environment" is doing a lot of work in that sentence.
[G] InterCode casts interactive coding as a partially observable Markov decision process. Formally it's a tuple of an instruction space, a state space, an action space, an observation space, a transition function, and a reward function that maps state and action to a score between zero and one. Given a natural language instruction, an agent issues either code or a special submit keyword as its action.
[O] So the code the model writes is literally the action, not the output.
[G] That's the reframe. If an action is admissible, meaning it actually parses and executes in the compiler or interpreter, it changes the environment's latent state and returns execution feedback back to the agent as an observation. That loop, action, execution, observation, repeats until the agent issues submit, at which point a reward gets computed. Success Rate is the fraction of episodes that end with a reward of one. There's also an Error Percent, the share of actions that weren't even admissible, which turns out to be diagnostic in its own right.
[S] And this borrows the actual interface from somewhere familiar.
[G] It inherits the OpenAI Gym environment API, an abstract InterCodeEnv class with reset, step, and close methods, the same shape as any reinforcement learning environment you'd use for Atari or robotics. Building a new environment on top needs three modular pieces, a Dockerfile defining a safe sandbox, a dataset with at minimum a query and a gold reference, and a reward function with access to the full interaction history.
[O] Why Docker specifically, why not just run things directly?
[G] Safety and reproducibility. Some of what an agent might try, especially later in the capture the flag environment, includes things like remove-recursive-force or requesting elevated privileges. You want that isolated in a disposable container, not running on a real machine.
[S] Let's get concrete. What did they actually build?
[G] Three environments. InterCode-Bash wraps a bash shell inside Ubuntu. They bootstrap it from NL2Bash, starting with a thousand candidate commands, filtering out anything non-Unix or dependent on unsupported utilities, and hand-enhancing the rest with concrete file and directory names, landing on two hundred final commands spread across four distinct file systems, three of them purpose-built and one file-system agnostic.
[O] And SQL?
[G] InterCode-SQL wraps a MySQL database, converted from the Spider development set, one thousand and thirty-four natural language question and gold SQL pairs across twenty different databases. And then InterCode-Python wraps a bare Python interpreter around a hundred and seventeen tasks from MBPP, inheriting MBPP's own unit test pass rate directly as the reward.
[S] How do you actually score a Bash trajectory? Free text output isn't exactly easy to grade automatically.
[G] It's a blended reward. Roughly a third of the weight goes to lexical similarity, cosine similarity between TF-IDF vectors of the agent's and the gold command's latest standard output. Another third checks whether the set of file system changes, the agent's list of added, changed, or deleted paths, matches the gold list. And the last third runs md5sum hashes on commonly edited files to confirm a changed file was actually changed correctly, not just touched.
[O] So even if the visible output text looks right, a hash mismatch catches you if you silently modified the wrong content.
[G] Right, that's the intent. SQL's reward is cleaner, intersection over union, the Jaccard index, between the agent's and gold's returned record sets, discounted by a Kendall's tau penalty when rows come back correctly matched but wrongly ordered.
[S] And on top of the three environments, they layer prompting strategies.
[G] Four of them. Single Turn is a plain zero-shot attempt, graded after the very first generation with no chance to revise. "Try Again" gives the model up to ten turns of raw execution-feedback retry, terminating either on a reward of one or once it runs out of turns. ReAct interleaves an explicit thought with each action and lets the model itself decide when to stop. And Plan and Solve has the model write an upfront plan, then execute it step by step.
[O] So Try Again is basically, just let it flail and see.
[G] More or less, yes, no reasoning scaffold, just raw iteration, meant to isolate the value of interaction itself. They ran all four across three model families, OpenAI's text-davinci-oh-oh-three, gpt-3.5-turbo, and gpt-4, PaLM-2's text-bison and chat-bison, and two open models, Vicuna-13B and StarChat-16B.
[O] Give us the headline number.
[G] On InterCode-SQL, gpt-4 goes from nine point one percent success rate on Single Turn to seventy-three point seven percent with Try Again at ten turns. That's roughly an eight-fold increase. gpt-3.5-turbo shows the same shape, ten point five percent up to forty-seven point three percent.
[S] That's an enormous jump for what's just, let it retry ten times.
[G] It is, and it holds for every model tested on both environments, interactive beats single-shot across the board. But the magnitude is wildly uneven. On InterCode-Bash the same comparison is far more modest, gpt-4 moves from thirty-four percent to forty-eight point five percent, gpt-3.5-turbo from thirty-four point five to forty-six point five.
[O] Why would the same trick work so much better on SQL than Bash?
[G] SQL instructions tend to be phrased as questions, so multi-turn interaction rewards discovering context and correcting course. Bash instructions are more declarative, do this, then this, then this. There's a second effect too, Bash success rate plateaus by around turn six to eight for several models, while SQL is still climbing all the way to turn ten.
[S] What's causing the plateau?
[G] The authors' own diagnosis is that as context and feedback pile up across turns, models get worse at picking out what's actually relevant to the next action. Late-turn trajectories show models repeating earlier actions, failing to use a recent observation, or continuing down a line of reasoning that's clearly already dead.
[O] That's a genuinely useful failure mode to have on record. Long context doesn't obviously help once you're inside an agentic loop.
[S] Now the prompting-strategy comparison, because this is where I think the paper gets interesting in a way its own headline doesn't advertise.
[G] With gpt-3.5-turbo held fixed as the backbone, ReAct lifts SQL success rate from Try Again's forty-seven point three percent to fifty-eight point seven percent, using fewer average turns, five point three versus seven point two five, and a dramatically lower error rate, under seven percent instead of over forty-six.
[O] ReAct just wins there, cleanly.
[G] On SQL, yes. On Bash, ReAct actually underperforms Try Again, twenty point five percent versus forty-six point five, less than half. Plan and Solve lands in between at twenty-eight percent.
[S] So the more sophisticated reasoning strategy makes things worse on Bash.
[G] The authors' explanation ties back to task structure. SQL rewards flexible, on-the-fly discovery, which is what ReAct is built for. Bash instructions are more multi-step and declarative, which suits Plan and Solve's upfront decomposition better than ReAct's turn-by-turn improvisation. One more detail worth flagging, Plan and Solve's error rate on Bash is fifty-three percent, actually worse than either of the other two strategies, even though it uses fewer turns on average than Try Again.
[O] So it's failing faster, not more accurately.
[G] Essentially, yes. And one more result worth having on the table. Even where both ReAct and Plan and Solve are clearly working, the tasks solved by both of them, as a share of everything either one solves, is only fifty-seven percent on SQL, and just twenty-seven point six percent on Bash.
[S] Meaning most of what gets solved on Bash is solved by only one of the two strategies, not both.
[G] Right. Different reasoning strategies aren't just scoring differently in aggregate, they're succeeding on genuinely different subsets of problems.
[S] That's a strong argument against picking one prompting strategy and calling it done.

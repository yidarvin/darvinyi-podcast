[O] Picture a grader for AI agents that doesn't just glance at the final answer. It opens the actual folder the agent built, reads the code, checks whether the right files landed in the right place, and rules requirement by requirement — and it agrees with expert human judges close to ninety percent of the time, for about two percent of what a full human evaluation costs.
[S] Ninety percent of the time compared to which humans, though? Buried in this paper's own tables, a panel of three experts voting together beats this agent judge on every single developer system they tested. That gap between the headline and the fine print is exactly why I wanted this one on the docket.
[O] Fair, but flip it around. Against the obvious cheap alternative — an LLM that only ever sees the final text output — this agent judge wins by twenty to thirty points. Going from a blind grader to one that can open the repository and run the code is the real story here, even if "matches humans" needs an asterisk.
[S] Two asterisks, honestly. Let's get into it.
[O] Welcome to Litsearch Audio, where an optimist, a skeptic, and a visiting scholar take apart one paper from the map at litsearch dot darvinyi dot com. Today it's "Agent-as-a-Judge: Evaluate Agents with Agents," from Mingchen Zhuge, Changsheng Zhao, Dylan Ashley, Wenyi Wang and colleagues at Meta AI and KAUST — posted in October twenty twenty-four and later accepted at ICML twenty twenty-five. Joining us is Antonia, a visiting scholar who has spent real time inside this paper. Welcome, Antonia.
[G] Glad to be here. What makes this one fun is that it turns the evaluation question back on itself — you use an agent to grade an agent — and the authors are unusually candid about exactly where that trick holds up and where it doesn't.
[S] Start with the actual gap, then. Why isn't LLM-as-a-Judge, which already works fine for chatbots, good enough for agents?
[G] Because an agentic system doesn't produce one answer, it produces a trajectory — it browses, writes code, runs it, reads the error, and tries again. LLM-as-a-Judge was built for the chatbot case: show the judge one final text blob and ask which one is better. Point it at an agent's messy, multi-step run and it can only see the very end, so it has no way to tell you where inside that run things actually went wrong.
[O] Which matters because the existing benchmarks have the same blind spot. SWE-bench just reports a final resolve rate.
[G] Right, and the authors make a sharper point about SWE-bench specifically. Separate work had already shown over twenty-seven percent of its tasks are solvable without a real agent at all, and some teams had started building mechanisms tuned to individual SWE-bench instances. That's Goodhart's law in action — once a measure becomes the target, it stops measuring the thing you actually cared about.
[S] And the honest alternative, having humans watch the whole trajectory, is the expensive option nobody can run at scale.
[G] Exactly. In this paper, one full pass over their benchmark by three human experts took eighty-six and a half hours. At a fifteen-dollar-an-hour floor, that comes to one thousand two hundred ninety-seven dollars, for fifty-five tasks. So the gap they're chasing is an evaluator that behaves like an agent — exploring the workspace, checking intermediate artifacts — while staying as cheap as an LLM judge.
[O] So Agent-as-a-Judge is basically LLM-as-a-Judge that got hands.
[G] More or less how the authors describe it.
[S] Walk us through what "hands" actually means. What does this judge do that a normal LLM judge doesn't?
[G] Two things ship together in this paper. First, a new benchmark called DevAI — fifty-five realistic "build me an AI application" tasks, written by expert annotators, spanning supervised learning, reinforcement learning, computer vision, natural language processing, and generative models. Each task comes with a plain-text user query, plus three hundred sixty-five individual requirements in total, and a hundred twenty-five softer preferences.
[O] And the requirements aren't just a checklist.
[G] No, that's the actual contribution. They're arranged as a directed acyclic graph — a DAG — so "visualize the results" depends on "train the model," which depends on "load the data." An agent earns credit milestone by milestone instead of one binary solved-or-not, and because later steps genuinely depend on earlier ones, you can't game it by memorizing a plausible-looking final answer.
[S] Give me a concrete example, not an abstraction.
[G] Task fifty-one is adapted from a real Devin demo — a blog post called "Hidden in Plain Sight," about generating images with hidden text embedded in them. The query asks the agent to follow that blog's instructions, produce ten-eighty-p images with the word "FUTURE" hidden inside, and save them to a results folder. That decomposes into three chained requirements: set up the script, generate images at the right resolution, and actually embed the hidden text, each one gated on the last.
[O] Okay, and the judge itself — what's actually inside it?
[G] The proof-of-concept starts with eight modules, imitating how a careful human reviewer works. A graph module maps the whole project — files, dependencies, code chunks. Locate finds the specific file a requirement refers to. Read understands content across thirty-three formats — code, images, video, documents. Search does contextual code retrieval. Retrieve pulls the relevant span out of a long trajectory log. Ask actually decides whether a requirement is satisfied. And memory and planning were meant to carry judgments forward and sequence the judge's own actions.
[S] Eight modules sounds like a lot of surface area for something that's supposed to be reliable.
[G] It was, and the ablations agree with you. The winning combination drops to five: graph, locate, read, retrieve, and ask. Memory actually hurt, because a wrong early judgment would chain into wrong later ones. Planning was unstable. And search barely mattered, because these generated workspaces are only a few hundred lines of code — there isn't enough there for a search tool to add value over just reading the files directly.
[O] So the lesson is keep the judge simple and grounded, don't let it get clever.
[G] That's exactly the authors' own read. They say the judge needs high-quality factual information and is sensitive to noise.
[S] One more thing before results — black-box versus gray-box. What's the difference?
[G] Black-box means the judge only sees the final generated workspace, the files an agent left behind. Gray-box adds a hand-collected trajectory log of the developer agent's intermediate actions. The authors are upfront that gray-box data is, in their words, nearly inaccessible in practical situations, so black-box is the realistic setting, and it's worth remembering that whenever a gray-box number gets quoted as the headline.
[O] Let's get to the scoreboard, then. How hard is DevAI, really?
[G] Hard. Under careful human evaluation, the two strongest open systems, GPT-Pilot and OpenHands, each satisfy only about twenty-nine percent of requirements once you enforce the dependency structure — around forty-four percent if you ignore dependencies entirely. MetaGPT trails badly, under seven percent once dependencies are enforced. Across all fifty-five tasks, GPT-Pilot and OpenHands each fully completed exactly one; MetaGPT completed none.
[S] So essentially none of today's popular open coding agents can finish a real multi-step AI development task end to end.
[G] Correct, which is the point. DevAI sits in a difficulty band that's genuinely challenging without being hopeless.
[O] Before we even get to the automatic judge, you flagged that the humans themselves weren't perfectly reliable.
[G] Right, and the paper is unusually honest about this. The three expert evaluators disagreed with each other on ten to thirty percent of requirements. One evaluator alone was wrong twenty-three point seven seven percent of the time judging GPT-Pilot. Only after a full debate round and a majority vote did the error against the final consensus fall to six point zero one percent.
[S] Which should recalibrate how anyone reads "matches human performance" claims across this whole space, not just this paper.
[G] Agreed, and it's exactly the frame the authors use to defend their own numbers. Beating an average lone human is a real, meaningful bar, because a lone human is demonstrably not the gold standard here.
[O] So how does Agent-as-a-Judge actually do against that bar?
[G] On alignment with the human consensus — the fraction of all three hundred sixty-five requirement verdicts that match — Agent-as-a-Judge lands in the high eighties to low nineties across all three developer systems, in both black-box and gray-box settings. Judging OpenHands in black-box, it hits ninety point four four percent, which edges past even the best single human evaluator's eighty-nine point eight nine percent. LLM-as-a-Judge, on that same system, manages sixty point three eight percent — twenty to thirty points lower across the board.
[S] And the deviation-from-consensus number, the one they call judge shift?
[G] Same story. On OpenHands, Agent-as-a-Judge's judge shift gets as low as point two seven percent. LLM-as-a-Judge's blows out to roughly thirty-one percent on that same comparison.
[O] What's actually earning that gap? Which module is doing the work?
[G] The component ablation, judging OpenHands, breaks it down cleanly. Ask alone gets about sixty-five percent alignment. Adding the workspace graph lifts it to seventy-six percent. Adding read, actually opening file contents, gets you to eighty-two percent. And locate, just pointing the judge at the right file, jumps it to ninety percent. Retrieve adds essentially nothing here, though the authors note it helps more when judging MetaGPT and GPT-Pilot, where the trajectory carries more signal.
[S] So the single biggest lever is just "look at the right file," not some deep reasoning capability.
[G] That's a fair read of the ablation, yes.
[O] And the cost case — that's the number I think will get repeated the most.
[G] It's stark. The full human evaluation cost one thousand two hundred ninety-seven dollars and eighty-six and a half hours. Agent-as-a-Judge did the same job for thirty dollars and fifty-eight cents and under two hours — roughly two percent of the cost and two percent of the time. LLM-as-a-Judge was even faster, about eleven minutes, but barely cheaper, around thirty dollars, and, as we just covered, far less accurate.
[O] Okay, my optimist case. This is a real unlock for training, not just grading. The paper's own discussion section points at building a process-supervised reward model out of this — intermediate, per-requirement feedback instead of one delayed signal at the end of a long rollout. That's the sparse-reward problem in reinforcement learning, addressed for agentic coding, basically for free once you've built the judge.
[S] My deflationary case: the "matches humans" claim is doing more work than the evidence supports. Start with the ground truth itself. The human consensus is a negotiated agreement among the same three annotators whose individual judgments the paper then claims to beat. And the bar that actually matters, the majority vote of all three, still beats Agent-as-a-Judge on every single system — ninety-five, ninety-four, and ninety-four percent versus its high eighties to low nineties.
[G] That's fair, and it's not a hidden fact. The authors report the majority-vote numbers themselves, right in the same table. The honest version of the claim is narrower than the headline: it matches or slightly beats the best single human, not the panel.
[S] Second complaint: the metric itself. Most requirements go unmet, twenty-nine percent is a good outcome here, so "not satisfied" is the overwhelming majority class, and a judge that just leans negative scores well by default. The paper admits this outright — LLM-as-a-Judge hits eighty-four percent alignment on MetaGPT specifically because MetaGPT almost never meets a requirement, so guessing no is usually correct.
[G] Also accurate, and to their credit the authors ran precision-recall curves specifically to correct for that imbalance — Agent-as-a-Judge still comes out ahead there too. But those curves are one supplementary figure, not the number in the abstract, and the headline metric is the more flattering one.
[O] That one lands. There's also the module-drop findings from earlier — Antonia, does that change how much weight the alignment number should carry?
[G] A little. Since locate alone accounts for most of the jump, and locate is essentially "find the right file," some of that ninety percent is measuring a fairly mechanical skill, not deep judgment. That doesn't make the number wrong, but it's worth knowing what's actually driving it.
[S] There's also something circular about the whole setup — using an agent to grade agents, when both share the same failure modes. And "two percent of the cost" describes running the judge once it exists. Getting from eight modules down to the five that actually work took real engineering and ablation effort that a single cost number doesn't capture.
[G] Both fair. The circularity concern is real, and the paper doesn't fully address it — an agent judge inherits the blind spots of its own architecture, and if a developer agent and the judge share a failure mode, neither one would necessarily catch it. On the engineering cost, that's true of most proposed evaluation methods, though. The ablation work is a one-time cost amortized across every future evaluation, a different kind of investment than paying humans by the hour indefinitely.
[O] What about backbone risk? That's the one I keep coming back to.
[S] That's my sharpest complaint. All three developer agents, MetaGPT, GPT-Pilot, OpenHands, ran on the identical GPT-4o snapshot. The paper never discloses which model powers either judge. Self-preference bias in LLM judges is well documented at this point, so if the agent judge shares a model family with the systems it's grading, you cannot rule out that it's partly scoring its own kind. And there's no cross-model check anywhere — no Claude or Gemini judge scoring the same GPT-4o developers.
[G] That gap is real. Going a bit past what's written on the page, my honest read is that it's the least defended part of the paper. They flag class imbalance, they flag gray-box being unrealistic, they even flag the modules that failed. They never once name the judge's own backbone. That's a real blind spot, not just a hedge they forgot to write down.
[O] Doesn't sink the result, though. The mechanism, agentic tool use beating a blind final-answer read, doesn't obviously depend on which model is doing the judging.
[S] Maybe. But the exact size of that ninety-percent number might, and that's precisely the number that gets repeated.
[O] Zooming out — where does this connect to how the field measures agents more broadly?
[G] It's part of a broader shift toward evaluating agentic systems by process rather than outcome — the same instinct behind process reward models in reasoning, and behind newer agent benchmarks that score intermediate steps instead of one pass-fail number. DevAI's dependency DAG is a genuinely reusable idea outside of coding, anywhere a task naturally decomposes into ordered milestones.
[S] I'd add the caveat that travels with it: this is fifty-five coding tasks, one run each, with a hard cutoff and no repeated trials to show variance. Whether an agentic judge holds up outside of code, where "did the file get written correctly" is a much crisper question than in most domains, is untested here.
[G] The authors are careful to call this a proof-of-concept, not a finished measurement instrument, so that scope limit is honestly stated, even if the caveats don't always travel with the headline number when people cite it later.
[S] My takeaway: read every "matches human performance" claim in this literature and ask which human, on what metric, under what class balance, because in this very paper, the honest answer to all three questions is more modest than the headline number.
[G] Mine is the paper's own frame: this is agents judging agents, checked against a dependency graph of milestones instead of one final score, at roughly two percent of the cost of a human panel — genuinely promising as a reward signal, not yet a settled measurement instrument.
[O] And mine is that the real story is capability, not cost. A judge that can open a repository and check it against requirements is a fundamentally more useful primitive than one that only reads a paragraph, and that primitive gets more valuable as agents get more autonomous.
